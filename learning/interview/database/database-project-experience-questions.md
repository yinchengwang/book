## 题型说明

```
每位候选人独立完成一个题目或场景
判定标准：候选人是否能给出具体的细节、数字、代码、架构图或排查过程。
如果回答只有理论没有实战细节，分数直接降档。
```

**评估框架**：

| 指标 | 优秀 | 一般 | 差 |
|---|---|---|---|
| 具体性 | 有具体 SQL、具体数字、具体时间线 | 笼统描述"用了索引" | "我优化过数据库" |
| 深度 | 能说到锁、IO、执行计划层面 | 说到加了索引 | 根本不知道为什么 |
| 反模式意识 | 知道哪些做法看似优化实则埋坑 | 照本宣科 | 不知道 |
| 复盘能力 | 说了事后做了什么改进 | 只说了当时怎么修 | 修完就完了 |

## 一、SQL 性能优化实战

### SC1：你的项目中遇到过最复杂的 SQL 优化是什么？讲讲具体的表结构和 SQL

**追问链**：

```
① 原 SQL 长什么样？（必须能写出来）
② 原执行计划的 type 和 Extra 是什么？
③ 数据量多少？优化前跑了多久？优化后跑了多久？
④ 索引怎么加的？联合索引的字段顺序怎么决定的？
⑤ 优化的核心瓶颈是什么？—— IO？CPU？锁？
```

**踩坑点（加分项）**：
- 加了索引但走了错索引 → 用了 `USE INDEX`/`FORCE INDEX` 或删了错索引
- 发现 `range` 查询导致联合索引后续列失效 → 调整索引顺序
- `filesort` 优化 → 排序字段加入索引
- 深分页问题 → 游标分页或延迟关联

**编造鉴别**：候选人如果说"优化了一个很慢的查询，加了索引就从 10 秒变 0.1 秒"，反问加了什么索引，如果回答"就是在 WHERE 字段上加了索引"，说明没遇到过真实问题——真实场景往往需要分析为什么已有索引没走对。

### SC2：慢查询日志分析——如果线上突然出现大量慢查询，你的排查 SOP 是什么？

**要求候选人给出具体的步骤，而不是"看慢查询日志"这种一句话回答**

**预期回答**（步骤越详细越好）：

```
第 1 分钟：止血
  SHOW FULL PROCESSLIST → 看有没有大量 Waiting for table metadata lock
  → 如果有，找到源头 DDL/DML 进程 KILL
  → 如果没有，看哪些 SQL 执行时间最长

第 5 分钟：定位根因
  - 慢查询日志：最近 1 分钟的慢 SQL 分布（pt-query-digest）
  - 有没有新上线的 SQL？
  - 有没有数据量突增？
  - 索引统计信息是否过时？（ANALYZE TABLE）
  - 锁等待？（INFORMATION_SCHEMA.INNODB_TRX）

第 15 分钟：应急处理
  - 索引缺失 → 在线加索引（gh-ost/pt-osc）
  - 走了错索引 → FORCE INDEX 或删错索引
  - 数据量暴增 → 考虑归档/分表
  - 连接池打满 → 临时调大 max_connections + 排查连接不释放的原因

事后：复盘文档
  - 5Why 分析：为什么慢？为什么没提前发现？监控告警为什么没覆盖？
  - 改进：慢查询阈值从 2s 降到 500ms + 慢查询自动上报
```

### SC3：假设有个查询：SELECT * FROM t WHERE a = 1 AND b > 10 ORDER BY c LIMIT 100。你有一个联合索引 (a, b, c)，你觉得这个查询能完全走索引吗？

**实战分析题**：

**实际情况**：

```sql
INDEX(a, b, c)

WHERE a = 1          → 索引第一列，等值匹配 ✅
  AND b > 10         → 索引第二列，范围匹配 ✅
ORDER BY c           → 索引第三列——❌ 排序不走索引！
```

为什么 ORDER BY c 不走索引？因为 `b > 10` 是范围查询，匹配到的是多个 b 值，`c` 在这些 b 值中不是有序的。

**优化方案**：

```sql
-- 方案一：改成 (a, c, b) —— 但 b 不能做范围了
INDEX(a, c, b)

-- 方案二：MySQL 8.0 降序索引 —— 但解决不了范围导致的无序
INDEX(a, b, c DESC)

-- 方案三：拆成两次查询（查询 b > 10 的最近 100 个 a 值，再过滤）
-- 不如接受 filesort（如果数据量不大）

-- 方案四：去掉 ORDER BY，业务层排序
```

**追问**：如果数据量是 100 条，你在意这个 filesort 吗？100 万条呢？

**答案**：100 条不在意。100 万条要优化。**优化要建立在数据量和业务场景上，不能为了优化而优化。**

## 二、高并发实战

### SC4：你遇到过数据库连接池被打满的情况吗？怎么排查和处理的？

**追问链**：

```
① 你怎么知道连接池打满了？（监控指标是什么？）
② 是客户端连接池满了还是 MySQL 连接数满了？
③ 具体原因：
   - 慢查询导致连接不释放？
   - 连接泄漏（未 close）？
   - 突发流量？
   - 死锁导致事务挂起？
④ 怎么解决的？短期措施和长期措施分别是什么？
```

**典型真实案例**：

```
现象：某天 10:00，监控告警，应用报错 "Cannot get connection from pool"
      MySQL 的 Threads_connected 飙升到 2000+

排查过程：
  SHOW FULL PROCESSLIST → 大量 SQL 在 "Sending data" 状态
  同一张表、同一类 COUNT 查询
  → 该查询 10ms → 用户反复刷新 → 连接不释放 → 打满

根因：前一天上线了一个新功能，首页调用了一个 COUNT 查询
     但上线前测试环境数据量小（1000 条），线上 500 万条

处理：
  临时：KILL 掉堆积的查询，加索引
  长期：慢查询阈值设为 500ms，上线前用生产数据量压测
```

### SC5：假设业务高峰时期（双 11/促销），读 QPS 突然涨了 10 倍，你怎么保证数据库不崩？

**开放式设计题**：

**期望回答**（越体系化越好）：

```
第一层：缓存
  ① Redis 缓存热数据（缓存穿透/击穿/雪崩都要考虑）
  ② 本地缓存（Caffeine/Guava Cache）—— 二级缓存兜底
  ③ 缓存预热 + 过期时间错峰

第二层：读写分离
  ① 主库写，从库读
  ② 从库可以水平扩展（一主多从）
  ③ 注意：从库延迟可能导致数据不一致

第三层：限流
  ① 接口层面限流（令牌桶/漏桶）
  ② SQL 层面：SET MAX_EXECUTION_TIME

第四层：降级
  ① 非核心查询走 ES 或干脆返回缓存数据
  ② 展示"系统繁忙"而不是直接 502

第五层：扩容
  ① 分库分表（如果已经是瓶颈）
  ② 实际提前容量规划，不可能等高峰了再扩容
```

**追问**：如果 10 倍 QPS 中还有 5% 是写操作呢？

**答案**：
- 写操作无法靠从库扩展
- 分库分表（把写压力分散到多个主库）
- 写操作异步化（MQ 削峰，最终一致）
- 批量写入合并（比如 100ms 窗口内的 INSERT 合并成一条批量 INSERT）

### SC6：你的项目中有没有遇到过热点数据问题？比如一条记录被上万人同时更新？

**追问链**：

```
① 具体场景是什么？（秒杀库存？点赞数？计数器？）
② 怎么发现是热点问题的？（TPS 监控？锁等待？）
③ 怎么解决的？
```

**常见的解决方案及其取舍**：

| 方案 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **Redis 原子操作** | `INCR/DECR` 在 Redis 中完成，异步落库 | 性能极好 | 可能丢数据 |
| **行锁拆分** | 把一条库存记录拆成 100 个槽位 | 分散锁竞争 | 需要额外管理槽位 |
| **批量合并** | 点赞等操作攒 10ms 批量更新一次 | 减少更新次数 | 延迟增加 |
| **乐观锁** | `UPDATE t SET cnt=cnt+1 WHERE id=1 AND version=ver` | 无锁阻塞 | 失败重试 |
| **异步队列** | 写请求入 MQ，数据库端串行消费 | 削峰填谷 | 最终一致 |

**编造鉴别**：如果候选人说"我们用 Redis 分布式锁解决了"，追问：
- 锁的粒度是什么？库？表？行？
- 解锁失败怎么处理的？（锁超时？看门狗？）
- 如果持有锁的进程挂了，其他进程要等多久？

高级回答应该是："只在必要的**最后一步**用锁，前面的校验和预扣都用 Redis 原子操作完成。"

## 三、分布式/分库分表实战

### SC7：你们项目做过分库分表吗？分片键怎么选的？踩过什么坑？

**追问链**：

```
① 为什么要分？（数据量多大？单表多少行？）
② 用什么方案？（ShardingSphere？MyCAT？还是业务代码自己路由？）
③ 分片键选的什么？为什么选它？
④ 分了多少个库、多少个表？
⑤ 有没有做过扩容？扩的时候痛苦吗？
```

**选分片键的核心思路**：

```
第一步：列出所有查询 SQL，统计频率
第二步：找出出现频率最高的 WHERE 条件字段
第三步：该字段是否能均匀分布？是否有热点？
第四步：该字段是否几乎不变？
```

**常见坑**：

```
坑 1：选了 order_id 做分片键（高频查询按 user_id 查 → 广播到所有分片）
坑 2：分片键值分布不均匀（按地区分片，北京上海一个库，西藏一个库）
坑 3：分片键会变（用户换了手机号 → 数据需要迁移）
坑 4：自增主键全局不唯一（需要分布式 ID 生成器）
坑 5：跨分片查询（分页+排序，需要各分片返回 Top-N，聚合再排序）
```

**追问题**：如果没有完美的分片键（所有查询都带不同条件），你怎么做？

**答案**：
- 建立索引映射表：把辅助查询条件映射到主分片键（如 `user_phone → user_id`）
- ES 辅助：跨分片搜索走 ES，返回 ID 再查分片
- 异构双写：同时按不同维度冗余存储

### SC8：你们的主从同步有没有出现过延迟？怎么发现、怎么处理的？

**追问链**：

```
① 怎么监控的？（Seconds_Behind_Master？这个指标有什么局限？）
② 什么原因导致的延迟？
   - 从库硬件比主库差？
   - 从库同时在做查询？
   - 大事务？
   - 主库有 DDL？
   - 网络带宽？
③ 怎么解决的？
④ 如果业务不能接受延迟（比如刚下单就要查到），怎么保证？
```

**排查经验**：

```sql
-- 查看主从延迟（只能精确到秒）
SHOW SLAVE STATUS\G
-- Seconds_Behind_Master: N

-- 更精确的主从延迟检测：在主库写一条带时间戳的记录
-- 在从库查询这条记录，计算时间差
```

**解决方案分级**：

```
① 减少延迟
   - 从库升级硬件（SSD + 足够内存）
   - 并行复制（MySQL 5.7+ slave_parallel_workers）
   - 大事务拆小（比如 100 万行 UPDATE 拆成 1000 批 × 1000 行）

② 业务层解决"读不到刚写的数据"
   - 强制读主：写完后一分钟内的读请求走主库
   - 前端策略：提交成功后延迟跳转（等 1-2s）
   - 缓存策略：写主库同时写 Redis，读优先读 Redis
```

### SC9：你有没有设计过分库分表后的分布式 ID 生成方案？讲讲你的方案及其优劣

**实战方案对比**：

| 方案 | ID 结构 | QPS | 有序性 | 优点 | 缺点 |
|---|---|---|---|---|---|
| **UUID** | 32 位随机字符串 | ∞ | ❌ | 生成极简单 | 太长、无序、影响 B+树性能 |
| **数据库自增** | AUTO_INCREMENT | <10 万 | ✅ | 简单 | 单点、性能瓶颈 |
| **Redis INCR** | `INCR id_key` | ~10 万 | ✅ | 性能好 | Redis 挂了重复 |
| **雪花算法** | 时间戳+机器ID+序列号 | 理论~69 万/秒 | ✅ | 去中心化 | 时钟回拨问题 |
| **号段模式** | 批量取号段 | 取决于号段大小 | ✅ | 灵活、容错 | 号段浪费 |

**追问**：雪花算法的时钟回拨问题你怎么解决？

**高级回答**（按水平排序从低到高）：

```
L2：如果时钟回拨，抛异常，等时间追上
L3：记录上次生成 ID 的时间，如果时钟回拨了但没超过阈值，用备用序列号区间
L4：使用"未来时间"思想——机器启动时记录一个基准时间，ID 里存的是"未来时间偏移量"
    只要时钟回拨不超过基准时间，就没有问题
    如果回拨超过基准时间 → 抛异常或等待
```

## 四、方案设计题

### SC10：设计一个短链接系统，说说存储层怎么设计

**考察点**：候选人能否把"技术选型"讲清楚，而不是只讲业务逻辑。

**追问方向**：

```
① 短链接和原始 URL 的映射存在哪里？（MySQL？Redis？）
② 短链接 key 怎么生成的？（hash + 去重？发号器？）
③ 如果用户访问短链接，是先查缓存还是先查 DB？
④ 如果某个短链接被高频访问（如推广大促链接），怎么抗？
⑤ 如果原始 URL 超长（2000 字符），对索引有什么影响？
```

**存储层参考方案**：

```sql
-- 核心表设计
CREATE TABLE short_url (
  id        BIGINT PRIMARY KEY AUTO_INCREMENT,
  short_key VARCHAR(8) NOT NULL UNIQUE,     -- 短链接码
  origin_url VARCHAR(2048) NOT NULL,        -- 原始 URL
  expire_at DATETIME,                       -- 过期时间
  created_at DATETIME NOT NULL DEFAULT NOW(),
  INDEX idx_expire(expire_at)               -- 用于定期清理过期数据
);

-- 缓存设计
-- Key: short:url:{short_key}
-- Value: {origin_url, expire_at}
-- TTL: 过期时间（比 DB 短，防止数据不一致）
-- 缓存穿透保护：布隆过滤器（Bloom Filter）
```

### SC11：设计一个秒杀系统的库存扣减方案，说说为什么这么设计

**追问链**：

```
① 库存存在哪里？（Redis？MySQL？）
② 用户点"立即购买"时，先扣库存还是先下单？
③ 超卖怎么防止？
④ 如果 Redis 挂了怎么办？
⑤ 峰值 TPS 多少？怎么压测出来的？
```

**推荐方案（多级防御）**：

```
前端层
  └── 按钮置灰（防止重复提交）
  └── 限流（同一用户 1s 内最多请求 1 次）

应用层
  └── Redis 预扣库存（Lua 脚本保证原子性）
  └── 令牌桶限流（总流量控制在 DB 能承受的范围）

数据库层
  └── UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0
  └── 行锁，天然防超卖
  └── 异步落单（MQ 削峰）
```

**追问**：如果库存最后一步（如支付超时）需要回退库存，你的回退方案是什么？

**答案**：
- Redis：使用 Lua 脚本 INCR 回退
- DB：UPDATE stock SET count = count + 1 WHERE id = ?
- 注意：回退比扣减更危险（恶意刷回退可以无限刷库存）
- 防线：回退必须有原始订单 ID 校验 + 频率限制 + 风控

### SC12：设计一个多租户 SaaS 系统的数据库层，三个方案各有什么优劣？

**三种方案对比**：

```
方案 A：独立数据库（一个租户一个 DB）
  ✅ 隔离性最好，备份恢复独立，可针对大客户分配专用资源
  ❌ 维护成本高，连接数多，跨租户查询几乎不可能

方案 B：共享数据库，独立 Schema（一个 DB 多个 Schema）
  ✅ 隔离性中等，可共用连接池
  ❌ 每个租户需要独立迁移脚本，跨 Schema 查询复杂

方案 C：共享数据库，租户 ID 字段（所有租户在同一张表）
  ✅ 维护最简单，开发效率最高
  ❌ 隔离性最差，一个租户的慢查询可能影响所有人
  ❌ 所有租户加索引/改表需要评估对全局的影响
```

**加分回答**："我们用**混合方案**——大部分小客户走方案 C，大客户走方案 A 或独立部署。"

### SC13：你们做数据迁移/大表 DDL 的方案是什么？有没有出现过事故？

**追问链**：

```
① 数据量多大？迁移到什么目标？
② 用了什么工具？（gh-ost？pt-osc？）
③ 怎么保证数据不丢不重？
④ 怎么切换的？（灰度切流量？）
⑤ 有没有出过事故？
```

**关键经验**：

- 迁移过程中 Binlog 格式必须是 **ROW**（Statement 格式在某些函数下可能重现不出同样的结果）
- 切换时最好保留**回滚能力**（比如 DNS 切流量而不是改代码）
- 迁移后的**数据校验**比迁移本身更重要
- 大表 DDL 最怕的不是时间长，而是**切换阶段的写阻塞触发业务超时雪崩**

### SC14：MySQL 批量插入 100 万条数据，怎么做最快？

**考察点：候选人是否理解 MySQL 的写入瓶颈和优化手段**

**逐步优化**：

```
❌ 逐条 INSERT（最慢）
   INSERT INTO t VALUES (...);  -- 100 万次

❌ 逐条 INSERT 在事务中（省了部分 commit 开销）
   BEGIN;
   INSERT ...;
   ...
   COMMIT;  -- 仍然 100 万次 SQL 解析

✅ 批量 INSERT
   INSERT INTO t VALUES (...), (...), ...;  -- 每批 1000 条

✅ 批量 INSERT + 事务（每批一个事务）
   BEGIN;
   INSERT ... (1000 条);
   COMMIT;

✅ LOAD DATA INFILE（最快，但对格式要求严格）
   LOAD DATA INFILE '/tmp/data.csv' INTO TABLE t;
```

**性能对比（经验值）**：

| 方式 | 100 万条耗时 |
|---|---|
| 逐条 INSERT | ~600s |
| 每批 1000 条 + 事务 | ~5-8s |
| LOAD DATA INFILE | ~2-3s |

**追问**：批量插入时，对现有索引和查询有什么影响？

**答案**：
- 每条 INSERT 都要维护索引（B+ 树分裂）
- 索引越多，插入越慢（建议导入前删非必要索引，导入后重建）
- 大量写入导致 Buffer Pool 刷脏页 → 可能影响读查询的响应时间
- 如果使用 InnoDB：`innodb_flush_log_at_trx_commit=2` 可以加速但可能丢 1 秒数据

### SC15：设计一个数据库连接池的健康检查方案

**考察点：候选人是否理解连接池的工作原理和网络问题**

**追问链**：

```
① 怎么检测连接已断开？（TCP 层？应用层？）
② 怎么避免检测本身成为性能负担？
③ 如果有大量空闲连接被 DB 侧断开（wait_timeout），怎么办？
```

**健康检查方案**：

```
被动检测：
  - 执行 SQL 时捕获 CommunicationsException
  - 标记连接为"已失效"，从池中移除
  - 下次获取连接时创建新连接

主动检测：
  - 定时发送 SELECT 1（或特定心跳 SQL）
  - HikariCP：keepaliveTime（建议 30s）+ connectionTimeout（建议 5s）
  - 检测间隔不要太短（否则压力都在 DB 上）

兜底策略：
  - 获取连接时验证：connectionTestQuery = "SELECT 1"
  - 只在连接空闲超过一定时间时才验证（减少开销）
```

**加分回答**：
- 用了 **Proxy** 方案（如 ProxySQL、MaxScale）而不是直连 DB → 连接池管理在 Proxy 层，应用更轻量
- 遇到过 `wait_timeout` 导致连接半夜批量断开 → 早上第一个请求全部超时 → 加了 `connectionTestQuery` 预热
