## 维度一：基础理论（L1-L2）

### Q1：KNN 与 ANN 的区别是什么？为什么实际系统中几乎都用 ANN？

**难度**：L1

**参考答案**：

| | KNN（精确） | ANN（近似） |
|---|---|---|
| 结果 | 绝对精确，返回全局最近邻 | 近似结果，可能遗漏真实最近邻 |
| 时间复杂度 | O(n×d) | O(log n) ~ O(√n)，取决于索引类型 |
| 空间 | 仅存原始向量 | 额外存索引结构（图/聚类/哈希表） |
| 数据量扩展 | 1 万条以上显著变慢 | 可扩展到千万/亿级 |

**核心原因**：当数据量超过 10 万条时，暴力扫描的内存带宽和计算延迟会达到不可接受的程度（百毫秒 ~ 秒级）。ANN 通过预构建索引将延迟压缩到毫秒级，而召回率通常可以做到 95%+，在工程上是值得的 trade-off。

### Q2：假设 100 万条 128 维 float32 向量，暴力搜索一次需要多少次浮点运算？延迟大约多少？

**难度**：L2

**参考答案**：

- 每条向量与查询向量的 L2 距离：128 次减法和乘加 ≈ 128 × 2 = 256 FLOP
- 100 万条：10^6 × 256 = **2.56 亿次浮点运算**
- 现代 CPU（AVX2/FMA，~50 GFLOPS）：约 **5ms**
- 但如果用纯标量实现 + 内存瓶颈（128×4=512 字节/条 × 100 万 ≈ 512MB，cache miss 严重），实际延迟可能在 **20-100ms**。

这就是为什么即使暴力搜索可以到毫秒级，HNSW 等索引仍然追求 **0.1-1ms** 的延迟。

### Q3：什么是"维度灾难"？在什么维度开始显现？

**难度**：L2

**参考答案**：

维度灾难是指：随着向量维度增加，高维空间中任意两点之间的距离趋于相等，最近邻与最远邻的距离之比趋近于 1，导致"最近邻"和"最远邻"几乎无法区分。

数学上可以证明：在 d 维空间中，如果各维度独立，最大距离与最小距离之比满足：

lim<sub>d→∞</sub> (dist<sub>max</sub> - dist<sub>min</sub>) / dist<sub>min</sub> → 0

**经验阈值**：大约 d > 20 时开始显现，d > 100 时已经非常明显。所以对 128/256/768/1536 维的向量，单纯依赖距离度量进行精确 KNN 意义不大——所有候选之间的距离几乎一样。

**实际影响**：这是为什么 PQ 量化、LSH 哈希等方法在高维场景下效果受限的根本原因。也是 IVF/HNSW 这类"先缩小搜索空间再精确计算"的策略更有效的原因。

### Q4：L2、内积、余弦三种距离的适用场景和等价关系？

**难度**：L1

**参考答案**：

| 度量 | 公式 | 适用场景 |
|---|---|---|
| L2（欧氏） | √Σ(aᵢ-bᵢ)² | 默认选择，向量范数有意义时 |
| 内积（IP） | Σaᵢ×bᵢ | 排序场景、向量已归一化 |
| 余弦 | cosθ = (a·b)/(‖a‖×‖b‖) | 文本 Embedding（关注方向而非长度） |

**等价关系**：如果向量已经 **L2 归一化**（‖a‖=‖b‖=1），则：
‖a-b‖² = 2 - 2cosθ → L2 和余弦完全等价
a·b = cosθ → 内积和余弦完全等价

所以很多系统（如 Faiss）实际只实现 L2 和 IP，余弦通过先归一化再用 IP 来等价实现。

### Q5：Faiss 中 IndexFlatIP 和 IndexIVFFlat 返回的 distance 含义为什么不一样？

**难度**：L2

**参考答案**：

- **IndexFlatIP**（暴力内积）：返回的是**内积值**。对归一化向量，就是余弦相似度，值越大越相似。
- **IndexIVFFlat**（倒排 + 精确向量）：返回的是 **L2 距离的平方**。值越小越相似。

这是因为 Faiss 内部对 IVF 的 search 实现统一返回距离，而 flat 和 IVF 使用的 metric_type 处理逻辑不同。这是 Faiss 的一个经典陷阱。

**排查方式**：看 Faiss 文档中每个 index 的 `search()` 返回值说明，或直接看源码 `IndexIVF.cpp` 中 `search` 调用的 `quantizer->search` 和 `indexIVF_stats` 的处理逻辑。

## 维度二：索引算法深度（L2-L4）

### Q6：IVF 的全流程（建库 + 搜索），nlist 和 nprobe 怎么调？

**难度**：L2

**参考答案**：

**建库流程**：
1. **训练阶段**：对全部向量做 K-Means 聚类，nlist 个聚类中心（coarse quantizer）
2. **向量分配**：每个向量分配到最近的聚类中心，建立倒排链表（聚类 ID → 向量列表）
3. （可选）对残差向量做 PQ 量化，压缩存储 → IVF-PQ

**搜索流程**：
1. 查询向量与 nlist 个聚类中心计算距离
2. 选择最近的 nprobe 个聚类
3. 在这 nprobe 个桶内进行暴力搜索（或 PQ 距离计算）
4. 合并结果并排序

**调参经验**：

| 参数 | 影响 | 推荐值 |
|---|---|---|
| nlist | 越多 → 每个桶越小 → 搜索越快，但聚类中心训练越难 | sqrt(N) ~ N/10（经验公式） |
| nprobe | 越大 → 召回越高 → 延迟线性增加 | 从 1 开始，倍增直到召回达标 |

**典型曲线**：从 nprobe=1 到 nprobe=10，召回率提升最快（如从 50%→85%）；nprobe=10 到 100，提升趋缓（85%→95%）。调参时找到曲线上**拐点**即可，不必追求极端。

### Q7：HNSW 的核心数据结构？为什么比 NSW 快？

**难度**：L3

**参考答案**：

**核心数据结构**：多层图（Multi-layer Navigable Small World Graph）

- 最底层（Layer 0）：包含所有数据点，每个节点连接 M 个邻居（双向边）
- 越往上层：节点越少（通过指数衰减概率 ρ = 1/ln(M) 决定节点出现在第几层）
- 最顶层：仅少数几个节点

**为什么比 NSW 快**：

NSW（Navigable Small World）只有单层图，搜索时从随机入口开始贪婪遍历，在最坏情况下退化为 O(n)。HNSW 通过**分层**解决了这个退化问题：

1. 搜索从顶层（节点最少）开始，快速"跳到"目标区域（宏观定位）
2. 逐层下降，每层做局部精细搜索（微观定位）
3. 总搜索复杂度 O(log n)，与 NSW 的 O(√n) 相比显著降低

**类比**：这类似于在一本书中搜索内容——NSW 是从第一页逐页翻；HNSW 是先看目录确定章节（顶层），再跳到具体段落（中层），最后逐句查找（底层）。

**关键参数**：
| 参数 | 作用 | 典型值 |
|---|---|---|
| M | 每个节点的双向边数 | 16-32（越大召回越高，内存越大） |
| efConstruction | 建图时的搜索宽度 | 100-500（越大图质量越高，建图越慢） |
| efSearch | 搜索时的动态候选集大小 | 50-500（越大召回越高，延迟越高） |
| Mmax | 上层节点的最大边数 | M×2 |

### Q8：HNSW 的内存占用怎么估算？100 万条 128 维 float 要多少内存？

**难度**：L3

**参考答案**：

**原始向量**：10^6 × 128 × 4 = **512 MB**

**HNSW 图结构**（每节点 M=32 条边，双向边）：
- 每个邻居 ID 存 int（4 字节）
- 总边数 ≈ N × M × 2（双向）
- 边内存：10^6 × 32 × 2 × 4 = **256 MB**

**层级索引**（每节点平均层数 ≈ 1/ln(M) ≈ 0.42 层）：
- 每层额外存节点 ID 和层号：约 **32 MB**

**总计**：约 **800 MB**（原始向量 + 图结构 + 索引开销）

实际工程中还需考虑对齐 padding、内存分配器开销等，一般在 **1-1.5 GB** 左右。如果使用 FP16 压缩原始向量，可降到 **600-700 MB**。

### Q9：DiskANN 相对于 HNSW 解决了什么问题？核心技术是什么？

**难度**：L3

**参考答案**：

**解决的问题**：HNSW 需要将所有向量和索引图**全部加载到内存**中才能搜索。当数据量达到 10 亿级以上，内存需求达到几百 GB ~ TB 级，单机无法承载。

**核心技术**：

1. **Vamana 图**：DiskANN 自己设计了一种不同于 HNSW 的图结构，特点是：
   - 控制每个节点的出度（R 参数），降低图结构的内存占用
   - 图直径更小，确保搜索跳数更少
   - 建图算法比 HNSW 简单，不需要分层

2. **压缩向量（PQ 量化）**：原始向量用 PQ 压缩后驻留内存，用作粗略距离计算；精确向量存在 SSD 上，仅在候选集精简时读取

3. **SSD 感知的搜索**：
   - 搜索时在内存中用压缩向量做快速距离近似
   - 对压缩距离排序后的候选集，按 batch 从 SSD 读取精确向量做重排
   - 利用 SSD 的顺序读取带宽（远高于随机读取）做批量化访问

**典型性能**：单机 10 亿 128 维向量，内存 32 GB（压缩向量）+ SSD，召回率 95%@10，延迟约 5-10 ms。

### Q10：IVF 和 HNSW 的本质区别是什么？各适合什么场景？

**难度**：L2

**参考答案**：

| | IVF（划分法） | HNSW（图法） |
|---|---|---|
| 核心思想 | 分割空间 → 缩小搜索范围 | 图遍历 → 逐跳逼近目标 |
| 建库方式 | K-Means 聚类 + 倒排 | 逐点插入 + 多层图连接 |
| 搜索方式 | select → probe → merge | 逐层下降 → 局部贪婪搜索 |
| 召回 @高延迟 | 受限于 nprobe 覆盖范围 | 可接近 100% |
| 建库速度 | 快（K-Means 可 GPU 加速） | 慢（逐点插入，串行依赖） |
| 动态增删 | 差（新增数据需重建或影响聚类分布） | 好（天然支持逐点插入） |
| 内存开销 | 低（仅存向量 + 倒排索引） | 高（存图结构边） |
| 高并发 | 好（每个 probe 可并行） | 中等（搜索路径依赖前一步结果） |

**选型建议**：

| 场景 | 推荐 |
|---|---|
| 高召回率 + 低延迟 | HNSW |
| 超大规模 + 内存有限 | IVF + PQ |
| 批量导入 + 离线检索 | IVF（建库快） |
| 实时插入频繁 | HNSW |
| 极低延迟（<1ms） | HNSW |

## 维度三：工程实现（L2-L3）

### Q11：手写一个简化版 IVF 搜索的伪代码

**难度**：L2

```
// ===== 建库阶段 =====
function build_ivf(vectors, nlist):
    // 1. K-Means 聚类，得到聚类中心 centroids (nlist × dim)
    centroids = kmeans(vectors, nlist)

    // 2. 构建倒排索引
    inverted_lists = vector<vector>[nlist]
    for each vector v, id i:
        c_id = argmin( distance(v, centroids) )
        inverted_lists[c_id].push( (v, i) )

    return { centroids, inverted_lists }

// ===== 搜索阶段 =====
function search_ivf(query, index, nprobe, top_k):
    // 1. 计算查询到各聚类中心的距离
    dist_to_centroids = [distance(query, c) for c in index.centroids]

    // 2. 选最近的 nprobe 个聚类
    probe_ids = argsort(dist_to_centroids)[:nprobe]

    // 3. 在每个 probe 桶内暴力搜索
    candidates = []
    for c_id in probe_ids:
        for (v, id) in index.inverted_lists[c_id]:
            d = distance(query, v)
            candidates.push( (d, id) )

    // 4. 全局排序取 Top-K
    sort(candidates by distance)
    return candidates[:top_k]
```

**面试追问**：有没有想过当 nprobe 个桶内总向量数加起来非常多时，这一步反而比暴力搜索还慢？实际的解决方式是什么？

**答**：是的，极端情况下如果 nlist 设置不合理，或者数据分布极度不均匀，某些桶可能包含绝大多数向量。实际系统中有两种应对方式：
1. 建库时对每个桶设置**最大容量限制**，超出则二次聚类（类似 HNSW 的 splitting）
2. 搜索时在打散到桶内之前，先对候选集用 PQ/SQ8 距离做一次粗略过滤，只对 top-N 候选算精确距离

### Q12：PQ（乘积量化）是怎么压缩向量的？M 和 bytes 的关系？

**难度**：L3

**参考答案**：

**压缩流程**（4 步）：

```
原始向量 d=128 维, float32
     ↓
① 分割成 M=8 个子向量, 每个 16 维
     ↓
② 对每个子空间做 K-Means, K=256（8 bits）
     得到 8 个码本, 每个码本 256 个质心
     ↓
③ 每个子向量用最近质心的 ID（1 byte）编码
     原始: 128 × 4 = 512 bytes
     压缩: M × 1 = 8 bytes
     ↓
④ 存储: codebooks + codes（编码向量）
```

**关键关系**：

- **M**：子空间数，也是每个向量的编码字节数
- **K**：每个子空间的聚类中心数，通常 K=256（1 byte 编码）
- **总码本大小**：M × K × (d/M) × sizeof(float) = M × 256 × (d/M) × 4 = 1024d bytes（通常几百 KB，可忽略）
- **压缩率**：原始(M×K×4) / 压缩后(M×1) = 4K = **1024 倍**（K=256 时）→ 实际通常说 **512x → 8 bytes**（d=128,M=8）

**类型对比**：

| 类型 | 每向量字节数 | 召回率 | 特点 |
|---|---|---|---|
| Flat（无压缩）| d × 4 | 基准 | 精确 |
| SQ8 | d × 1 | ~98-99% | 4x 压缩，几乎无损 |
| PQ (M=8) | 8 | ~85-95% | 64x 压缩 |
| PQ (M=16) | 16 | ~90-97% | 32x 压缩 |
| SQ4 | d × 0.5 | ~90-95% | 8x 压缩 |

### Q13：混合搜索（向量 + BM25）怎么做？RRF 的原理和存在的问题？

**难度**：L2

**参考答案**：

**RRF（Reciprocal Rank Fusion）公式**：

```
score(d) = Σ_{r ∈ R} 1 / (k + rank_r(d))
```

其中 R 是各路召回的结果集，rank_r(d) 是文档 d 在第 r 路中的排名，k 是平滑常数（通常 k=60）。

**为什么用 RRF 不用分数融合**：
- 各路召回的打分尺度不同（向量是余弦相似度 0-1，BM25 是 0 到几十），无法直接加权
- RRF 只关心排名，不关心绝对分数，天然跨路可比

**RRF 的问题**：
1. **排名深度假设**：假设所有路的排名意义相同，但实际不同路召回的 Top-1 置信度差异很大
2. **不支持选择性加权**：如果某路召回质量明显好于另一路，RRF 没法给高分加权
3. **对分数分布不敏感**：忽略了候选内部的分数差距

**改进方案**：学习式加权 RRF（为每路分配不同权重 w_r）或使用 DETR 等模型做端到端融合。

## 维度四：系统设计（L3-L4）

### Q15：设计一个支持 10 亿条 128 维向量的搜索服务。需要多少内存？怎么分片？

**难度**：L4

**参考答案**：

**内存估算**（原始向量）：

10^9 × 128 × 4 = **512 GB**（原始 float32 向量）

不同方案的对比：

| 方案 | 索引类型 | 内存需求 | 延迟 | 召回率 | 成本 |
|---|---|---|---|---|---|
| 单机内存 | HNSW | ~1.5 TB | <1ms | ~99% | 极高（需要大内存服务器） |
| 分布式 4 节点 | HNSW | ~400 GB/节点 | 1-3ms | ~99% | 高 |
| 单机 + PQ 压缩 | IVF-PQ | ~16 GB（压缩后） | 5-10ms | ~90% | 低 |
| 单机 + SSD | DiskANN | ~32 GB | 5-15ms | ~95% | 低 |

**分片策略**（3 种主流方案）：

1. **基于哈希的文档分片（Doc-partition）**：按 doc ID 哈希将数据均匀分布到各分片。搜索时 broadcast 到所有分片，各分片返回各自 Top-K，全局合并。
   - 优点：实现简单，负载均衡
   - 缺点：每次搜索都需要查询所有 shard，QPS 受限于最慢分片

2. **基于聚类的索引分片（Index-partition）**：对全局向量做聚类，每个分片负责一部分聚类中心。搜索时先判断落在哪个分片，只查相关分片。
   - 优点：搜索范围小，延迟低
   - 缺点：动态增删复杂，热点分片问题

3. **复制分片（Replication）**：每个分片持有完整索引，请求随机路由。适合读多写少的场景。

**推荐方案**：对于 10 亿规模，建议采用 **Doc-partition + 压缩索引**：
- 4 节点 × 250 GB 原始向量（内存没问题）
- 每节点建 HNSW + PQ，兼顾速度和内存
- 用一致性哈希保证扩容时数据迁移最少

### Q16：如果系统要求 99.9% 召回率 @10，你选什么索引？为什么？

**难度**：L3

**参考答案**：

**首选 HNSW**。

原因：
1. **图方法的召回天花板最高**：HNSW 可以通过增加 efSearch（如从 100 增加到 500+）将召回率推到接近 100%，代价是搜索时间线性增加
2. **IVF 的召回天花板受限于 nprobe 覆盖比**：假设 nlist=4096，桶平均大小 N/nlist。要覆盖到 99.9% 的候选，nprobe 需要大到几乎覆盖所有桶，退化为暴力搜索
3. **HNSW 有理论保证**：在温和假设下，HNSW 的搜索能在 O(log n) 步内收敛到全局最近邻

**实际配置**：
- M = 32（保证图连通性）
- efConstruction = 400（建高质量图）
- efSearch = 500-1000（搜索时扩展候选集）
- 预期延迟 2-5ms（100 万-1000 万规模）

但要注意：HNSW 的内存开销很大，如果内存不足，DiskANN 是更好的选择（也能在高 efSearch 下达到类似召回率）。

### Q17：实时插入对不同索引有什么影响？HNSW 支持动态增删吗？IVF 呢？

**难度**：L3

**参考答案**：

**HNSW**：
- **添加**：天然支持。每插入一个新点，执行一次搜索找到它的近邻，然后建立双向连接。复杂度 O(log n)，与搜索相当。
- **删除**：支持但不优雅。通常用 tombstones（标记删除），需要定期 compact 回收空间。删除后图的连通性可能下降，需要定期重建。
- **逐点插入的建图质量 vs 批量建图**：逐点插入顺序敏感，随机顺序最好；如果数据有序列插入（如按时间），建图质量会退化，需要 shuffle。

**IVF**：
- **添加**：需要更新 coarse quantizer 结果。如果直接将新向量分配到最近的已有聚类中心，**聚类中心本身不会更新**。随着插入量增大，聚类中心与实际数据分布产生偏移（drift），召回率逐渐下降。
- **删除**：理论上支持（从倒排链表中移除），和 HNSW 一样需要 compact。
- **重建策略**：当增删量达到总数据量的 10-20% 时，需要重建 coarse quantizer 和重新分配所有向量。

**DiskANN**：
- **添加**：通过内存中的 buffer（类似于 LSM-tree 的思路），周期性合并到主图
- **删除**：标记删除 + 定期图压缩

**总结**：

| 操作 | HNSW | IVF | DiskANN |
|---|---|---|---|
| 插入 | O(log n) ✅ | O(1) 但需重建 ⚠️ | O(log n) via buffer |
| 删除 | ⚠️ tombstones | ⚠️ 需重建 | ⚠️ tombstones |
| 增删耐受度 | 高 | 低（<10-20%） | 中 |

### Q18：在分布式系统中做向量检索，Doc-partition 和 Index-partition 的优劣？

**难度**：L4

**参考答案**：

| | Doc-partition | Index-partition |
|---|---|---|
| 搜索方式 | 广播到所有分片，各自 Top-K，全局合并 | 路由到部分分片 |
| 跨分片通信 | 高（所有分片参与每次查询） | 低（仅部分分片参与） |
| 负载均衡 | 好（数据均匀分布） | 中（热点聚类中心问题） |
| 扩容/缩容 | 好（一致性哈希） | 差（需要重新聚类） |
| 召回率损失 | 无（所有分片都查了） | 可能丢召回（查了错误分片） |
| 索引异构支持 | 好（各分片可用不同索引） | 差（需统一索引） |
| 实现复杂度 | 低 | 高 |

**工程中的选择建议**：

- 数据量 < 1 亿，QPS < 1000 → **Doc-partition** 简单够用
- 数据量 > 10 亿，QPS > 10000 → **Index-partition** 或混合方案
- 生产实践：Doc-partition + 足够的副本做负载均衡，是目前大多数分布式向量数据库（Milvus、Qdrant）的实际选择

### Q19：如何在向量搜索中加入过滤条件（如"只搜某类别的数据"）？

**难度**：L3

**参考答案**：

有三种主流实现方式：

**方法一：先过滤后搜索（Filter-then-Search）**
- 先根据过滤条件筛选出符合条件的 ID 集合
- 只在集合内做向量搜索
- 优点：结果 100% 满足过滤条件
- 缺点：如果过滤后的集合太小或分布不均匀，向量索引退化
- 适用：过滤条件**高选择性**（过滤掉大部分数据）

**方法二：先搜索后过滤（Search-then-Filter）**
- 先做向量搜索得到 Top-K × oversample_factor 个候选
- 再对候选施加过滤条件，删除不满足条件的
- 如果过滤后不足 K 个，则扩大搜索范围再次尝试
- 优点：搜索速度快，不受过滤条件影响
- 缺点：有可能过滤掉所有候选
- 适用：过滤条件**低选择性**（只排除少量数据）

**方法三：IVF + ID 位图（Bitmap Filtering）**
- 在 IVF 的倒排索引结构中，每个桶额外维护一个位图（bitmap）标记每条向量的属性
- 搜索时在 probe 阶段直接跳过不满足条件的向量
- 优点：过滤和搜索同时进行，延迟最优
- 缺点：实现复杂，内存开销增大（每个属性需要一个 bitmap）

**工程建议**：用**方案二**为主（search-then-filter + oversample），当过滤条件命中率极低时回退到**方案一**。

## 维度五：场景与排查（L2-L3）

### Q20：RAG 场景下 chunk size 和 embedding 维度对检索质量的影响？

**难度**：L2

**参考答案**：

**Chunk Size 的影响**：

| 切得太小 | 切得太大 |
|---|---|
| 语义不完整，单个 chunk 无法回答问题 | 信息被稀释，检索精确度下降 |
| 检索出的片段可能只是"碎片" | 浪费大模型的上下文窗口 |
| 需要更多的 chunk 才能覆盖完整上下文 | 多个主题混在同一个 chunk 中 |

**经验值**：
- 常用 256-512 tokens（约 200-400 个中文字符）
- 递归字符切分（`\n\n` → `\n` → `。` → `，`）优先保持语义完整性
- 引入 10-20% 重叠避免边界断裂

**Embedding 维度的影响**：
- 越小（如 384/512 维）：计算快，内存省，但语义区分能力有限
- 越大（如 1536/3072 维）：语义更丰富，但维度灾难加剧
- 实际效果：并非维度越高越好。对于 RAG 场景，**768-1536 维**是性价比最高的范围。3072 维（如 text-embedding-3-large）在特定高区分度场景有效，但一般场景收益不明显。

### Q21：用户反馈搜索结果不准，排查思路是什么？

**难度**：L2

**参考答案**：

系统化的排查流程（从外到内）：

```
① 检查 Embedding 质量
   ├── 用户问题本身是否清晰？→ 问句改写
   ├── Embedding 模型是否适合该领域？→ 抽样手动检查语义相似度排序
   └── 有无 out-of-domain 问题？

② 检查检索召回情况
   ├── 召回率测试（Recall@K）：用标准测试集评估
   ├── 是否是 nprobe/efSearch 参数太低？→ 增大参数看召回是否提升
   └── 检索到的文档是否和问题相关？→ 看原始距离分数分布

③ 检查 Re-rank 效果
   ├── Cross-Encoder 是否真的改善了排序？
   └── 有没有把相关文档排到后面？

④ 检查最终生成质量
   ├── Faithfulness：模型有没有曲解检索到的文档？
   └── Answer Relevance：模型回答是不是跑题了？

⑤ 检查知识库本身
   ├── 知识库是否覆盖了这个问题？
   ├── 文档本身有没有过时或错误？
   └── Chunk 切分是否合理？
```

**黄金法则**：用 RAGAS 或人工标注的测试集跑一套量化指标（Recall@K、Faithfulness、Answer Relevance），用数据说话，不要靠感觉。

### Q22：Cold Start 问题——新数据刚入库，用户就搜不到，怎么解决？

**难度**：L3

**参考答案**：

冷启动问题在不同索引下有不同表现：

**问题本质**：
- **IVF**：新向量分配到已有聚类中心，但 coarse quantizer 未更新，新数据不会被分到最优桶，召回率下降
- **HNSW**：新点插入后立即可达，搜索质量不受影响（HNSW 天然解决冷启动）
- **DiskANN**：新数据先存在 buffer 中，在合并到主图之前，搜索可能 miss

**解决方案**：

1. **双缓冲区策略（Double Buffer）**：维护一个小的"新鲜索引"（纯内存、暴力搜索或小 HNSW）和一个大的"主索引"。新数据写入新鲜索引，查询时同时查两个索引合并结果。当新鲜索引积累到一定量（如总数据的 5%），触发一次合并操作，将新鲜索引合并到主索引中。

2. **软删除 + 懒重建**：对 IVF，标记老数据为"过时"，在新数据达到阈值时重建所有聚类。

3. **增量索引 + 周期性重建**：参考 Lucene/Milvus 的分段合并策略（segment merge）。

**生产实践**：多数系统采用 **方案一**（双缓冲），对用户透明，延迟可控。

### Q23：数据分布极不均匀（如某些类别的向量数量是其他类别的 1000 倍），对搜索有什么影响？怎么处理？

**难度**：L3

**参考答案**：

**影响**：

1. **IVF**：热门类别的聚类桶可能包含绝大多数向量，导致 nprobe 不起作用（探测热门桶后冷门桶未覆盖，或探测冷门桶后热门桶只搜了一小部分）
2. **HNSW**：热门区域的节点度很高（连接了大量节点），冷门区域节点度很低，图结构不平衡，搜索路径容易偏向热门区域

**处理方法**：

```
① IVF：平衡 K-Means
   在 K-Means 训练时引入容量约束，限制每个桶的最大向量数
   对超容桶做二次拆分 → 子聚类

② HNSW：动态度约束
   监控每个节点的出度，对热门节点主动剪枝（减少无用边）
   Mmax 参数限制最大度，超过后保留最近的 Mmax 个邻居

③ 两阶段搜索
   先识别查询所属类别，再在该类别内搜索
   跨类别查时，按类别分别取 Top-K 后合并
```

### Q25：你觉得当前向量数据库领域最大的技术挑战是什么？

**难度**：L4

**参考答案**（开放性，以下是几种有争议的观点）：

**挑战一：流式重建（Streaming Rebuild）**
- 痛点：现有索引策略随着增量插入，质量逐步退化。HNSW 的图连通性下降，IVF 的聚类偏移。
- 理想状态：在不阻塞读写的前提下，对索引做无缝重建。
- 现状：Milvus 的 compact 会阻塞写入；Qdrant 的 segment merge 有短暂延迟。
- 关键问题：如何设计一个真正**无锁的流式索引重建**算法？

**挑战二：多模态统一检索**
- 痛点：文本、图像、音频、视频的向量分属不同 Embedding 空间，无法直接对比。
- 现状：分别建索引、分别检索、再融合 —— 不是真正意义上的多模态检索。
- 理想：一个统一的度量空间，所有模态可相互检索。CLIP 做了文本-图像的跨模态，但还远不够。

**挑战三：极低成本索引**
- 痛点：10 亿向量用 HNSW 需要 TB 级内存，大部分中小企业用不起。
- 方向：压缩索引 + SSD 分层（DiskANN 思路的极致优化），目标是**用 16GB 内存 + 普通 SSD 检索 10 亿向量**。

**挑战四：检索与生成的深度协同**（Search-Generation Co-design）
- 痛点：RAG 中检索和生成是解耦的两阶段，检索器不知道最终生成需要什么，生成器只能被动接受检索结果。
- 方向：可微分的端到端检索-生成联合优化；LLM 的注意力信号可以反向指导检索。

### Q26：我们这个项目里自己实现了 IVF/HNSW/DiskANN/BM25，你怎么理解"从零实现"和"调库"之间的取舍？

**难度**：L4

**参考答案**：

> 这道题没有标准答案，考察候选人的工程价值观。

**从零实现的价值**：
- 深入理解算法原理——调一万次库不如自己写一遍
- 可控性强——可以针对特定场景做定制优化（如我们的 BM25 自定义分词器）
- 不依赖第三方库的维护状态——Faiss 的某些功能多年未更新，自己实现可以与时俱进

**"站在巨人的肩膀上"的价值**：
- 避免重复造轮子——Faiss 团队花数年优化的 SIMD 距离计算、多线程调度，个人项目很难超越
- 稳定性——Faiss 在生产环境中被验证过数十亿次
- 生态兼容——Milvus/Qdrant 等数据库已经做好了分布式、持久化、监控等

**最佳实践的决策框架**：

```
要解决的问题           → 建议策略
──────────────────────────────────────────────────
学习/教学/理解算法     → 从零实现（如本项目）
核心差异化竞争力       → 自己实现（如 BM25 分词器）
标准化操作             → 调库（如 L2 距离计算）
算法仍在快速发展       → 调库 + 跟进学术界进展
```
