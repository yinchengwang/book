# ART 自适应基数树

## 概述

ART（Adaptive Radix Tree）是一种**自适应**的基数树，根据 key 前缀长度自动选择最优的节点类型，在空间和性能间取得平衡。

## 设计背景

- 2013 年由 Viktor Leis 等人提出
- 用于**内存数据库**索引
- 被 HyPer 数据库采用

## 节点类型

ART 使用 4 种节点类型，根据子节点数量自适应切换：

| 类型 | 子节点数 | 空间 | 适用场景 |
|------|----------|------|----------|
| N4 | 4 | 紧凑 | 前 2 位唯一 |
| N16 | 16 | 中等 | 前 4 位唯一 |
| N48 | 48 | 较大 | 前 6 位唯一 |
| N256 | 256 | 最大 | 前 8 位唯一 |

```
N4 节点结构:
┌──────┬─────┬──────────┐
│ Type │Cnt=2│ Keys[4]  │
├──────┴─────┼──────────┤
│            │Children[4]│
└────────────┴──────────┘

N256 节点结构:
┌──────┬─────┬──────────────┐
│ Type │Cnt=2│Children[256]│
└──────┴─────┴──────────────┘
```

## 与传统基数树的对比

| 维度 | ART | 传统基数树 |
|------|-----|-----------|
| 空间效率 | 自适应，节省空间 | 固定大小，浪费空间 |
| 节点类型 | 4 种（N4/N16/N48/N256） | 单一 |
| 性能 | 接近二叉搜索树 | 较慢 |

## 数据结构

```c
typedef struct art_node {
    art_node_type_t type;    // 节点类型
    uint32_t count;          // 子节点数量
    uint32_t prefix_len;     // 压缩前缀长度
    uint8_t *prefix;         // 压缩前缀
} art_node_t;

// N4 节点
typedef struct {
    art_node_t node;
    uint8_t keys[4];
    struct art_node *children[4];
} art_n4_t;

// N256 节点
typedef struct {
    art_node_t node;
    struct art_node *children[256];
} art_n256_t;
```

## 核心操作

### 查找
```
key = [0x01, 0x02, 0x03, 0x04]

1. 从根节点开始
2. 检查压缩前缀（如果有）
3. 读取下一个字节，选择对应子节点
4. 重复直到找到叶子或未命中
```

### 插入
1. 沿 key 路径导航
2. 匹配前缀
3. 选择子节点或创建新叶子
4. 若子节点数超过阈值，**切换到更大的节点类型**

### 节点切换
```
N4 节点子节点超过 4 个 → 切换到 N16
N16 节点子节点超过 16 个 → 切换到 N48
N48 节点子节点超过 48 个 → 切换到 N256
```

## 操作复杂度

| 操作 | 时间复杂度 | 说明 |
|------|------------|------|
| 查找 | O(k) | k 为 key 长度 |
| 插入 | O(k) | 含可能节点切换 |
| 删除 | O(k) | |

## 应用场景

| 系统 | 用途 |
|------|------|
| HyPer | 内存数据库索引 |
| Peloton | 试验性数据库 |
| 键值存储 | 高性能内存索引 |

## 面试要点

1. **ART 的自适应体现在哪里？** - 根据子节点数量自动选择节点类型
2. **ART 有哪几种节点类型？** - N4、N16、N48、N256
3. **ART 与 B+Tree 的对比？** - ART 更紧凑，内存效率更高
4. **ART 的压缩前缀是什么？** - 多个节点共享的前缀，减少比较次数
