# 网络命名空间工程对照笔记

## 容器网络基础场景

在容器化部署场景中，网络命名空间是实现容器间网络隔离的核心机制。每个容器运行在自己的网络命名空间中，拥有独立的网络视图：

1. **独立网络栈**: 容器拥有自己的网络接口、IP地址、路由表、防火墙规则
2. **端口隔离**: 不同容器可以绑定相同的端口（如都绑定 80 端口）而不会冲突，因为它们在不同的网络命名空间中
3. **网络策略隔离**: 一个容器内的 iptables 规则不会影响其他容器

典型的基础场景包括：
- Web 服务容器：每个容器运行独立的 Nginx/Apache 实例，监听相同的端口
- 微服务架构：不同服务容器化部署，通过内部网络通信
- 数据库容器：独立的数据库实例，与应用容器网络隔离

## Kubernetes Pod 网络隔离

Kubernetes 架构中，Pod 是最小的调度单元，而网络命名空间在其中扮演关键角色：

**Pod 网络模型**:
- 每个 Pod 分配独立的网络命名空间（通过 pause 容器实现）
- Pod 内所有容器共享该网络命名空间，因此：
  - 同一 Pod 内的容器可以使用 localhost 相互通信
  - 共享相同的 IP 地址和网络端口空间
  - 同一个 Pod 内的多个容器可以绑定相同端口

**Pod 间网络隔离**:
- 不同 Pod 属于不同的网络命名空间，完全隔离
- Pod 间通信通过网络插件（CNI）配置的虚拟网络
- 命名空间级别的网络策略可以控制 Pod 间的访问

**Kubernetes 网络隔离机制**:
- NetworkPolicy 资源定义 Pod 级别的网络访问控制
- CNI 插件（如 Calico、Cilium）负责实现网络命名空间配置
- Service 和 Ingress 资源提供 Pod 间的负载均衡和服务发现

## Docker/容器网络原理

Docker 容器网络的实现依赖于 Linux 网络命名空间和相关网络技术：

**Docker 网络驱动**:
1. **bridge (桥接模式)**:
   - 创建 docker0 网桥
   - 每个容器有独立的网络命名空间
   - 容器通过 veth pair 连接到 docker0 网桥
   - 通过 NAT 实现容器与外部网络的通信

2. **host (主机模式)**:
   - 容器直接使用主机的网络命名空间
   - 无网络隔离，共享主机的网络栈

3. **overlay (覆盖网络)**:
   - 跨主机容器网络
   - 使用 VXLAN 隧道技术
   - 需要 Docker Swarm 或 Kubernetes 支持

4. **macvlan**:
   - 容器拥有真实的 MAC 地址
   - 容器像物理机一样直接连接到网络

**容器网络工作原理**:
```
宿主机
  ├── eth0 (物理网卡)
  ├── docker0 (网桥, 172.17.0.1/16)
  └── vethxxx (veth pair 端点)
       │
       └── netns:container1 (容器网络命名空间)
             ├── vethyyy (veth pair 容器端, 172.17.0.2/16)
             └── lo (loopback)

容器访问外部网络流程:
  容器内请求 → veth → docker0 → NAT → eth0 → 外部网络
```

**容器网络安全**:
- 容器间默认网络隔离
- 可以通过网络策略控制访问
- 敏感操作需要 capabilities 限制
- 用户命名空间（user namespace）可进一步增强安全性

## 工程实践要点

1. **网络隔离验证**: 使用 `ip netns` 命令验证容器网络命名空间配置
2. **性能考虑**: 网络命名空间有一定开销，高性能场景需评估影响
3. **调试工具**: `nsenter`、`ip netns exec` 是调试容器网络的重要工具
4. **监控指标**: 关注网络命名空间相关的网络延迟和吞吐量指标
