# MVCC 在 bufmgr.c 中的实现

## 概述

MVCC（Multi-Version Concurrency Control，多版本并发控制）是 PostgreSQL 存储引擎实现事务隔离的核心机制。在 bufmgr.c 的缓冲区管理器中，MVCC 通过元组头部的特殊字段实现多版本管理，允许读操作不阻塞写操作，从而提供高效的并发控制。

## 核心字段

### t_xmin（创建事务ID）

`t_xmin` 字段记录创建该元组版本的事务ID。当一个事务插入新元组时，存储引擎将当前事务ID写入该字段。这个字段用于判断元组版本是否对当前查询可见——只有当创建事务已经提交（或对当前事务可见）时，该版本才是可见的。

在缓冲区页面中，每个元组头部都包含这个字段。页面首次分配给元组使用时，`t_xmin` 被设置为当前事务ID，之后不再改变（除非元组被复用）。

### t_xmax（删除事务ID）

`t_xmax` 字段记录删除或更新该元组版本的事务ID。当事务删除或更新一个元组时，存储引擎将当前事务ID写入该字段。如果 `t_xmax` 为 0，表示该版本当前未被任何事务标记为删除。

这个字段的存在使得"已删除"的元组版本仍然可以被其他事务的快照看到，只要删除事务尚未提交。这是快照隔离级别的基础保证。

### t_chain（版本链指针）

`t_chain` 是指向元组旧版本的指针。在 PostgreSQL 的实现中，当元组被更新时，旧版本通过 `t_chain` 链接到新版本，形成一个链表结构。每个元组版本只知道自己的"下一代"版本，而不知道"上一代"版本。

这个链表从新到旧排列，读取时从最新版本开始遍历，直到找到对当前快照可见的版本为止。如果所有版本都不可见，说明元组对当前事务不存在（已被其他事务删除）。

## 可见性判断规则

### 快照隔离级别

在 PostgreSQL 的默认隔离级别（读已提交）下，可见性判断需要考虑当前快照中的活跃事务列表：

1. **t_xmin 可见性**：创建元组的事务必须是已提交的，或者是当前事务本身。如果创建事务在当前快照中标记为活跃，则该版本不可见。

2. **t_xmax 可见性**：如果 `t_xmax > 0`，需要判断删除事务是否已提交。只有当删除事务已提交时，元组版本才被视为已删除。如果删除事务尚未提交（或在当前快照中活跃），该版本仍然可见。

3. **组合判断**：`version_visible = (t_xmin_visible) && (!t_xmax_exists || !t_xmax_visible)`

### 多版本遍历

当访问一个元组时，缓冲区管理器首先获取页面的指定元组 slot，然后从该 slot 开始沿版本链向下遍历，直到找到第一个对当前快照可见的版本。这个过程需要：

- 持有页面级别的锁（防止并发修改）
- 为每个版本执行可见性判断
- 可能需要遍历多个版本（取决于并发更新的程度）

## 与缓冲区管理的集成

### 页面结构

MVCC 元组存储在缓冲区页面的固定格式中。每个页面包含页面头部、元组头部和元组数据。页面头部维护页面级的统计信息和空闲空间指针。元组头部包含 MVCC 字段（t_xmin、t_xmax、t_chain）以及元组大小、属性信息等。

### 缓冲区原语

bufmgr.c 提供了原语级别的页面和元组操作原语：

- `ReadPage(page_id)`：读取页面到缓冲区
- `GetTuple(page, offset)`：获取页面中指定偏移的元组
- `MarkDirty(page)`：标记页面为脏页
- `UnpinPage(page)`：释放页面锁

这些原语与 MVCC 逻辑结合，为上层执行器提供一致的多版本访问接口。

### 写时复制（Copy-on-Write）

当事务更新元组时，不会直接修改原有版本，而是分配新版本。旧版本保留在原位置（可能被其他事务的快照看到），新版本写入页面空闲空间或新页面。这避免了读操作被写操作阻塞。

## GC 清理机制

### 垃圾版本识别

随着事务推进，会产生越来越多的旧版本。这些旧版本对所有活跃快照都不可见，成为垃圾数据。识别垃圾版本的条件：

1. `t_xmax > 0`：版本已被删除
2. `t_xmin < oldest_active_xmin`：创建事务已结束
3. `t_xmax < oldest_active_xmin`：删除事务已结束

只有同时满足这三个条件，版本才能被安全回收。

### Vacuum 清理

PostgreSQL 的 vacuum 进程负责清理垃圾版本。清理过程：

1. 扫描表的所有页面
2. 识别满足清理条件的元组版本
3. 移动有效元组以压缩空间
4. 更新页面空闲空间指针
5. 更新可见性映射

### 对缓冲区的影响

GC 清理需要小心处理缓冲区中的脏页。如果被清理的版本恰好在缓冲区中，需要确保页面不会被意外修改。清理操作通常需要持有排他锁，并可能触发页面写回。

## 总结

bufmgr.c 中的 MVCC 实现通过 t_xmin、t_xmax 和 t_chain 三个核心字段，配合快照可见性判断逻辑，实现了高效的并发控制。多版本机制使得读操作可以访问一致的历史快照，而不受并发写操作的影响。版本链结构支持细粒度的并发更新，而 GC 机制则保证了存储空间的及时回收。
