# 两阶段提交（2PC）- 工程对照

## 本学习卡演示的 2PC 协议在工程轨的实现对照

### 1. 分布式事务背景

2PC（Two-Phase Commit）是分布式系统中保证跨节点事务一致性的核心协议。

### 2. 协议流程

**第一阶段：Prepare**
```
协调者 → 参与者: "可以提交吗？"
参与者 → 协调者: "可以/不可以"
参与者: 写入 Prepare Log（持久化）
```

**第二阶段：Commit/Rollback**
```
如果全部可以：
  协调者 → 参与者: "提交"
  参与者: 提交，释放锁，写入 Commit Log
否则：
  协调者 → 参与者: "回滚"
  参与者: 回滚，释放锁
```

### 3. 工程实现要点

**日志持久化**（关键）：
- Prepare Log 必须先于任何操作落盘
- Commit Log 写入后才算提交成功
- 恢复时根据日志状态决定后续操作

**故障恢复流程**：
| 日志状态 | 恢复动作 |
|---------|---------|
| 无日志 | 视为未开始 |
| Prepare Log | 等待协调者指令或查询其他参与者 |
| Commit Log | 已提交，无需操作 |
| Abort Log | 已回滚，无需操作 |

### 4. 局限性

1. **协调者单点故障**：崩溃后参与者阻塞等待
2. **同步阻塞**：Prepare 阶段参与者持有锁
3. **脑裂风险**：网络分区可能导致不一致

**解决方案**：三阶段提交（3PC），引入超时机制和预提交状态

### 5. 工程对照建议

工程轨的分布式演进模块：

| 模块 | 文件位置 | 与 2PC 的关系 |
|------|---------|---------------|
| **Raft 共识** | `engineering/src/db/consensus/raft.c` | Raft 的 Log Replication 阶段类似 2PC 的 Prepare-Commit 流程 |
| **分片路由** | `engineering/src/db/sharding/sharding.c` | 多节点事务需要先确定目标分片，再执行协调 |

**Raft vs 2PC 对照**：

| 2PC 概念 | Raft 实现 |
|---------|----------|
| 协调者 | Leader 节点 |
| 参与者 | Follower 节点 |
| Prepare 投票 | AppendEntries 请求 |
| Commit 提交 | 日志 commit 后 apply |
| 故障恢复 | Leader election + 日志恢复 |

**关键区别**：
- Raft 通过 leader election 解决协调者单点问题
- Raft 是非阻塞的（异步复制）
- 2PC 更简单，适合低延迟场景

### 6. 学习扩展

- 3PC（三阶段提交）如何解决 2PC 的阻塞问题
- Paxos/Raft 共识算法与 2PC 的区别
- SAGA 模式：长事务的补偿机制

---

## 总结

本学习卡通过模拟协调者和参与者，展示了 2PC 协议的核心流程和故障恢复机制。关键理解点：
1. Prepare 阶段必须持久化（Prepare Log）
2. 协调者根据 Prepare 结果决定提交或回滚
3. 恢复时根据日志状态决定后续操作
4. 2PC 有协调者单点故障和同步阻塞的局限性
