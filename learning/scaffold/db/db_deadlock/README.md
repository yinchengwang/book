# 数据库死锁处理学习

## 简介

本模块演示数据库死锁处理的核心机制，包括：

- **等待图（Wait-For Graph）**：事务间等待关系的可视化表示
- **死锁检测算法**：通过环检测发现死锁
- **超时机制**：基于时间的死锁解决策略
- **牺牲者选择**：死锁回滚时的策略选择

## 死锁的四个必要条件

```
┌─────────────────────────────────────────────────────────┐
│  Coffman 条件（死锁必要条件）                            │
├─────────────────────────────────────────────────────────┤
│  1. 互斥条件    - 资源每次只能被一个事务占用              │
│  2. 持有并等待  - 事务持有资源同时请求其他资源             │
│  3. 不可抢占    - 已分配资源不能被强制夺取                │
│  4. 循环等待    - 形成事务间的等待环                      │
└─────────────────────────────────────────────────────────┘
```

## 等待图原理

等待图是有向图：
- **节点**：代表事务（T0, T1, T2...）
- **边**：T1 → T2 表示 T1 等待 T2 释放资源

```
示例：T0 持有 R0 等待 R1，T1 持有 R1 等待 R0

    R0 ──占有──┐     ┌──占有── R1
               │     │
               ▼     ▼
    ┌──────────────┐ ┌──────────────┐
    │     T0       │ │     T1       │
    └──────┬───────┘ └───────┬──────┘
           │                 │
           └───等待───► T1   └───等待───► T0
```

当等待图存在环时，说明发生了死锁。

## 死锁处理策略

### 1. 死锁预防（Prevention）
- **一次性申请**：事务开始时申请所有需要的资源
- **资源排序**：按固定顺序申请资源，避免循环
- **层次锁定**：将资源分层，只能同层申请

### 2. 死锁检测（Detection）
- 定期或在特定时机检查等待图
- 检测到环即发现死锁
- 常用算法：DFS 环检测、WFG 时序检测

### 3. 死锁恢复（Recovery）
- **牺牲者选择**：选择回滚代价最小的事务
  - 持有资源最少
  - 等待时间最长
  - 已完成工作量最少
- **回滚策略**：部分回滚 vs 完全回滚

### 4. 超时机制（Timeout）
- 简单但可能误伤长时间运行的事务
- 需要合理设置超时阈值

## 学习要点

1. **理解等待图**：等待图是检测死锁的核心数据结构
2. **环检测算法**：DFS 深度优先搜索可有效检测环
3. **权衡策略**：检测频率 vs 系统开销
4. **牺牲者选择**：没有最优策略，需要根据场景选择
