# 乐观并发控制 (Optimistic Concurrency Control)

## 简介

乐观并发控制（OCC）是一种并发控制策略，**假设事务冲突较少**，因此先乐观地执行操作，在提交阶段才检测是否发生冲突。

与悲观锁（如两阶段锁 2PL）不同，乐观锁在**读取时不加锁**，只在提交时校验数据的版本号。如果版本号在读取和提交之间发生变化，说明存在并发冲突，此时事务需要**回滚并重试**。

## 核心三阶段

```
┌─────────────────────────────────────────────────────────────┐
│                      乐观锁工作流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────┐    ┌─────────────┐    ┌──────────────┐        │
│  │  读取    │ -> │    验证     │ -> │    写入      │        │
│  │ Read    │    │  Validate   │    │   Write      │        │
│  └─────────┘    └─────────────┘    └──────────────┘        │
│       │              │                    │                 │
│   记录版本号      再次检查版本         版本匹配则写入         │
│                                                             │
│  版本一致 ──────────────> 执行写入                           │
│  版本冲突 ──> 回滚并重试                                     │
└─────────────────────────────────────────────────────────────┘
```

## 适用场景

- **低冲突率**：冲突较少的业务场景（如多数用户只读、偶尔更新）
- **读多写少**：读操作远多于写操作的场景
- **短事务**：事务执行时间短，冲突概率低

## 不适用场景

- **高冲突率**：频繁并发修改同一数据的场景（乐观锁会导致大量重试）
- **长事务**：执行时间越长，期间被其他事务修改的概率越高

## 版本号实现方式

1. **递增整数版本号**：每次更新时 `version++`
2. **时间戳**：使用最后修改时间（精度有限）
3. **CAS（Compare-And-Swap）**：硬件级别的原子操作

## 编译与运行

```bash
# 编译
gcc -std=c11 -Wall -o optimistic main.c

# 或使用 Makefile
make

# 运行
./optimistic

# 运行测试
make test
```

## 程序输出说明

程序包含三个测试场景：

1. **测试1 - 无冲突场景**：正常转账，无并发干扰，验证 OCC 基本流程
2. **测试2 - 冲突检测**：模拟外部事务修改数据，验证版本冲突检测
3. **测试3 - 多轮冲突**：连续被外部修改，验证重试机制

- [ ] 理解冲突检测原理（提交时比较版本号）
- [ ] 了解重试机制的实现与注意事项（最大重试次数、退避策略）
- [ ] 分析乐观锁的适用场景与局限性

## 扩展思考

1. **与 MVCC 的关系**：MVCC（多版本并发控制）是一种更通用的技术，乐观锁是其应用场景之一
2. **与悲观锁的对比**：何时选择乐观锁？何时选择悲观锁？
3. **重试策略**：固定重试次数 vs 指数退避
4. **饥饿问题**：频繁冲突可能导致某些事务永远无法提交
