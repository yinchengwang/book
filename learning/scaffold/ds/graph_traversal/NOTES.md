# 图遍历在工程中的应用（工程对照说明）

## 1. BFS/DFS 在工程中的映射

### BFS - 广度优先搜索

在数据库查询优化器中，BFS 用于探索查询计划的等价变形空间。优化器从根节点（逻辑计划）出发，按层次展开物理实现（SeqScan、IndexScan、HashJoin 等），优先找到代价最小的计划。Redis 集群的节点发现也采用 BFS，从已知节点出发，逐层探测新节点，维护集群拓扑。

### DFS - 深度优先搜索

DFS 的递归后序在编译器中用于 AST 的语义分析。函数调用链的回溯、依赖图的循环检测都依赖 DFS。数据库死锁检测算法（Wait-Die/Wound-Wait）本质上是对事务等待图的 DFS，检测环即发现死锁。

## 2. 连通分量在工程中的映射

### 数据库索引结构

B+Tree 的叶节点链表天然将所有数据页组织成一个连通结构，而 B-Tree 的分裂路径形成树状连通分量。分区表的每个分区是独立的连通分量，全局索引需要跨越多个分区搜索。

### 分布式系统

Redis Cluster 将槽位（slot）按区间划分，每个槽位内部节点形成连通分量。节点故障时，通过 Gossip 协议探测并重组连通分量，恢复服务。

## 3. 拓扑排序在工程中的应用

### 任务调度系统

数据库的查询计划本质是一个 DAG（算子树），拓扑排序决定算子的执行顺序。Spark 的 Stage 划分也基于 DAG 的拓扑序：每个 Stage 内部无依赖，可并行执行。

### 依赖管理

Cargo/Gradle/Maven 的依赖解析等价于拓扑排序问题。编译顺序（Makefile）、Docker 层构建顺序都依赖拓扑排序确保依赖在被依赖者之前完成。

### 数据库 MVCC

PostgreSQL 的 Vacuum 和 HOT 更新（Heap-Only-Tuple）需要按页面版本链的依赖关系进行清理，实质是对元组版本图的拓扑排序。

## 4. 工程实现要点

- **性能优化**：使用邻接表而非矩阵节省空间，使用显式栈避免递归深度溢出
- **环检测**：有向图的拓扑排序可判断是否存在环，DFS 变体更高效
- **并发安全**：多线程任务调度时，拓扑排序结果需加锁保护或使用无锁队列
