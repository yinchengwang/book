# 备份恢复 · 学习笔记

## 核心概念

1. **冷备**：停服后复制数据文件，简单一致但影响可用性
2. **热备**：运行时备份，需要 WAL 日志确保一致性
3. **全量备份**：完整数据集副本，恢复快但存储大
4. **增量备份**：自上次全量/增量以来的变化，节省空间
5. **PITR（时间点恢复）**：连续 WAL 归档后可将数据库恢复到任意时间点

## 备份对比

| 类型 | RTO（恢复时间） | RPO（数据丢失） | 存储成本 |
|------|---------------|----------------|---------|
| 全量备份 | 短（直接恢复） | ≤ 上次备份 | 高 |
| 增量备份 | 长（需先恢复全量） | ≤ 上次备份 | 低 |
| WAL 归档 | 中等 | 秒级 | 中等 |
| 冷备 | 短 | ≤ 上次冷备 | 高 |
| 热备 | 短 | ≤ WAL 归档点 | 高 |

## 工程对照

灾备策略是数据库系统的最后一道防线。本项目的 `engineering/src/db/core/wal.c`
实现了 Write-Ahead Logging（WAL），通过 `wal_write()` 在数据写入前记录日志。
`wal_recover()` 在启动时重放 WAL 日志恢复未刷盘的已提交事务。与本文
Python 文件模拟不同的是，工程实现的 WAL 需要处理页面级别的 redo/undo、
LSN（日志序列号）的严格线性增长、Checkpoint 与 WAL 的截断协调、
以及崩溃恢复时的事务回滚（未提交事务的 undo）。
