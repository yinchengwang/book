# 分库分表 · 学习笔记

## 核心概念

1. **水平分片（Horizontal Sharding）**：同一张表的不同行分布在不同节点，结构一致
2. **垂直分片（Vertical Sharding）**：按列拆分到不同表或节点，每张表结构不同
3. **Hash 分片**：`hash(shard_key) % N` 均匀分布，但扩缩容时需迁移大量数据
4. **范围分片**：按时间或 ID 范围划分，扩缩容易但可能数据倾斜
5. **分布式事务**：跨分片的写入需要两阶段提交（2PC）或 Saga 模式保障一致性

## 分片对比

| 策略 | 均匀性 | 扩缩容 | 查询支持 |
|------|--------|--------|---------|
| Hash 分片 | ✓ 均匀 | ✗ 难（数据迁移） | 仅等值查询 |
| 范围分片 | ✗ 可能倾斜 | ✓ 易 | 范围查询友好 |
| 列表分片 | 可控 | 中等 | 按区域/分类 |

## 工程对照

分布式数据库的分库分表机制是本项目 `engineering/src/db/shard/` 中实现的目标。
`shard.c` 的路由表维护分片映射关系，通过 `shard_route(key)` 确定目标节点。
与本文的 Python 模拟不同，工程实现需要处理分布式查询的计划分片（下推
到各分片并行执行后再合并）、分片节点探活、重试和故障转移，以及跨分片
事务的 2PC 协调。
