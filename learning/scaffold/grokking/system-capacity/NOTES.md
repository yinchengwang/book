# 容量估算 学习笔记

## 核心概念

- **QPS (Queries Per Second)**: 每秒请求数，系统吞吐量的核心指标
- **TPS (Transactions Per Second)**: 每秒事务数，适用于数据库/支付场景
- **峰值因子**: 一般按日均 QPS 的 5-10 倍估算峰值
- **存储成本**: 以年为单位计算数据增长，副本系数 × 压缩比

## 关键公式

| 指标 | 公式 | 示例 |
|------|------|------|
| 日均 QPS | DAU × 人均操作 / 86400 | 1 亿×200/86400≈231,481 |
| 峰值 QPS | 日均 QPS × 峰值因子 | 231,481×5≈1,157,407 |
| N 年存储 | 单条大小×日增量×365×N | 512B×5亿×365×5≈445TB |
| 带宽 | 响应大小×8×峰值QPS | 2KB×8×1.15M≈18.4Gbps |

## 工程对照

`engineering/` 轨中的数据库存储引擎使用了容量规划的直接实践。`engineering/src/db/buf/bufmgr.c` 中的 Buffer Pool 大小由 `shared_buffers` GUC 参数控制（默认 128MB），这就是容量规划在数据库内核中的体现——从磁盘到内存的容量映射需要精确计算。`engineering/include/db/guc.h` 中定义的参数（`work_mem`、`max_connections`、`shared_buffers`）提供了运行时容量调整的能力，这对应于系统设计中"预留扩展空间"的原则。`engineering/src/db/core/initdb.c` 在集群初始化时分配预写日志 (WAL) 空间，同样是容量规划的具体实施——WAL 段大小和数量需根据预估的写入量提前配置，设计不当会导致频繁的日志切换或磁盘空间不足。

## 面试要点

1. 面试中注重"估算思路"而非精确数值，合理假设要比精确计算更重要
2. 常用基准: 单机 MySQL ~1k QPS, Redis ~50k QPS, Nginx ~10k QPS
3. 存储计算需注意副本和压缩，带宽需考虑峰值而非均值
