# 机器学习基础 · 学习笔记

## 核心概念

- **监督学习**：使用带标签的数据训练模型，包括回归（预测连续值）和分类（预测离散类别）
- **无监督学习**：使用无标签数据发现隐藏结构，如聚类、降维
- **过拟合**：模型过于复杂，记住训练数据噪声而非真实模式 → 高方差，低偏差
- **欠拟合**：模型过于简单，无法捕捉数据的真实模式 → 高偏差，低方差
- **训练/测试集划分**：评估模型泛化能力的关键实践

## 工程对照

ML 基础概念在本仓库中有直接对应。以 `guc.c`（`engineering/src/db/core/guc.c`）为例，数据库的配置参数调优本质上就是一个超参数选择问题：

- **shared_buffers 的选择**：太小则欠拟合（Buffer Pool 无法缓存热点页面，I/O 瓶颈），太大则过拟合（过度占用系统内存，在其他 workload 上表现差）
- **work_mem 的平衡**：排序和哈希操作的内存上限，类比 ML 中的正则化参数——设定过紧限制算法能力（欠拟合），设定过松允许单个查询消耗过多资源（过拟合到特定查询模式）
- **wal_level 的权衡**：日志级别设置相当于 ML 中的模型复杂度选择。`minimal` 级别（欠拟合）丢失了恢复能力，`logical` 级别（过拟合）带来了巨大开销

这种「配置即超参数」的视角帮助理解为什么 PostgreSQL 的默认配置在生产环境中往往不够——默认配置好比一个在「通用训练集」上调优的模型，对于特定 workload（测试集）需要进一步 fine-tune。

此外，本仓库测试体系的分层设计（`engineering/test/` 下的单元测试 → 集成测试 → 端到端测试）与 ML 中的交叉验证思想异曲同工：单元测试验证单个组件（如单次 fold 训练）、集成测试验证组件交互（如跨 fold 稳定性）、端到端测试验证整体能力（如最终测试集评估）。本仓库 `test` 目录下针对不同场景（正常路径、边界路径、异常路径）设计测试用例，本质上就是在构建一个有代表性的验证集。
