# Big O 分析

## 概述

Big O notation 描述算法性能随输入规模增长的变化趋势，是评估算法效率的标准工具。

## 核心复杂度等级

| 复杂度 | 名称 | 增长特征 | 代表算法 |
|--------|------|----------|----------|
| O(1) | 常数 | 不随 n 变化 | 数组访问 |
| O(log n) | 对数 | 每次减半 | 二分查找 |
| O(n) | 线性 | 成正比 | 线性搜索 |
| O(n log n) | 线性对数 | 接近线性 | 高效排序 |
| O(n²) | 平方 | 双重循环 | 冒泡排序 |
| O(2^n) | 指数 | 翻倍增长 | 递归斐波那契 |

## 工程对照

在实际生产代码中，算法复杂度直接影响系统性能和资源消耗。

**数据库查询优化**: 数据库引擎在执行查询时，需要选择最优的执行计划。例如，对于 100 万条记录的表，全表扫描是 O(n)，而使用索引可以降至 O(log n) 或 O(1)。PostgreSQL 的查询优化器会评估不同索引路径的成本，选择 O(log n) 的 B-Tree 索引扫描而非 O(n) 的顺序扫描，这在海量数据场景下可以带来数量级的性能提升。

**缓存系统设计**: Redis/ Memcached 等缓存系统需要考虑 O(1) 访问特性。LRU 缓存淘汰算法在处理过期键时，如果使用 O(n) 的线性扫描，性能会随缓存键数量增长而下降。优秀的实现会采用 O(1) 的双向链表 + Hash 表结构，确保操作常数时间。

**Web 服务高并发**: 处理高并发请求时，算法复杂度决定了系统的吞吐量上限。每个请求的处理时间 O(f(n)) 乘以并发数，决定了系统能承载的 QPS。在 API 网关中，如果路由匹配是 O(n) 的线性扫描，当路由规则达到数千条时，延迟会显著增加；而使用前缀树(Trie) 可以优化至 O(m)，其中 m 是路径长度。

**日志处理系统**: 大规模日志分析需要选择合适的算法。例如，在实时流处理中，滑动窗口聚合如果使用 O(n) 的逐元素累加，每次滑动都需要 O(n) 重新计算；而使用前缀和可以在 O(1) 时间完成窗口和的计算。

## 实践要点

1. **关注最坏情况**: Big O 通常描述最坏情况复杂度
2. **忽略常数项**: O(2n) 和 O(n) 都视为 O(n)
3. **忽略低阶项**: O(n² + n) 简化为 O(n²)
4. **空间换时间**: 有时可用 O(n) 空间换取 O(1) 时间

## 复杂度选择建议

- 优先 O(1) > O(log n) > O(n) > O(n log n)
- 避免 O(n²) 及更差的复杂度
- 考虑数据规模: n=100 时 O(n²) 可能可接受
- 考虑实际场景: 实时系统 vs 离线批处理
