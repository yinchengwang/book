# 链表技巧

## 概述

链表是动态数据结构的代表，与数组形成互补。理解链表的优势和局限对工程选型至关重要。

## 核心题型

### 反转链表 (LeetCode #206)

**问题**: 将单链表反转。

**解法对比**:

| 方法 | 时间 | 空间 | 说明 |
|------|------|------|------|
| 迭代 | O(n) | O(1) | 三指针遍历 |
| 递归 | O(n) | O(n) | 递归栈空间 |

**工程应用**: 数据库中的 undo log 回滚操作、函数调用栈的回溯。

### 环检测 (LeetCode #141)

**问题**: 判断链表是否有环。

**Floyd 算法要点**: 快慢指针从同一起点出发，快指针每次两步、慢指针每次一步，若有环则必然相遇。证明：当 slow 入环后，fast 每次比 slow 多走一步，相对距离逐减 1，必追上。

**工程应用**: 检测文件系统中的循环引用、GC 的循环引用检测。

### 合并两个有序链表 (LeetCode #21)

**问题**: 合并两个升序链表为一个升序链表。

**工程应用**: 数据库多路归并排序中的归并阶段、Hadoop/Spark 的 shuffle 合并。

### 快慢指针找中点 (LeetCode #876)

**问题**: 找链表中间节点。

**工程应用**: 在归并排序的链表版本中用于切分链表、在二分搜索链表时定位中轴。

## 工程对照

**数组 vs 链表 —— 存储与访问特征对比**:

| 维度 | 数组 | 链表 |
|------|------|------|
| 内存布局 | 连续内存，缓存友好 | 离散内存，缓存命中率低 |
| 随机访问 | O(1) 通过索引 | O(n) 必须遍历 |
| 插入/删除 | O(n) 需移动元素 | O(1) 已知位置时 |
| 空间开销 | 固定弹性（扩容时 O(n)）| 每个节点额外指针开销 |
| 扩容代价 | 连续内存分配 + 复制 | 无，天然动态增长 |

**实际工程选型场景**:

1. **数据库 Buffer Pool 页面链表**: LRU 链表用双向链表实现页面淘汰，因为需要频繁将页面提到头部、从尾部移除，链表 O(1) 的头部插入和尾部删除完美适配。如果换成数组，每次页面访问都要移动大量元素。

2. **操作系统的进程队列**: 就绪队列采用链表结构，因为进程数量动态变化且需要频繁插入和删除。使用数组会导致频繁的内存复制和固定容量限制。

3. **消息队列 (Kafka/RabbitMQ)**: 索引结构使用数组（紧凑存储 + 二分查找），消息体本身存储使用类链表结构（分段追加）。这说明在实际系统中，数组和链表经常组合使用。

4. **CPU 内存分配器 (jemalloc/tcmalloc)**: 不同大小的空闲块用链表串联（free list），因为需要快速切分和合并。但元数据统计用数组存储，因为只需要 O(1) 的索引访问。

5. **Redis 的 SDS (Simple Dynamic String)**: 使用类数组结构而非 C 字符串，关键在于预分配 + 惰性释放平衡了数组扩容的代价，同时保留了 O(1) 长度访问。这说明在工程中，纯数组或纯链表都很少见，更多是"数组 + 链表"的混合结构。

**关键决策原则**:
- 读多写少、需要随机访问 → 数组
- 写多读少、需要频繁插入删除 → 链表
- 实际工程中，哈希表（数组 + 链表）和跳表（多级链表索引）才是更常见的选择

## 复杂度总结

| 题型 | 时间复杂度 | 空间复杂度 | 技巧 |
|------|------------|------------|------|
| 反转链表（迭代） | O(n) | O(1) | 三指针 |
| 反转链表（递归） | O(n) | O(n) | 递归回溯 |
| 环检测 | O(n) | O(1) | 快慢指针 |
| 合并有序链表 | O(n+m) | O(1) | 哑节点 + 归并 |
| 找中点 | O(n) | O(1) | 快慢指针 |

## 面试要点

1. 链表题的核心是"画图+指针"——先在纸上画出链接关系再写代码
2. 哑节点(dummy node)可以大幅简化边界处理
3. 快慢指针是链表题中最常用的技巧（环检测、找中点、找倒数第K个）
4. 递归解法虽然优雅，但面试时注意说明空间复杂度 O(n) 的代价
