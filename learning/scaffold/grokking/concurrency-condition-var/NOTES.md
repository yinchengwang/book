# 并发 - 条件变量 学习笔记

## 核心概念

1. **条件变量**：让线程在条件不满足时阻塞等待，条件满足时被唤醒
2. **pthread_cond_wait**：原子性地释放互斥锁并阻塞，被唤醒后重新获取锁
3. **pthread_cond_signal**：唤醒一个等待线程（通常优化为唤醒优先级最高或有特殊需求的线程）
4. **pthread_cond_broadcast**：唤醒所有等待线程
5. **虚假唤醒（Spurious Wakeup）**：即使没有 signal，wait 也可能返回，所以要用 while 循环判断条件

## 信号量 (semaphore) vs 条件变量 (condvar)

| 特性 | 信号量 | 条件变量 |
|------|--------|---------|
| 状态 | 内部计数器 | 无状态，需配合共享变量 |
| 使用场景 | 资源计数、控制并发数 | 等待特定条件成立 |
| 初始化 | sem_init | pthread_cond_init |
| 广播 | sem_post 只能 +1 | cond_broadcast 唤醒全部 |
| 虚假唤醒 | 不存在 | 存在（需 while 循环） |

## 工程对照

本 demo 实现了一个基础的有界缓冲区生产者-消费者模型。在 `engineering/src/db/storage/wal/wal.c` 中，WAL（Write-Ahead Logging）模块的线程间通信使用了类似的模式：

WAL 缓冲区的写入和刷盘机制：
- **WAL Buffer**：日志记录先写入内存缓冲区，达到阈值（WAL 段大小）或事务提交时刷盘
- **刷盘协调**：WAL 缓冲区刷新时，追加写入的线程需要等待刷盘完成——这本质上是一种生产者-消费者协调
- **LSN 追踪**：每个线程在写入后获取日志序列号（LSN），Buffer Pool 的刷盘机制确保在数据页写入前，对应的 WAL 日志已经落盘

虽然当前 WAL 实现没有直接使用 pthread_cond_wait/signal，但其**生产者生产日志记录、消费者（刷盘线程）消费**的架构与条件变量模式高度一致。WAL 的这种设计保证了"先写日志、再写数据"的核心约束，同时通过**批量刷盘**优化了 I/O 性能——这正是条件变量模式在生产环境中的典型体现：生产者快速生产，消费者批量消费，二者通过条件变量解耦。

## 面试要点

1. **正确的条件变量使用模式**：必须用 `while` 循环而不是 `if` 检查条件（防止虚假唤醒）
2. **pthread_cond_wait 的原子语义**：释放锁 + 进入睡眠是原子操作
3. **信号丢失问题**：在检查条件和 wait 之间如果 signal 已经发出，wait 将永远阻塞
4. **Mesa 语义 vs Hoare 语义**：POSIX 使用 Mesa 语义（被唤醒后不保证条件仍满足）
5. **广播的适用场景**：多个线程等待不同条件，或所有消费者都需要处理新数据时
