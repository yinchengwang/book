# Lock-Free 无锁数据结构 学习笔记

## 核心概念

**无锁编程的核心**：不使用互斥锁、信号量等阻塞同步原语，仅依靠原子操作（CAS/LL-SC）实现线程安全的数据结构。关键性质：
- **Lock-Free**：至少一个线程能在有限步数内完成操作，系统级不会死锁
- **Wait-Free**：所有线程都能在有限步数内完成操作（更严格）

**CAS（Compare-And-Swap）**：`if (*p == old) { *p = new; return true; } return false;`

**ABA 问题**：CAS 只比较指针值，无法检测"A→B→A"的变化。线程读取 top=A，CAS 时 A 已被重用，导致结构损坏。

**标签指针（Tagged Pointer）**：在指针的高位或相邻位附加递增标签，CAS 同时检查指针和标签，使 ABA 无法通过（旧指针+旧标签不可能匹配新标签）。

## 性能对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| 互斥锁 | 实现简单，适用范围广 | 存在锁竞争、上下文切换、优先级反转 |
| 无锁 | 无阻塞、可扩展性好 | 实现复杂，ABA、内存回收（Hazard Pointer/RCU） |
| 标签指针 | 轻量级 ABA 缓解 | 依赖对齐位，64位系统可用 |

## 工程对照

本项目的高性能场景中，无锁思路体现在多个地方：

- **工程组件 vs Lock-Free 对比**：
  - 本数据库存储引擎的 Buffer Pool 使用锁来保护页面表（hashtable bucket lock），如果页面查找用无锁 hash map 实现，可以减少锁竞争，提高多核扩展性。
  - HNSW 向量索引的层间遍历本质上是"读多写少"场景，使用锁保护邻接表确实增加了多线程插入时的开销——理论上可用 RCU 或 Hazard Pointer 来保护邻接表的并发读。

- **自旋锁（spinlock）是无锁的退化形式**：自旋锁用 CAS 抢锁，抢不到就忙等。无锁结构则完全避免了"等待"。
- **业界实践**：Java 的 ConcurrentLinkedQueue、Linux 内核的 kref、以及各种 RCU 原语都是无锁思想的工程实现。C11 `<stdatomic.h>` 提供了跨平台基础，但真正的无锁数据结构的正确性验证极其困难。

## 面试要点

1. ABA 问题：场景 + 影响 + 缓解方案（标签指针 / HPA / RCU）
2. CAS 的 ABA 敏感本质：只比"值"不比"状态"
3. 内存回收（Memory Reclamation）是无锁结构第二大难题——数据被 pop 后不能马上 free，因为可能有其他线程还在读
4. Hazard Pointer 是常见方案：写前标记指针，读者延迟回收
5. 无锁 ≠ 高性能：高竞争下 CAS 重试导致 cacheline bouncing，可能比互斥锁更差
