# 并发 - 互斥锁 学习笔记

## 核心概念

1. **互斥锁（Mutex）**：同一时间只有一个线程可以持有锁，保护临界区
2. **死锁（Deadlock）**：两个或多个线程互相等待对方释放锁，永久阻塞
3. **读写锁（RWLock）**：读读不互斥、读写互斥、写写互斥
4. **ABBA 死锁**：线程1 持有 A 等 B，线程2 持有 B 等 A
5. **锁的粒度**：大粒度锁简单但并发度低，细粒度锁复杂但性能好

## 死锁产生的四个必要条件

| 条件 | 说明 | 破解方法 |
|------|------|---------|
| 互斥 | 资源一次只能被一个线程使用 | 无法避免（锁的本质） |
| 持有并等待 | 线程拿着一个锁去等另一个锁 | 一次性获取所有锁 |
| 不可剥夺 | 不能强制从线程夺取锁 | trylock + 回退 |
| 循环等待 | 存在 A→B→C→A 的等待链 | 固定加锁顺序 |

## 工程对照

本 demo 演示了基础互斥锁与读写锁的使用模式。在 `engineering/src/db/storage/lock/lock_mgr.c` 中，数据库锁管理器实现了远比 POSIX 互斥锁复杂的**多粒度锁协议**：

- **锁层级**：DATABASE → TABLE → PAGE → ROW，支持意向锁（IS/IX/SIX）
- **锁兼容矩阵**：定义了 5 种锁类型之间的兼容关系（如 IS 与 IX 兼容，X 与任何锁不兼容）
- **死锁检测**：使用等待图（Wait-For Graph）定期检测死锁，而非简单超时
- **锁升级**：支持从行锁升级到页锁/表锁

与 POSIX 的 pthread_mutex 仅提供基本的 lock/unlock 不同，数据库锁管理器需要处理：
1. **事务隔离级别**：不同隔离级别下锁的获取和释放时机不同（2PL vs Optimistic）
2. **意向锁**：告诉上层"我在更细粒度上已有锁"，避免逐行检查
3. **锁降级**：事务完成后，写锁可以降级为读锁

这种复杂锁协议的设计权衡：增加并发度（细粒度行锁）vs 增加开销（更多锁管理、死锁检测成本）。

## 面试要点

1. 死锁预防策略：固定加锁顺序、trylock + 回退、锁超时
2. 读写锁适用的场景：读远多于写
3. 锁的粒度选择：行锁 vs 表锁，细粒度锁可能引入死锁风险
4. 条件变量 vs 忙等待：条件变量不消耗 CPU，忙等待浪费 CPU
5. 递归锁（pthread_mutexattr_settype + PTHREAD_MUTEX_RECURSIVE）：同一线程可多次加锁
