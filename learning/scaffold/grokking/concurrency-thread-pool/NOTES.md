# 线程池学习笔记

## 核心概念

线程池的核心思想是"复用"——预先创建一组线程，通过任务队列分发工作，避免反复 `pthread_create` / `pthread_join` 的开销。

**关键设计维度**：
1. **线程数量**：固定池 vs 弹性池（最小/最大线程数）
2. **任务队列**：有界队列（可拒绝） vs 无界队列（可能 OOM）
3. **拒绝策略**：丢弃/阻塞调用者/抛出异常/ caller-runs
4. **关闭方式**：优雅关闭（等待队列清空） vs 立即关闭

## 三种任务调度模型对比

| 模型 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 每请求一线程 | 实现简单 | 线程创建销毁开销大 | 低并发 |
| 线程池 | 线程复用、可控 | 实现复杂、调参困难 | 高并发通用 |
| 协程/纤程 | 极轻量、高吞吐 | 需要语言/运行时支持 | IO 密集型 |

## 工程对照

本项目虽然没有独立的 connection_pool.c，但 `engineering/src/db/consensus/raft.c` 中的多线程模型展示了类似的模式：

1. **后台线程循环**：Raft 的 `tick` 线程不断检查超时和状态转换（follower → candidate → leader），其模式与本演示的工作线程循环 `while (1) { wait; execute; }` 本质相同——都是"事件循环 + 状态机"。区别在于 Raft 的 tick 线程定期轮询，而线程池用条件变量阻塞等待任务。

2. **锁保护范围**：Raft 使用 `pthread_mutex` 保护所有共享状态读写，与本演示的线程池保持一致——关键区别在于 Raft 的锁持有时间可能跨越多个操作（如整个 log append 期间），而线程池的设计原则是"取任务时持锁，执行任务时不持锁"，这是为了避免任务间的锁竞争。

3. **扩容策略**：工程中线程池的扩容通常考虑 CPU 核数和任务类型（CPU 密集型 vs IO 密集型）。CPU 密集型线程数 ≈ CPU 核数；IO 密集型可为 CPU 核数 * 2（等待 IO 时不占 CPU）。本演示 `tp_expand` 简单地增加一个线程，而真实系统（如 Java 的 ThreadPoolExecutor）会基于队列长度和 `keepAliveTime` 决策是否扩容。

4. **拒绝策略**：本演示采用"直接拒绝"，工程中更常见的是"调用者运行"（caller-runs）——将任务退给提交者线程执行，自然形成反压力。

## 面试要点

1. 线程池的核心优势是减少线程创建/销毁开销，控制并发度
2. 任务队列满时的拒绝策略是系统健壮性的关键
3. 优雅关闭必须等待已提交任务完成，用 shutdown flag 通知
4. 线程池中的条件变量使用模式：`while (条件) wait()` 避免虚假唤醒
