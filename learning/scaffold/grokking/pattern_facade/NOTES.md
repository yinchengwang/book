# 外观模式 学习笔记

## 核心概念

- **外观模式 (Facade)**: 为复杂子系统提供统一的高层接口
- **封装**: 隐藏子系统内部细节和依赖关系
- **简化**: 对外提供最简单的调用入口

## 关键要点

| 角色 | 说明 |
|------|------|
| Facade | 统一的高层接口, 知道哪些子系统类负责哪些请求 |
| 子系统 | 实现具体功能, 不知道 Facade 的存在 |
| 客户 | 只与 Facade 交互, 不直接操作子系统 |

## 工程对照

外观模式在数据库系统中体现为 SQL 接口的封装。用户通过 SQL 语句（外观）操作数据库，而不需要了解底层的 Buffer Pool、WAL、BTree 索引、锁管理器等子系统。`engineering/src/db/core/db_server.c` 中的数据库服务器实现的 PostgreSQL Wire 协议就是一层"外观"——它把 SQL 解析、查询优化、存储引擎调用、事务管理等子系统统一到一个网络协议接口后。GUC 配置系统 (`engineering/src/db/core/guc.c`) 也类似——用户只需设置 `shared_buffers = 128MB` 这一个参数，底层涉及的内存分配策略都由系统自动协调。

## 面试要点

1. 外观 vs 适配器: 外观提供新接口, 适配器兼容旧接口
2. 外观 vs 中介者: 外观简化接口, 中介者协调交互
3. 外观不限制客户直接访问子系统 (不是防火墙)
