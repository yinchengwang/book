# 死锁 学习笔记

## 核心概念

1. **死锁四条件**：互斥（Mutual Exclusion）、持有并等待（Hold and Wait）、不可剥夺（No Preemption）、循环等待（Circular Wait）
2. **哲学家就餐问题**：经典死锁模型，5 位哲学家和 5 根叉子，每人需要两根叉子才能吃饭
3. **资源排序（Resource Ordering）**：给资源编号，按固定顺序获取，打破循环等待条件
4. **等待图（Wait-For Graph, WFG）**：有向图，节点是线程/事务，边 T1→T2 表示 T1 在等待 T2 持有的资源

## 死锁四条件

| 条件 | 说明 | 破解策略 |
|------|------|---------|
| 互斥 | 同一资源同时只能被一个线程持有 | 无法避免（锁的本质要求） |
| 持有并等待 | 持有一个锁的同时去申请另一个锁 | 一次性获取所有锁（原子申请） |
| 不可剥夺 | 已获取的锁不能被强制夺走 | trylock + 回退（非阻塞获取） |
| 循环等待 | 存在 T1→T2→...→T1 的等待链 | 固定加锁顺序（资源排序） |

## 工程对照

本 demo 演示了经典的哲学家就餐问题及两种解决策略。在 `engineering/src/db/storage/txn/lock_mgr.c` 中，数据库锁管理器实现了生产级别的死锁处理：

| 方面 | 本 Demo | engineering 锁管理器 |
|------|---------|---------------------|
| 资源类型 | 叉子（pthread_mutex_t） | 行锁/页锁/表锁（多粒度锁） |
| 检测方式 | 等待图 DFS 环检测 | 等待图 DFS + 超时检测 |
| 受害者选择 | 无（仅检测） | 按事务年龄（选择最年轻/回滚成本最小的事务） |
| 检测触发 | 手动调用 | 定期后台线程检查（deadlock_timeout GUC） |
| 锁释放 | 手动解锁 | 事务提交/回滚时自动释放所有锁 |

生产级死锁处理的关键设计决策：
1. **检测 vs 预防**：工程中更常见检测 + 回滚（死锁不常见时），而非预防（预防可能过度限制并发度）
2. **受害者选择**：选择 undo 日志量最小、或最年轻的事务回滚，而非随机选择
3. **检测间隔**：MySQL InnoDB 默认每 1 秒检测一次等待图（死锁不频繁时减少开销）

## MySQL InnoDB 死锁机制

MySQL InnoDB 采用 WFG 检测 + 两阶段锁协议（2PL）。相比简单超时检测（如 innodb_lock_wait_timeout），WFG 检测的优势在于：
- **即时发现**：在环形成时立即检测，而非被动等待超时，用户体验更好
- **精准回滚**：只回滚环中的受害者事务，而非所有等待超时的事务
- **成本感知**：选择 undo 日志量最小的事务作为受害者，减少重做成本

缺点是检测本身需要遍历图，锁竞争激烈时 WFG 扫描本身可能成为瓶颈。
