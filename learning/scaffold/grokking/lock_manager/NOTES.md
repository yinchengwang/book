# 锁管理器 · 学习笔记

## 核心概念

1. **行锁（Row Lock）**：锁定单行，并发度高，锁管理开销大
2. **表锁（Table Lock）**：锁定整表，并发度低，但无锁管理开销
3. **意向锁（Intention Lock）**：表级标记，快速判断表操作与行锁是否冲突
4. **两阶段锁（2PL）**：事务分扩展阶段（只加锁不解锁）和收缩阶段（只解锁不加锁）
5. **死锁**：循环等待条件；检测方式：等待图有环 → 选中一个事务回滚

## 锁兼容矩阵

| 已持有 → 申请 ↓ | IS | IX | S | X |
|-----------------|---|---|---|---|
| IS（意向读） | ✓ | ✓ | ✓ | ✗ |
| IX（意向写） | ✓ | ✓ | ✗ | ✗ |
| S（读锁） | ✓ | ✗ | ✓ | ✗ |
| X（写锁） | ✗ | ✗ | ✗ | ✗ |

## 工程对照

本项目的 `engineering/src/db/txn/lock.c` 实现了锁管理器。核心数据结构是
`LockHashTable`（哈希表，键为 `(rel_oid, tuple_id)` → 锁条目），通过
`lock_acquire()` / `lock_release()` 接口管理。`lock_deadlock_detect()`
定期在等待图上运行环检测（DFS）。与本文演示的不同在于，工程实现需要处理
锁升级（Lock Upgrade，S→X）、锁排队（Wait Queue）和死锁 Victim 选择。
