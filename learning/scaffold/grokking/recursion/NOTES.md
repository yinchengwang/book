# 递归思想

## 工程对照

### 递归 vs 迭代的选择

在工程实践中，递归和迭代各有适用场景：

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| 树/图的遍历 | 递归 | 代码简洁，天然符合树结构定义 |
| 数值计算（阶乘/Fibonacci） | 迭代 | 避免栈溢出，性能更稳定 |
| 分治算法（快速排序/归并排序） | 递归 | 分治思想与递归天然契合 |
| 深度优先搜索 | 递归或显式栈 | 递归深度可控时优先递归 |
| 动态规划 | 迭代+记忆化 | 避免栈开销，结果可复用 |

**工程经验**：生产代码中，递归深度超过 1000 时应考虑改用迭代或增加栈大小验证。在 PostgreSQL 存储引擎中，索引遍历和表达式求值都大量使用递归，但都设置了安全的递归深度限制。

### 工程对照总结

递归在算法层面是强大的工具，但在工程层面需要谨慎使用。核心原则是：
1. **确保有正确的 base case**，否则会导致无限递归
2. **评估递归深度**，必要时使用尾递归或迭代改写
3. **记忆化搜索**，将指数级降为多项式级
4. **关注栈空间消耗**，与堆空间的时间换空间 trade-off

## 重点

### 尾递归优化

尾递归是指递归调用是函数执行的最后一个操作：

```c
// 普通递归：需要保存调用栈
long long factorial_recursive(int n) {
    if (n <= 1) return 1;
    return n * factorial_recursive(n - 1);  // 递归后还有乘法
}

// 尾递归：可被编译器优化
long long factorial_tail(int n, long long acc) {
    if (n <= 1) return acc;
    return factorial_tail(n - 1, n * acc);  // 递归调用是最后操作
}
```

C11 标准未强制要求尾递归优化，但 GCC/Clang 在启用优化时会进行尾递归优化（`-O2` 或更高）。MSVC 对尾递归优化支持较弱。

### 栈溢出风险

递归调用的最大问题是栈空间消耗：

- **Linux 默认栈大小**：8MB（`ulimit -s` 查看）
- **每次递归调用**：约占用 100 字节至数 KB（取决于局部变量大小）
- **安全递归深度估算**：约 10 万次（保守估计 1 万次以内）

**风险场景**：
1. 斐波那契普通递归：调用次数呈指数增长，n=50 时会超过数百万次调用
2. 链表反转递归：链表过长时可能导致栈溢出
3. 深度搜索：树/图过深时栈空间耗尽

**防御策略**：
1. 使用 `gcc -Wformat-overflow` 检测潜在溢出
2. 使用 `ulimit -s` 临时调整栈大小（`ulimit -s unlimited`）
3. 编译时使用 `-fsanitize=address` 检测栈溢出
4. 工程实践中设置最大递归深度保护
