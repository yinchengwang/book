# 并发 - 线程基础 学习笔记

## 核心概念

1. **线程（Thread）**：进程内的独立执行流，共享地址空间和文件描述符
2. **pthread_create**：创建线程，指定入口函数和参数
3. **pthread_join**：等待线程结束，回收资源，获取返回值
4. **竞态条件（Race Condition）**：多个线程同时访问共享数据且至少一个是写操作时，结果取决于执行时序
5. **原子性**：`counter++` 实际是读-改-写三步，非原子操作

## pthread 常用函数对比

| 函数 | 用途 | 说明 |
|------|------|------|
| pthread_create | 创建线程 | 返回线程 ID |
| pthread_join | 等待线程 | 阻塞直到目标线程退出 |
| pthread_detach | 分离线程 | 线程退出时自动释放资源 |
| pthread_self | 获取自身 ID | 线程内调用 |
| pthread_equal | 比较线程 ID | 跨平台兼容 |

## 工程对照

本项目代码演示了手动创建 POSIX 线程的基础操作。在实际的 `engineering/src/redis/` Redis 移植代码中，Redis 使用的是**单线程事件循环模型**（基于 epoll/kqueue），而不是多线程 pthread 模型。

Redis 的核心设计理念是：**数据操作在主线程（事件循环）中串行执行，完全避免锁竞争**。这与本 demo 中演示的多线程累加形成鲜明对比——Redis 通过单线程回避了整个并发同步问题。

不过，Redis 6.0+ 在以下场景使用了多线程：
- **I/O 线程**：处理网络读写，主线程将连接分发给多个 I/O 线程并行处理，但命令执行仍然在主线程串行执行
- **后台任务**：`BIO_LAZY_FREE`（惰性删除大 key）、`BIO_CLOSE_FILE`（关闭文件）等由后台线程处理

本仓库的 Redis 移植代码是单线程的，专注于数据结构（链表、跳表、SDS）而非并发。多线程的使用体现了**分层设计的权衡**：网络 I/O 可以并行，但数据操作必须串行以保证一致性。

## 面试要点

1. 线程与进程的区别：共享地址空间 vs 独立地址空间
2. 竞态条件产生的三个条件：多线程 + 共享数据 + 至少一个写操作
3. 解决方案：互斥锁、原子操作、无锁数据结构
4. `pthread_join` 不调用会导致僵尸线程（资源泄漏）
5. 返回值必须指向线程退出后仍有效的内存（用 malloc 或静态变量）
