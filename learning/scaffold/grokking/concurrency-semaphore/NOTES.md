# 并发 - 信号量 学习笔记

## 核心概念

1. **信号量（Semaphore）**：一个整数计数器 + 两个原子操作（P/V）
2. **P 操作（sem_wait）**：如果计数器 > 0，减一并立即返回；否则阻塞等待
3. **V 操作（sem_post）**：计数器加一，唤醒一个等待的线程
4. **二元信号量**：值只有 0 和 1，可用作互斥锁
5. **计数信号量**：值 > 1，控制多份资源的并发访问

## 信号量 vs 互斥锁 vs 条件变量

| 特性 | 信号量 | 互斥锁 | 条件变量 |
|------|--------|--------|---------|
| 所有者 | 无 | 有（同一线程解锁） | 无 |
| 初始化值 | 任意非负整数 | 始终 1（锁定） | N/A |
| 虚假唤醒 | 不会 | 不适用 | 会 |
| 复杂条件 | 只能计数 | 不适用 | 任意条件 |
| 用途 | 资源计数 | 临界区保护 | 条件等待 |

## 工程对照

本 demo 演示了 POSIX 信号量的三种使用模式：互斥锁、资源池和生产者-消费者。在实际的 `engineering/src/db/storage/mm/` 多模态存储引擎中，连接池和资源管理使用了信号量风格的控制模式：

数据库连接池需要限制最大并发连接数，这本质上是一个计数信号量问题——sem_init(&pool, 0, max_connections)。当所有连接被占满时，新请求等待其他请求释放连接。在 `engineering/src/db/storage/buf/bufmgr.c` 的 Buffer Pool 管理中，空闲 buffer 列表使用条件变量等待可用页面，这与计数信号量在语义上等价——当缓冲池满时，请求页面资源的线程必须等待其他线程释放页面后才能继续。

信号量常用于**准入控制**（Admission Control）和**限流**（Rate Limiting）场景，而条件变量更适合复杂的条件等待逻辑。二者的选择取决于需求：如果只是简单的"资源计数"，信号量更简洁；如果需要等待"缓冲区内数据超过 N 条才消费"这样的复杂条件，条件变量更灵活。

## 面试要点

1. P/V 操作的原子性是信号量的基础保证
2. 信号量没有"所有权"概念——一个线程可以 V 另一个线程 P 的资源，这是灵活性也是风险
3. sem_init 的 pshared 参数：0（线程间同步）、非 0（进程间同步，需共享内存）
4. 经典同步问题：生产者-消费者（有界缓冲区）、读者-写者、哲学家就餐
5. 二元信号量和互斥锁的区别：互斥锁有优先级继承，解决优先级反转问题
