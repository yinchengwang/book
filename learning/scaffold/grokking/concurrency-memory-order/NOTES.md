# 内存序 (Memory Order) 学习笔记

## 核心概念

C11 `<stdatomic.h>` 定义了六种内存序（memory_order），从弱到强：

| 内存序 | 说明 | 典型用例 |
|--------|------|---------|
| `relaxed` | 仅原子性，无顺序约束 | 计数器递增 |
| `consume` | 数据依赖序（C17 已弃用） | 指针传递 |
| `acquire` | 防止后续操作重排到前面 | 读锁/读标志 |
| `release` | 防止前面操作重排到后面 | 写锁/写标志 |
| `acq_rel` | acquire + release | 读-改-写 |
| `seq_cst` | 全局全序（默认值） | 安全默认 |

## CPU 架构差异

| 架构 | 内存模型 | 特点 |
|------|---------|------|
| x86/x64 | 强序 (TSO) | 除了 store-load 重排外，其他重排基本不发生 |
| ARMv8 | 弱序 | 允许多种重排，需显式屏障 |
| POWER | 极弱序 | 比 ARM 更多重排可能性 |

**关键洞察**：在 x86 上测试内存序差异很难复现问题，因为 x86 天然强序。这也是为什么多线程 bug 在开发环境无法复现，上线到 ARM 服务器后突然出现。

## 工程对照

本项目代码中涉及内存序考量的场景：

- **C 语言实现层**：大多数数据库代码使用互斥锁保护共享数据，锁原语内部已经包含了必要的 acquire/release 语义。直接使用原子操作时，需要关注内存序的选择。
- **Buffer Pool** 的 Clock-Sweep 的引用计数操作：在更新页面引用计数时，如果使用 relaxed 计数并使用 release 语义同步，可以减少不必要的屏障开销——但当前实现采用简单互斥锁，可读性优先。
- **BTree 索引**的并发搜索：预计未来引入 latch-free BTree 时，acquire/release 内存序的精细使用就变得关键，能避免 seq_cst 的多余屏障。

对比 C++ 的 `std::memory_order`，C11 的枚举是等价的，但 C++20 添加了更丰富的原子类型和 `std::atomic_ref`。

## 面试要点

1. **acquire 是"读获取锁"语义**：acquire 之后的读/写不能重排到 acquire 之前，确保读到的是最新的
2. **release 是"写释放锁"语义**：release 之前的读/写不能重排到 release 之后，确保写入对其他线程可见
3. **seq_cst 是全局一致性**：所有线程看到一个统一的执行顺序，代价最高
4. **relaxed ≠ 无序**：在同一个线程内，单线程的数据依赖仍然保持（不可能出现 a=b; c=a; 得到 c 比 a 新）
5. **volatile 不等于 atomic**：volatile 防止编译器优化（禁止寄存器缓存），但 volatile 不保证任何内存序或原子性
