# 模板方法模式 学习笔记

## 核心概念

- **模板方法 (Template Method)**: 在父类中定义算法骨架, 子类实现具体步骤
- **抽象方法**: 子类必须实现
- **具体方法**: 所有子类共享
- **钩子方法**: 可选扩展点, 子类可覆写

## 模板方法结构

| 方法类型 | 说明 | 子类控制 |
|----------|------|----------|
| 模板方法 (Template) | 定义算法骨架流程 | 不能覆写 |
| 抽象方法 (Abstract) | 必须由子类实现 | 完全控制 |
| 具体方法 (Concrete) | 父类提供默认实现 | 可选覆写 |
| 钩子方法 (Hook) | 控制模板方法中的决策 | 可选覆写 |

## 工程对照

框架设计中的模板方法模式在 `engineering/` 轨的测试体系中有直接体现。`engineering/test/` 下的所有 GoogleTest 测试文件遵循相同的测试骨架——`TEST()` 宏定义了测试的模板方法，每个测试只关注"特定的断言"这一差异部分。`cmake/ProjectUtils.cmake` 中的 `add_project_test()` 函数也是一个模板方法——它统一处理了源文件通配、可执行文件创建、gtest 链接等通用步骤。数据库的查询执行流程 (parser -> analyzer -> optimizer -> executor) 也是一个固定的算法骨架。

## 面试要点

1. 模板方法遵循"好莱坞原则"——"不要打电话给我们, 我们会打给你"
2. 模板方法在框架设计中应用广泛 (JUnit, Servlet, Spring)
3. 钩子方法提供灵活性而不强制子类实现
