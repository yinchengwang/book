# sort_basic 学习笔记

## 概念地图

基础排序是理解"为什么需要更好的排序"的必经之路：

- **冒泡排序**：相邻两两比较，每轮把最大值"浮"到最后。加了 `swapped` 标志后最优 O(n)（已有序），最坏 O(n²)。稳定
- **选择排序**：每轮找最小值放到前面。比较次数固定 O(n²)，交换次数 O(n)——是所有 O(n²) 排序中交换最少的。不稳定（交换可能打乱等值元素相对顺序）
- **插入排序**：像摸牌一样把元素插入已排序前缀。最优 O(n)（已有序），最坏 O(n²)。稳定。对"小数组 + 近乎有序"场景极快——这就是为什么 Timsort 和 Introsort 在小数组时退回到插入排序
- **稳定性**：等值元素排序后相对顺序不变。冒泡和插入稳定；选择不稳定。稳定性对"多级排序"（先按班级再按成绩）很重要

与 `sort_advanced` 卡的关系：本卡是地基——快排的 partition、归并的 merge 都建立在"比较两个元素、移动元素"这一基本操作上

## 踩坑记录

1. **冒泡内层循环边界**：`j < n - 1 - i` 而非 `j < n - 1`——已经冒到末尾的 i 个元素不需要再比较。写错边界多跑很多无用比较
2. **选择排序的不稳定性**：如 `[5, 5, 2]`，第一轮把第一个 5 换到 2 的位置，两个 5 的相对顺序变了——这是很多面试题的考点
3. **插入排序的 `j >= 0` 边界**：while 循环里 j 可能变成 -1，此时 `a[j+1]` = `a[0]`——key 比所有元素都小，放在最前面
4. **统计比较次数时的循环边界**：`for` vs `while` 的比较次数统计方式不同——for 循环每次迭代都算一次比较，while 循环只在条件判断处算

## 工程对照（≥100 字硬约束）

基础排序在工程侧的复用点：

1. **`qsort` 内部实现**：C 标准库 `qsort` 在小分区时退回到插入排序——因为插入排序在 n<16 时常数最小、无递归开销。本仓库 `engineering/src/algo-prod/` 中多处调用 `qsort` 进行排序
2. **BTree 叶子节点内部排序**：`engineering/src/db/index/btree/btreeam.c` 中 BTree 每个页内 key 保持有序——插入新 key 时本质是一次"页内插入排序"，因为页内 key 数通常 < 100，插入排序是最佳选择
3. **索引扫描的 ORDER BY**：`engineering/src/db/sql/sql_exec.c` 中当 ORDER BY 列无索引时，需要对结果集排序——小结果集用插入排序、大结果集用快排。本卡的选择排序比较次数多但交换少，适合元素移动成本高的场景（如磁盘页内排序）
4. **稳定性在工程中的意义**：`engineering/src/db/index/btree/` 中的 BTree 索引页内 key 排序必须稳定——等值 key 按插入顺序排列，保证范围扫描的顺序可预测

学完本卡后能动手的事：在 `engineering/src/db/sql/` 的 `sql_exec.c` 中识别出排序调用点，评估当前数据集大小是否适合插入排序优化
