# LSH 算法原理

## 1. 算法概述

LSH (Locality-Sensitive Hashing，局部敏感哈希) 是一种基于哈希的近似最近邻搜索算法，由 Indyk 和 Motwani 在 1998 年提出。

### 核心思想

通过**哈希函数族**将相似向量映射到相同桶中：
- **局部敏感**：相似向量哈希碰撞概率高
- **降维映射**：将高维向量压缩为低维哈希码
- **多表冗余**：多个哈希表提升召回率

### 适用场景

- 超大规模数据集（亿级向量）
- 对召回率要求不高的场景
- 内存受限的环境
- 实时查询（低延迟）

### 优缺点

| 优点 | 缺点 |
|------|------|
| 构建速度快 O(n) | 召回率相对较低 |
| 查询速度快 O(1) | 内存占用较高（多表） |
| 支持多种距离度量 | 参数调优复杂 |
| 可扩展性好 | 假阳性率较高 |

## 2. 数据结构

### 多表哈希结构

```
┌─────────────────────────────────────────────────────┐
│              哈希函数族 (Hash Family)                 │
│  h₁, h₂, ..., hₙ  (随机采样生成)                     │
└────────────┬────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│                  哈希表数组                           │
│                                                      │
│  Table 0    Table 1    Table 2    ...  Table L-1   │
│  ┌─────┐    ┌─────┐    ┌─────┐         ┌─────┐     │
│  │ 桶0 │    │ 桶0 │    │ 桶0 │         │ 桶0 │     │
│  │ 桶1 │    │ 桶1 │    │ 桶1 │         │ 桶1 │     │
│  │ ... │    │ ... │    │ ... │         │ ... │     │
│  │ 桶N │    │ 桶N │    │ 桶N │         │ 桶N │     │
│  └─────┘    └─────┘    └─────┘         └─────┘     │
│                                                      │
│  每个桶存储: {id₁, id₂, ..., idₖ}  (向量 ID 列表)  │
└─────────────────────────────────────────────────────┘
```

### 复合哈希函数

每个哈希表使用 `k` 个基础哈希函数组合：

```
g(x) = (h₁(x), h₂(x), ..., hₖ(x))
```

组合后的哈希码空间：`2^k` 个桶。

### 桶结构

```c
typedef struct lsh_bucket {
    int32_t *vector_ids;    // 向量 ID 数组
    int32_t count;          // 当前数量
    int32_t capacity;       // 容量
} lsh_bucket_t;

typedef struct lsh_table {
    lsh_bucket_t *buckets;  // 桶数组
    int32_t n_buckets;      // 桶数量（= 2^k）
    hash_func_t *funcs;     // 哈希函数数组（k 个）
} lsh_table_t;

typedef struct lsh_index {
    lsh_table_t *tables;    // 哈希表数组
    int32_t n_tables;       // 表数量（L）
    int32_t n_hash;         // 每表哈希函数数（k）
    int32_t dims;           // 向量维度
    lsh_type_t type;        // LSH 类型

    /* Phase 2: 存储后端集成 */
    storage_backend_t *storage;
    heap_vector_store_t *heap_store;
    vector_ref_t *vector_refs;
    uint32_t vector_refs_size;
    persist_control_t *persist_ctrl;
} lsh_index_t;
```

## 3. 核心算法

### 3.1 哈希函数类型

#### Bitwise LSH (汉明距离)

```
h(x) = xᵢ  // 随机采样第 i 位
```

适用于二值向量，哈希函数随机选择向量的一位。

#### p-stable LSH (L2 距离)

```
h(x) = ⌊(a·x + b) / w⌋
```

其中：
- `a` 是从 p-stable 分布（如高斯分布）采样的随机向量
- `b` 是 [0, w] 区间的随机偏移
- `w` 是桶宽度参数

**p-stable 分布性质**：
对于 L2 距离，使用高斯分布（2-stable）：
```
||x - y||₂ < r  ⇒  P(h(x) = h(y)) > p₁
||x - y||₂ > cr ⇒  P(h(x) = h(y)) < p₂
```

#### SimHash (余弦相似度)

```
h(x) = sign(a·x)  // 符号函数
```

其中 `a` 是随机单位向量。相似向量的哈希码汉明距离小。

### 3.2 构建算法

```
Build(vectors, n, L, k):
    1. 生成 L 个哈希表
       for i = 1 to L:
           随机选择 k 个哈希函数 {h₁, ..., hₖ}
           创建空表，共 2^k 个桶

    2. 插入所有向量
       for each vector x:
           for each table t:
               计算哈希码 g(x) = (h₁(x), ..., hₖ(x))
               将 x.id 插入桶 g(x)
```

### 3.3 搜索算法

```
Search(query, k):
    1. 初始化候选集 C = ∅

    2. 遍历所有哈希表
       for each table t:
           计算 query 的哈希码 g(query)
           从桶 g(query) 中取出所有向量 ID
           加入候选集 C

    3. 精确距离计算（重排序）
       for each id in C:
           从 Heap 获取完整向量
           计算精确距离 d(query, vector)

    4. 排序返回 top-k
       按距离排序，返回前 k 个结果
```

**两阶段搜索模式**（Phase 2 重构后）：
1. **阶段 1**：哈希碰撞获取候选 ID
2. **阶段 2**：从 Heap 获取向量，计算精确距离
3. **阶段 3**：重排序返回 top-k

### 3.4 参数选择

| 参数 | 作用 | 增大时影响 |
|------|------|-----------|
| `L` (表数量) | 召回率 | 召回率↑ 内存↑ 查询时间↑ |
| `k` (哈希函数数) | 精度 | 精度↑ 召回率↓ 桶数↑ |
| `w` (桶宽度) | 容错 | 召回率↑ 精度↓ (p-stable) |

**典型参数组合**：

| 数据规模 | L | k | w | 预期召回率 |
|----------|---|---|---|-----------|
| 100万 | 32 | 18 | 4.0 | ~80% |
| 1000万 | 64 | 20 | 4.0 | ~75% |
| 1亿 | 128 | 22 | 4.0 | ~70% |

## 4. 时间/空间复杂度

### 构建复杂度

| 操作 | 时间复杂度 | 空间复杂度 |
|------|------------|------------|
| 生成哈希函数 | O(L·k·d) | O(L·k·d) |
| 插入向量 | O(n·L·k·d) | O(n·L) (最坏) |

### 查询复杂度

| 操作 | 时间复杂度 | 说明 |
|------|------------|------|
| 计算哈希码 | O(L·k·d) | d=维度 |
| 桶扫描 | O(L·B) | B=平均桶大小 |
| 重排序 | O(\|C\|·d) | \|C\|=候选集大小 |

**总体查询复杂度**：`O(L·k·d + |C|·d)`

### 空间复杂度

- **理论值**：`O(n·L)` (每向量每表一个引用)
- **实际值**：受桶分布影响，不均匀时可能更高

**内存估算公式**：
```
Memory ≈ n·L·4 bytes (ID 存储)
       + L·2^k·bucket_overhead
       + L·k·d·4 bytes (哈希函数参数)
```

## 5. 与其他索引对比

### 总体对比

| 索引 | 构建速度 | 查询速度 | 召回率 | 内存占用 |
|------|----------|----------|--------|----------|
| **LSH** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐ |
| HNSW | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐ |
| IVF | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| DiskANN | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

### LSH vs HNSW

| 特性 | LSH | HNSW |
|------|-----|------|
| **距离度量** | 多种（L2/Cosine/Hamming） | 任意度量 |
| **召回率** | 中等（70-80%） | 高（95%+） |
| **构建速度** | 快 O(n) | 慢 O(n log n) |
| **参数敏感性** | 高 | 中等 |
| **可扩展性** | 好（分布式友好） | 中等 |
| **动态更新** | 支持 | 支持 |

**适用场景选择**：
- **LSH**：超大数据集、低延迟要求、召回率要求不高
- **HNSW**：召回率优先、中等数据集、查询延迟容忍度高

### LSH vs IVF

| 特性 | LSH | IVF |
|------|-----|-----|
| **空间划分** | 哈希函数（数据无关） | 聚类（数据相关） |
| **训练阶段** | 无需训练 | 需要训练 |
| **召回率控制** | L 和 k 参数 | nprobe 参数 |
| **分布适应性** | 差（均匀假设） | 好（自适应） |

**适用场景选择**：
- **LSH**：数据分布未知、无训练数据、快速部署
- **IVF**：已知数据分布、有训练数据、召回率优先

## 6. 变体与优化

### 6.1 Multi-Probe LSH

**核心思想**：不只查询单个桶，而是探测相邻桶。

**探测策略**：
```
查询桶: g(query) = (h₁, h₂, ..., hₖ)
探测桶: (h₁±1, h₂, ..., hₖ)
        (h₁, h₂±1, ..., hₖ)
        ...
```

**优势**：
- 减少 L（表数量），降低内存
- 保持召回率
- 增加 CPU 开销（计算更多哈希码）

### 6.2 Entropy LSH

**核心思想**：通过扰动原始哈希函数生成多个哈希码。

**公式**：
```
g'(x) = g(x) ⊕ perturbation
```

适用于需要精确控制召回率的场景。

### 6.3 深度学习 LSH

**核心思想**：用神经网络学习哈希函数。

- **Optimal LSH**：通过优化学习最优投影
- **Deep Hashing**：端到端学习哈希函数

**优势**：数据相关，召回率更高

## 7. 实践建议

### 参数调优步骤

1. **确定 L 和 k**：
   ```
   L = log(1 - target_recall) / log(1 - p₁^k)
   ```
   其中 `p₁` 是相似向量的碰撞概率。

2. **调整 w（p-stable LSH）**：
   - 从 `w = 4.0` 开始
   - 测试不同 w 值的召回率
   - 选择召回率-速度最佳平衡点

3. **Multi-Probe 参数**：
   - 探测数量 `T = 2~10`
   - `T` 增大召回率提升，但查询变慢

### 性能优化技巧

1. **批量化处理**：
   - 批量插入向量
   - 批量查询（共享哈希计算）

2. **内存优化**：
   - 使用位图压缩桶 ID
   - 桶内 ID 排序，二分查找

3. **并行化**：
   - 不同哈希表并行查询
   - GPU 加速哈希计算
