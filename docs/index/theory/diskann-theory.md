# DiskANN 算法原理

## 1. 算法概述

DiskANN (Disk-based Approximate Nearest Neighbor) 是微软研究院在 2019 年提出的面向 SSD 的近似最近邻搜索算法，基于 Vamana 图结构，通过块式分页文件实现"图在磁盘、数据按需加载"，突破了全内存索引的容量上限。

### 核心思想

将向量图存储在 SSD 上，搜索时按需加载邻居信息，仅将压缩后的向量（PQ 码本）保留在内存中，从而在有限内存下索引十亿级向量。

### 核心特性

- **Vamana 图构建**：单层稀疏邻居图，通过 Robust Prune 保证图的可导航性
- **PQ/SQ/LVQ/RQ 量化压缩**：向量压缩后驻留内存，用于距离预筛
- **块式持久化存储**：图结构按页对齐存储在 SSD，减少随机 IO
- **增量插入与删除**：FreshDiskANN 支持动态更新

### 适用场景

- 超大规模向量检索（十亿级）
- 内存不足以容纳全量向量的场景
- SSD 延迟可接受（~100us 级）的在线服务
- 需要增量更新的生产系统

### 优缺点

| 优点 | 缺点 |
|------|------|
| 内存占用极低（仅 PQ 码本） | 搜索延迟受 SSD 影响 |
| 支持十亿级向量索引 | 构建时间较长 |
| 召回率高（95%+ @10） | 随机 IO 较多 |
| 支持增量更新 | 参数调优较复杂 |

## 2. Vamana 图算法

Vamana 是 DiskANN 的核心图构建算法，与 HNSW 的多层结构不同，Vamana 构建的是**单层稀疏邻居图**，通过精心设计的剪枝策略保证图的可导航性。

### 2.1 图结构

```
      ┌─────●─────┐
      │     │     │
      ●     │     ●      长程边（稀疏跳） + 短程边（精细搜） = 可导航小世界
      │╲    │    ╱│
      ●─────●─────●
```

Vamana 图的关键特征：
- **单层结构**：所有节点在同一层，无需分层导航
- **长程边 + 短程边**：既有远距离跳跃边，也有近距离精细边
- **有向图**：A→B 不意味着 B→A，但实践中通常双向连接

### 2.2 Robust Prune 剪枝策略

Robust Prune 是 Vamana 的核心创新，解决"给定候选池，如何选择最优邻居子集"的问题。

**问题**：节点 v 有候选池 C（按距离排序），需要从中选出最多 R 个邻居，使得从 v 出发能快速到达 C 中的每个点。

**朴素方法的问题**：直接选最近的 R 个候选，会导致所有邻居都聚集在 v 附近，形成"局部聚团"，远处的点难以到达。

**Robust Prune 算法**：

```
RobustPrune(v, C, R, α):
    1. 初始化结果集 S = ∅
    2. 将 C 按到 v 的距离排序（近→远）
    3. while C 非空 且 |S| < R:
        a. 取 C 中最近的点 p*
        b. 将 p* 加入 S，从 C 中移除 p*
        c. 对 C 中剩余的每个点 p:
            如果 α × dist(p*, p) ≤ dist(v, p):
                从 C 中移除 p    // p 被 p* "遮挡"
    4. 返回 S
```

**遮挡原理**：如果从 v 到 p* 比到 p 更近，且从 p* 到 p 也不远（α 倍以内），那么 p 被 p* "遮挡"——从 v 出发先到 p* 再到 p，不会比直接到 p 差太多。因此 p 不需要作为 v 的直接邻居。

**α 参数**：控制遮挡容忍度
- α = 1.0：严格遮挡，只保留真正必要的边，图最稀疏
- α > 1.0：容忍一定绕路，保留更多边，图更稠密，搜索更快
- 典型值：1.2

### 2.3 多轮剪枝流程

Vamana 构建采用两轮策略，避免早期过度丢弃：

```
VamanaBuild(vectors, R, L):
    1. 初始化：为每个节点随机选 R 个邻居
    2. 第一轮（α = 1.0）：
       - 对每个节点 v，从入口点搜索 L 个最近邻
       - 用 RobustPrune(v, candidates, R, 1.0) 剪枝
    3. 第二轮（α = 1.2）：
       - 对每个节点 v，从入口点搜索 L 个最近邻
       - 用 RobustPrune(v, candidates, R, 1.2) 剪枝
    4. 返回图
```

**为什么需要两轮**：
- 第一轮 α=1.0 严格剪枝，建立稀疏骨架，保证长程边不被过度剪掉
- 第二轮 α=1.2 放松剪枝，补充短程边，提高搜索精度
- 如果一开始就用 α=1.2，长程边容易被近处候选遮挡掉

### 2.4 冻结点种子

搜索和构图都从**冻结点**（frozen points）出发，保证图的可导航性：

- 冻结点是预先选定的少量节点（通常 1-10 个），在构图过程中位置固定
- 所有搜索从冻结点开始，确保不会因为入口点选择不当而陷入局部最优
- 冻结点通过在数据集上运行 K-Means 或随机采样选取

### 2.5 参数影响

| 参数 | 作用 | 增大时 |
|------|------|--------|
| R | 每节点最大邻居数 | 召回↑ 内存↑ 搜索↓ |
| L | 构图搜索候选数 | 图质量↑ 构建时间↑ |
| α | 遮挡容忍度 | 边数↑ 搜索速度↑ 内存↑ |

## 3. Merge-Vamana（分图合并）

### 3.1 适用场景

当数据量超过单机内存或单次构图时间过长时（亿级向量），Merge-Vamana 将数据分区后独立构建子图，再合并为全局图。

### 3.2 分区策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| Random | 随机分配，简单高效 | 数据无明显簇结构 |
| K-Means | 聚类分区，同簇向量在同一分区 | 数据有明显簇结构 |
| Coordinate-Range | 按坐标范围切分 | 地理/坐标数据 |

### 3.3 合并流程

```
 ┌─────────┐  ┌─────────┐  ┌─────────┐
 │ 子图 G1  │  │ 子图 G2  │  │ 子图 G3  │     ← 独立分区构建
 └────┬────┘  └────┬────┘  └────┬────┘
      └───────────┼───────────┘
                  ↓
         ┌─────────────────┐
         │  节点去重 + 边合并 │
         │  跨分区边构建    │
         └────────┬────────┘
                  ↓
         ┌─────────────────┐
         │  全局图 G       │
         └─────────────────┘
```

**合并步骤**：
1. **独立构建**：每个分区独立运行 Vamana 构建子图
2. **节点去重**：重叠区域的节点在多个子图中出现，需要去重
3. **边合并**：将各子图的边合并到全局图，对同一节点的邻居取并集
4. **跨分区边构建**：对重叠区域的节点，用 Robust Prune 补充跨分区边
5. **全局剪枝**：对合并后的图运行一轮 Robust Prune，控制邻居数

### 3.4 参数说明

| 参数 | 说明 | 建议值 |
|------|------|--------|
| partition_count | 分区数量 | 4-16 |
| overlap_ratio | 重叠比例（0.0-1.0） | 0.1 |

**重叠比例**：每个分区额外包含相邻分区 10% 的向量，用于构建跨分区边。overlap_ratio 越大，跨分区边质量越高，但构建时间也越长。

## 4. SPANN (SParse ANN)

### 4.1 核心思想

SPANN 是分区层次索引，通过将向量空间预先划分为多个分区，搜索时只访问最近的几个分区，大幅减少磁盘 IO。

### 4.2 搜索流程

```
 Query ──→ Partition Selection (路由)
                 │
     ┌───────────┼───────────┐
     ↓           ↓           ↓
   P1(skip)   P2(search)  P3(search)   ← 根据距离选择分区
     ↓           ↓           ↓
   跳过      搜索子图     搜索子图
                 ↓
         结果合并 → Top-K
```

**搜索步骤**：
1. **分区路由**：计算查询向量到各分区中心的距离，选择最近的 search_partitions 个分区
2. **分区搜索**：在选中分区内执行 Vamana 图搜索
3. **结果合并**：合并各分区搜索结果，返回 Top-K

### 4.3 参数说明

| 参数 | 说明 | 建议值 |
|------|------|--------|
| max_partitions | 最大分区数 | 64-256 |
| search_partitions | 搜索时访问的分区数 | 4-16 |
| min_partition_size | 每分区最小向量数 | 避免过多小分区 |

**search_partitions 与召回率**：search_partitions 越大，召回率越高，但 IO 也越多。通常 search_partitions = 4-8 即可达到 95%+ 召回率。

## 5. FreshDiskANN（增量更新）

### 5.1 核心思想

将索引分为**静态区（Frozen）**和**动态区（Fresh）**，支持高频增量插入，无需全量重建。

### 5.2 双区架构

```
 ┌─────────────────────┐    ┌─────────────────────┐
 │   静态区 (Frozen)   │    │   动态区 (Fresh)    │
 │                     │    │                     │
 │  向量 + 图结构       │    │  向量 + 图结构       │
 │  磁盘持久化         │    │  内存索引           │
 └─────────────────────┘    └─────────────────────┘
          ↓                          ↑
   搜索(磁盘IO)              插入(直接写入)
          ↓                          │
          └────────┬─────────────────┘
                   ↓
             结果合并 → Top-K
```

**静态区（Frozen）**：
- 存储在 SSD 上，包含已构建好的 Vamana 图
- 数据量大，搜索需要磁盘 IO
- 图结构稳定，搜索效率高

**动态区（Fresh）**：
- 存储在内存中，使用简化的图结构（如 HNSW 或朴素邻居图）
- 支持高频插入，新向量直接写入内存
- 数据量小，搜索速度快

### 5.3 搜索流程

1. 在静态区执行磁盘搜索，获取候选集 A
2. 在动态区执行内存搜索，获取候选集 B
3. 合并 A 和 B，返回 Top-K

### 5.4 合并策略

当动态区数据量达到 `merge_threshold` 时，触发合并：

1. 将动态区数据写入磁盘
2. 对合并后的数据重新运行 Vamana 构建（或增量构建）
3. 新的静态区替换旧的静态区
4. 清空动态区

**合并代价**：合并期间搜索性能可能下降（需要同时访问新旧静态区），但不会中断服务。

### 5.5 参数说明

| 参数 | 说明 | 建议值 |
|------|------|--------|
| fresh_capacity | 动态区容量 | 根据内存大小设置 |
| merge_threshold | 触发合并的阈值 | capacity 的 80% |

## 6. 量化压缩

DiskANN 的内存效率依赖于向量量化压缩，将高维浮点向量压缩为短码。

### 6.1 支持的量化方法

| 方法 | 说明 | 压缩比 | 精度损失 |
|------|------|--------|----------|
| PQ (Product Quantization) | 子空间独立量化 | 32x-64x | 中等 |
| SQ (Scalar Quantization) | 每维度独立量化 | 4x-8x | 较低 |
| LVQ (Learned Vector Quantization) | 学习型量化 | 8x-16x | 较低 |
| RQ (Residual Quantization) | 残差递进量化 | 32x-64x | 较低 |

### 6.2 PQ 在 DiskANN 中的作用

1. **内存驻留**：PQ 码本存储在内存中，用于距离预筛
2. **两阶段搜索**：
   - 第一阶段：用 PQ 码本计算近似距离，筛选候选
   - 第二阶段：从磁盘加载精确向量，重排结果
3. **参数**：
   - `subquantizers`（m）：子空间数，通常 32-64
   - `bits`：每子空间位数，通常 4/6/8

## 7. 时间/空间复杂度

| 操作 | 时间复杂度 | 空间复杂度 | 说明 |
|------|------------|------------|------|
| 构建 | O(n² log n) | O(n × R) | Vamana 两轮遍历 |
| 搜索 | O(R × log n) | O(L) | L 为搜索候选数 |
| 插入 | O(R × log n) | O(R) | Fresh 模式 |
| 合并 | O(n × R) | O(n × R) | Fresh→Frozen |

**内存占用**：O(n × m × bits/8)，其中 m 为 PQ 子空间数，bits 为每子空间位数。例如 10 亿向量、128 维、PQ-32×8bit，内存约 32GB。

**磁盘占用**：O(n × d × 4)，存储全精度向量 + 图结构。

## 8. 与其他索引对比

| 索引 | 搜索复杂度 | 内存占用 | 磁盘支持 | 增量更新 | 适用场景 |
|------|------------|----------|----------|----------|----------|
| **DiskANN** | O(R log n) | 低（PQ 码本） | SSD | Fresh 模式 | 超大规模、内存受限 |
| **HNSW** | O(log n) | 高（全内存） | 无 | 增量插入 | 高召回、动态更新 |
| **IVF** | O(√n) | 中 | 可选 | 需重训练 | 大数据集、精确控制 |
| **LSH** | O(k log n) | 高 | 无 | 增量插入 | L2/内积距离 |
| **ScaNN** | O(log n) | 中 | 无 | 有限 | 高吞吐、低延迟 |

### DiskANN vs HNSW

| 维度 | DiskANN | HNSW |
|------|---------|------|
| 图结构 | 单层 Vamana 图 | 多层 NSW 图 |
| 存储 | SSD + 内存 PQ | 全内存 |
| 容量 | 十亿级 | 受内存限制（通常亿级以内） |
| 搜索延迟 | ~1ms（SSD） | ~0.1ms（内存） |
| 构建时间 | 长（两轮遍历） | 中等 |
| 增量更新 | Fresh 双区 | 直接插入 |
| 召回率 | 95%+ @10 | 99%+ @10 |
