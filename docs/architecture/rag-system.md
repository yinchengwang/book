## 7. 面试高频知识点

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           RAG 面试高频问题                                     │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  分类 1: 基础概念                                                               │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  Q1: 什么是 RAG？为什么要用 RAG？                                       │   │
│  │  A:                                                                        │   │
│  │  - RAG = Retrieval-Augmented Generation (检索增强生成)                   │   │
│  │  - 用途: 让 LLM 引用最新/私有知识库回答问题，减少幻觉                    │   │
│  │  - 优势: 可更新知识、成本低于微调、可追溯答案来源                        │   │
│  │                                                                          │   │
│  │  Q2: RAG 的核心流程是什么？                                             │   │
│  │  A:                                                                        │   │
│  │  - 数据准备: 文档解析 → 分块 → 向量化 → 存储                            │   │
│  │  - 查询处理: 查询向量化 → 检索 → 重排序 → 上下文组装                    │   │
│  │  - 生成: Prompt + 上下文 → LLM → 输出答案                               │   │
│  │                                                                          │   │
│  │  Q3: RAG vs Fine-tuning 的区别？                                        │   │
│  │  A:                                                                        │   │
│  │  - RAG: 动态知识、即时更新、可追溯、成本低                               │   │
│  │  - Fine-tuning: 隐式知识、训练时间长、推理快、适合格式/风格              │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
│  分类 2: 分块策略                                                               │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  Q4: 如何选择合适的 chunk_size？                                        │   │
│  │  A:                                                                        │   │
│  │  - 太长: 引入噪声、超过 LLM 上下文窗口、稀释相关性                      │   │
│  │  - 太短: 丢失上下文、无法完整表达语义                                    │   │
│  │  - 经验值: 500-1000 tokens，常用 512 tokens                             │   │
│  │                                                                          │   │
│  │  Q5: chunk_overlap 的作用？                                             │   │
│  │  A:                                                                        │   │
│  │  - 保证跨 chunk 的信息不丢失                                             │   │
│  │  - 通常设为 chunk_size 的 10-20%                                        │   │
│  │                                                                          │   │
│  │  Q6: 语义分块 vs 固定分块的优缺点？                                     │   │
│  │  A:                                                                        │   │
│  │  - 语义分块: 保持语义完整，但大小不均匀                                  │   │
│  │  - 固定分块: 大小可控，但可能切断语义                                    │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
│  分类 3: 向量检索                                                               │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  Q7: 什么是 ANN 检索？为什么不用精确检索？                              │   │
│  │  A:                                                                        │   │
│  │  - ANN = Approximate Nearest Neighbor                                   │   │
```
