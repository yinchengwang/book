### 2.2 列式存储优势

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           列式存储 vs 行式存储                                   │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  查询场景: SELECT AVG(amount) FROM orders WHERE status = 'completed'            │
│                                                                                 │
│  ┌────────────────────────────────┬────────────────────────────────────────┐    │
│  │         行式存储               │              列式存储                 │    │
│  ├────────────────────────────────┼────────────────────────────────────────┤    │
│  │                                │                                        │    │
│  │  ┌────┬────┬────┬────┐       │  amount 列数据:                         │    │
│  │  │ 100│ 200│ 150│ 300│       │  [100, 200, 150, 300, 250, ...]         │    │
│  │  │ id │ cid│ amt│ st │       │                                         │    │
│  │  └────┴────┴────┴────┘       │  status 列数据:                         │    │
│  │  ┌────┬────┬────┬────┐       │  ["ok","pend","ok","canc",...]          │    │
│  │  │101 │ C1 │ 200│ ok │       │                                         │    │
│  │  └────┴────┴────┴────┘       │  只需读取两列: amount + status           │    │
│  │  ... 读取所有行所有列 ...     │  ✅ I/O 减少 60%+                       │    │
│  │                                │  ✅ 同列同类型，压缩率更高              │    │
│  │                                │  ✅ SIMD 向量化批量计算                 │    │
│  │  ❌ I/O 量大                   │                                        │    │
│  │  ❌ 无法利用 CPU 缓存          │                                        │    │
│  │  ❌ 无压缩优化                 │                                        │    │
│  └────────────────────────────────┴────────────────────────────────────────┘    │
│                                                                                 │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                           性能对比 (典型场景)                             │   │
│  │  ┌──────────────────────────────────────────────────────────────────┐  │   │
│  │  │  场景                    │  列式 vs 行式                          │  │   │
│  │  │--------------------------│----------------------------------------│  │   │
│  │  │  全列聚合 (COUNT/SUM)    │  5-10x 提升                            │  │   │
│  │  │  多列查询 (SELECT a,b,c) │  3-5x 提升                             │  │   │
│  │  │  点查询 (WHERE id = N)   │  0.5-1x (劣势)                         │  │   │
│  │  │  全表扫描                │  2-3x 提升                             │  │   │
│  │  └──────────────────────────────────────────────────────────────────┘  │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 2.3 执行引擎向量化

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                              向量化执行引擎                                       │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  传统行式火山模型 vs 向量化执行:                                                  │
│                                                                                 │
│  ┌────────────────────────────────┬────────────────────────────────────────┐    │
│  │   行式火山模型 (Row-based)      │       向量化执行 (Vectorized)          │    │
│  ├────────────────────────────────┼────────────────────────────────────────┤    │
│  │                                │                                        │    │
│  │  while ((row = getNext()) !=   │  while ((batch = getBatch()) != NULL) │    │
│  │         NULL) {                │  {                                     │    │
│  │      process(row);             │      // 一次处理 1024 行                │    │
│  │  }                             │      process_batch(batch);  // SIMD   │    │
│  │                                │  }                                     │    │
│  │  每行一次函数调用                │  每批次一次函数调用                     │    │
│  │  CPU 缓存命中率低               │  CPU 缓存命中率高                       │    │
│  │  无 SIMD 优化                  │  自动 SIMD 向量化                       │    │
│  │  分支预测开销大                 │  分支预测减少                           │    │
│  └────────────────────────────────┴────────────────────────────────────────┘    │
│                                                                                 │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  SIMD 加速原理:                                                           │   │
│  │                                                                          │   │
│  │  标量计算 (一次处理 1 个值):                                               │   │
│  │    result = a[0]+b[0], result = a[1]+b[1], result = a[2]+b[2], ...      │   │
│  │                                                                          │   │
│  │  SIMD 向量计算 (一次处理 4/8 个值):                                        │   │
│  │    result[0:4] = a[0:4] + b[0:4]  // AVX2 256bit = 4 × int64            │   │
│  │    result[4:8] = a[4:8] + b[4:8]  // 理论加速比 4-8x                       │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 2.4 查询优化器

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                              查询优化器                                          │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  查询: SELECT c.name, SUM(o.amount)                                             │
│        FROM orders o JOIN customers c ON o.customer_id = c.id                   │
│        WHERE o.create_time >= '2024-01-01'                                      │
│        GROUP BY c.name                                                          │
│        HAVING SUM(o.amount) > 10000                                             │
│        ORDER BY SUM(o.amount) DESC                                              │
│        LIMIT 10;                                                                │
│                                                                                 │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                         逻辑优化 (Rule-based)                            │   │
│  │                                                                          │   │
│  │  1. 列裁剪 (Column Pruning)                                              │   │
│  │     - 只读取: c.name, o.amount, o.create_time, o.customer_id            │   │
│  │     - 跳过: o.id, c.address, c.phone, ...                               │   │
│  │                                                                          │   │
│  │  2. 谓词下推 (Predicate Pushdown)                                        │   │
│  │     - 过滤条件 o.create_time >= '2024-01-01' 在存储层执行               │   │
│  │     - 减少读取数据量                                                      │   │
│  │                                                                          │   │
│  │  3. 分区裁剪 (Partition Pruning)                                         │   │
│  │     - 根据时间分区，只扫描 2024 年分区                                    │   │
│  │                                                                          │   │
│  │  4. 聚合下推 (Aggregation Pushdown)                                      │   │
│  │     - 部分聚合在存储层完成，减少数据传输                                   │   │
│  │                                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                    │                                           │
│                                    ▼                                           │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │                         物理优化 (Cost-based)                            │   │
│  │                                                                          │   │
│  │  ┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐    │   │
│  │  │   Scan: 10s     │     │   Join: 5s      │     │  Aggregate: 2s  │    │   │
│  │  │                 │     │                 │     │                 │    │   │
│  │  │ Filter: date    │────▶│ Hash Join      │────▶│ GROUP BY name   │    │   │
│  │  │                 │     │ Build: customers│     │                 │    │   │
│  │  │ 1000 rows       │     │ Probe: orders   │     │ 500 groups      │    │   │
│  │  └─────────────────┘     └─────────────────┘     └────────┬────────┘    │   │
│  │                                                           │             │   │
│  │                                                           ▼             │   │
│  │                                                   ┌─────────────┐       │   │
│  │                                                   │   Sort: 1s  │       │   │
│  │                                                   │  ORDER BY   │       │   │
│  │                                                   │  Top 10     │       │   │
│  │                                                   └─────────────┘       │   │
│  │                                                                          │   │
│  │  估算总成本: 18s                                                          │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```
