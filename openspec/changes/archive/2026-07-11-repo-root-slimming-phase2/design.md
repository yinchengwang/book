## Context

Phase 1 的设计是先迁移真实内容并保留旧顶层兼容入口，防止隐藏脚本、用户习惯和外部路径在同一变更中被硬切断。Phase 2 是配套删除阶段：只有当 canonical 路径稳定、旧路径引用被清理、验证矩阵通过后，才删除这些临时入口，完成根目录白名单最终状态。

## Goals / Non-Goals

**Goals:**

- 删除 Phase 1 创建或保留的旧顶层兼容 README、薄包装、alias 或清晰报错入口。
- 清理 active 文档、脚本、CI、CMake、测试和 smoke 中的旧顶层 canonical 引用。
- 验证根目录只剩共享区白名单与必要配置文件。
- 保留历史记录中的旧路径说明，但明确它们不是可用入口。
- 通过最终验证矩阵确认删除兼容入口后双轨仍可构建、测试和运行关键 smoke。

**Non-Goals:**

- 不重新迁移真实内容；真实内容迁移属于 Phase 1。
- 不恢复旧顶层目录作为长期入口。
- 不保留真实代码副本、符号链接或目录联接。
- 不扩大目录重组范围到 Phase 1 未覆盖的新架构调整。

## Decisions

### 1. Phase 2 删除所有旧顶层兼容入口

Phase 2 的核心交付是删除 Phase 1 保留的旧顶层 README/包装入口，使根目录最终满足共享区白名单。若某个旧入口仍被 active 代码或脚本依赖，必须先修复依赖，再删除入口；不得以“仍有人用”为理由长期保留。

替代方案是继续保留 README 指路；该方案会让兼容窗口无限延长，因此不采用。

### 2. 删除前必须通过引用清零闸门

删除前必须扫描旧顶层路径引用。active 文档、脚本、CI、CMake、测试、smoke 命令中的路径必须切到 canonical 路径。允许保留的旧路径引用只包括 OpenSpec 历史记录、归档说明、迁移记录或明确标记的历史上下文。

### 3. 根目录白名单作为最终验收

Phase 2 以根目录白名单为最终验收标准。除 `engineering/`、`learning/`、`docs/`、`openspec/`、`archive/`、`cmake/`、`scripts/`、`third_part/`、`reference/` 和必要配置文件外，不应存在历史业务、学习、演示、RAG、SDK、工具兼容入口。

### 4. 删除动作按兼容入口分批执行

Phase 2 仍按小批次执行：先引用扫描和删除清单，再删除工程旧入口，再删除学习旧入口，再删除 `tools/` 旧入口，最后做全量验证。失败时只回退当前批次。

## Risks / Trade-offs

- [仍有隐藏脚本依赖旧路径] → 删除前执行引用扫描和关键 smoke；若发现依赖，先修引用再删入口。
- [历史文档中的旧路径被误判为 active 引用] → 扫描报告区分 active 引用和历史引用，历史引用必须明确标记为历史上下文。
- [删除入口导致用户短期不适应] → Phase 1 已提供完整兼容窗口；Phase 2 文档保留迁移说明但不保留可用旧入口。
- [删除范围过大难以回退] → 按工程入口、学习入口、tools 入口分批删除，失败只回退当前批次。

## Migration Plan

1. 读取 Phase 1 的兼容入口清单和最终验证记录。
2. 扫描旧顶层路径引用，并分类为 active 引用、历史引用、OpenSpec/归档引用。
3. 修复 active 引用，确保全部指向 canonical 路径。
4. 分批删除工程旧入口、学习旧入口、`tools/` 旧入口。
5. 更新根 README、CLAUDE、AGENTS、架构文档和轨道 README，移除“临时兼容入口仍可使用”的表述。
6. 运行双轨构建/CTest、根入口构建、关键 smoke、产物落点检查和最终根目录白名单检查。

## Open Questions

Phase 2 的具体删除清单依赖 Phase 1 实施结果；Phase 1 完成时必须把兼容入口清单写入 Phase 2 tasks 或迁移记录。
