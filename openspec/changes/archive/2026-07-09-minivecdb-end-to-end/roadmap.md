## Phase A 详细计划

### Week 1-2: A1 向量索引持久化层

**目标**: 让向量数据可以落盘

**关键交付物**:
- `include/db/storage/vector_persist.h` - 持久化接口
- `src/db/storage/vector_persist.c` - 持久化实现
- `src/db/storage/vector/*.c` - 对接现有存储引擎

**里程碑**:
- 向量插入后可从磁盘恢复

### Week 3-4: A2 向量查询执行器

**目标**: 统一的向量查询管道

**关键交付物**:
- `src/db/core/vector_exec.c` - 执行器实现
- 两阶段检索管道集成
- Hybrid Search 集成

**里程碑**:
- 粗排→精排管道可工作

### Week 5-6: A3 API 层 + CLI 工具

**目标**: 可访问的用户接口

**关键交付物**:
- `proto/vector_service.proto` - gRPC 接口定义
- `src/db/api/` - gRPC 服务实现
- CLI 向量命令扩展

**里程碑**:
- 可通过 CLI/API 操作向量集合

### Week 7-8: A4 端到端集成测试

**目标**: 验证全链路正确性

**关键交付物**:
- `test/db/integration/` - 集成测试套件
- Collection CRUD 测试
- 故障恢复测试

**里程碑**:
- 所有集成测试通过

### 执行规则

1. **顺序执行**: 严格按 A1 → A2 → A3 → A4 → B1 → B2 → B3 → C2 顺序
2. **C1 可并行**: benchmark-suite (C1) 可在 Phase A 完成后与其他 B 阶段并行执行
3. **提交时机**: 每个小变更完成后立即提交代码并推送
4. **上下文压缩**: 提交后压缩上下文，保留变更状态记录
5. **循环执行**: 直到所有小变更完成，大变更才算完成
