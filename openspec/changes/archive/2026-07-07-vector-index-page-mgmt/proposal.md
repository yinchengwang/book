# 向量索引页管理与图去重优化

## Summary

优化向量索引层的存储和搜索质量，解决以下问题：
1. **重复向量污染图索引**：相同向量值在图中形成重复节点和边，降低搜索效率和精度
2. **空向量进入索引**：NULL/全零向量应仅存储在 Heap 中，不应参与图索引构建
3. **缺乏页管理**：HNSW 等图索引需要基于页的内存管理和持久化支持

## Motivation

### 问题 1：重复向量污染

```
当前问题：
┌─────────────────────────────────────────────────────────────┐
│ Heap 表层                                                   │
│  row_id │ id    │ vector                   │ metadata      │
│  ───────┼───────┼─────────────────────────┼───────────────│
│    1    │ 1001  │ [0.1, 0.2, 0.3, ...]   │ {...}        │
│    2    │ 1002  │ [0.1, 0.2, 0.3, ...]   │ {diff}       │ ← 相同向量
│    3    │ 1003  │ [0.1, 0.2, 0.3, ...]   │ {diff}       │ ← 相同向量
│    4    │ 1004  │ [0.4, 0.5, 0.6, ...]   │ {...}        │
├─────────────────────────────────────────────────────────────┤
│ 图索引层                                                    │
│  node_id │ vector              │ connections                │
│  ────────┼────────────────────┼───────────────────────────│
│    0     │ [0.1, 0.2, 0.3]    │ ← 0, 1, 2 互连，冗余！    │
│    1     │ [0.1, 0.2, 0.3]    │ ← 0, 1, 2 互连，冗余！    │
│    2     │ [0.1, 0.2, 0.3]    │ ← 0, 1, 2 互连，冗余！    │
│    3     │ [0.4, 0.5, 0.6]    │                           │
│                                                             │
│  问题：3 个相同向量在图中形成 6 条冗余边                      │
│  影响：搜索时可能返回多个距离为 0 的重复结果                  │
└─────────────────────────────────────────────────────────────┘

解决方案：
┌─────────────────────────────────────────────────────────────┐
│ 图索引层（去重后）                                           │
│  node_id │ vector              │ graph_connections          │
│  ────────┼────────────────────┼───────────────────────────│
│    0     │ [0.1, 0.2, 0.3]    │ ← 只保留 1 个节点         │
│    3     │ [0.4, 0.5, 0.6]    │                           │
│                                                             │
│  反向映射表：                                               │
│  graph_node_id → heap_row_ids                               │
│      0 → [1, 2, 3]                                          │
│      3 → [4]                                                │
└─────────────────────────────────────────────────────────────┘
```

### 问题 2：空向量污染

```
场景：向量字段可以为 NULL 或全零

当前问题：
- NULL/全零向量仍会插入图索引
- 图中出现孤立节点（与其他节点距离无穷大）
- 搜索结果可能包含无意义的空向量

解决方案：
- Heap 层：允许 NULL 向量，完整记录元数据
- 图索引层：跳过 NULL/全零向量，仅存储索引指针
```

### 问题 3：缺乏页管理

```
现有 HNSW 实现：
- 纯内存结构，无持久化
- 无内存页管理，大规模数据易 OOM
- DiskANN 有页管理但未与向量引擎集成

解决方案：
- 提取通用向量页管理模块
- 支持向量数据的分页加载/卸载
- 与 Heap 存储层解耦
```

## Goals

### 核心目标

1. **图去重机制** ✅ 已实现
   - 在向量插入索引前检测重复向量值
   - 使用 SimHash + Bloom Filter 高效检测
   - 建立反向映射表支持搜索结果回溯

2. **空向量过滤** ✅ 已实现
   - 检测并跳过 NULL/全零向量
   - 向量引擎层：向量可为 NULL
   - 图索引层：仅索引有效向量

3. **页管理架构** ✅ 已实现
   - 向量页管理模块 (vec_page)
   - Clock-Sweep 置换策略
   - 支持内存页和持久化

## Scope

### In Scope

- 新增 `vector_page` 模块：向量页管理
- 新增 `graph_dedup` 模块：图索引去重
- 修改 `vector_engine` 集成去重和过滤
- 修改 `diskann` 集成去重和过滤
- 新增单元测试

### Out of Scope

- 修改现有 `faiss_hnsw` 实现
- 实现完整的磁盘索引（后续迭代）
- 向量更新/删除操作

## Alternatives Considered

### 方案 A：修改现有 faiss_hnsw（不采用）

```c
// 在 faiss_hnsw_insert.c 中添加去重检查
int32_t faiss_hnsw_index_add_with_dedup(faiss_hnsw_t *index, int32_t n, 
                                         const float *vectors, bool *duplicates);
```

**缺点**：
- 需要修改核心索引结构
- 破坏现有 API 兼容性
- 与磁盘索引整合困难

### 方案 B：抽象层包装（不采用）

```c
// 在向量引擎层包装现有索引
typedef struct {
    faiss_hnsw_t *hnsw;      // 底层索引
    hash_table_t *dedup;     // 去重表
    vector_page_t *page_mgr; // 页管理
} vector_index_wrapper_t;
```

**缺点**：
- 双层索引结构，搜索效率低
- 内存占用翻倍
- 代码复杂度高

### 方案 C：分层解耦（采用）

```
┌─────────────────────────────────────────────────────────────┐
│                        向量引擎层                            │
│              (vector_engine.c - 对外接口)                    │
└─────────────────────────┬───────────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────────────┐
│                    过滤与去重层                              │
│              (graph_dedup.c - SimHash + Bloom)              │
└─────────────────────────┬───────────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────────────┐
│                    图索引层                                  │
│              (hnsw_index_t / vamana_index_t)                │
└─────────────────────────┬───────────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────────────┐
│                    页管理层                                 │
│              (vector_page.c - 内存/磁盘页)                   │
└─────────────────────────────────────────────────────────────┘
```

**优点**：
- 各层职责清晰，模块化
- 支持多种图索引实现
- 易于扩展和测试
- 性能开销最小

## Risks

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| SimHash 冲突 | 少量有效向量被误判为重复 | Bloom Filter 二次确认 + 可配置阈值 |
| 内存占用增加 | 反向映射表占用额外内存 | 使用压缩存储 + LRU 淘汰 |
| 搜索延迟增加 | 结果回溯到 heap 开销 | 异步预取 + 缓存热点记录 |
