# MVCC 并发控制设计

## Context

当前数据库内核使用排他锁实现事务，读取时加读锁，写入时加写锁。读写互相阻塞，无法支持高并发场景。

已完成第2期基本款：WAL 日志、事务支持、SQL 解析器、表管理。

现在需要实现 MVCC，使数据库支持快照隔离级别（SI），实现读写不阻塞。

## Goals / Non-Goals

**Goals:**
- 实现行级版本链，支持多版本存储
- 实现事务快照，支持快照读
- 实现可见性判断，返回正确版本
- 实现 VACUUM 垃圾回收，清理过期版本
- 支持 Snapshot Isolation 隔离级别

**Non-Goals:**
- 不实现 Serializable Snapshot Isolation (SSI)
- 不实现乐观锁冲突检测（写倾斜检测）
- 不实现分布式 MVCC（跨节点版本协调）

## Decisions

### 决策 1: 版本存储方式 - Append-only 存储

**选择：** Append-only 存储 + Undo 日志

**原因：**
- PostgreSQL 使用的方法，稳定可靠
- 新版本追加到文件末尾，旧版本通过 Undo 链访问
- 崩溃恢复简单，只需处理 Undo 日志

**替代方案：**
- 原地更新 + 版本链指针（SQLite 风格）：简单但并发控制复杂
- 原地更新 + 垃圾回收（MySQL InnoDB 风格）：实现复杂，需要 purger 线程

### 决策 2: 版本链结构 - 纵向版本链

**选择：** 纵向版本链（每行一个版本链表）

```
[HEAD] → [版本3] → [版本2] → [版本1] → NULL
         (最新)     (次新)   (最旧)
```

**原因：**
- 读取时从 HEAD 向下遍历找到可见版本
- 更新时在链表头部插入新版本
- 实现简单，版本查找 O(n)，但 n 通常很小（1-3）

**替代方案：**
- 横向版本链（事务维度）：复杂但对大事务友好
- 时间戳索引：需要额外索引结构

### 决策 3: 快照数据结构 - 活跃事务列表

**选择：** 记录 Xmin（创建事务ID）和 Xmax（删除事务ID）

```
版本可见性规则：
1. Xmin 未提交 → 不可见
2. Xmax 已提交且 Xmax < snapshot.xmax → 对快照不可见（已删除）
3. Xmax = 0 且 Xmin 已提交 → 可见
```

**原因：**
- PostgreSQL 使用的方法
- 快照只需记录：活跃事务列表 + 最小未分配事务ID
- 判断可见性 O(n)，但 n 通常很小

### 决策 4: 垃圾回收策略 - 懒清理

**选择：** 后台 VACUUM 线程 + 增量清理

**原因：**
- 不阻塞前台操作
- 每次清理一部分，避免长时间停顿
- 与 PostgreSQL 一致

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 版本链过长 | 读取性能下降 | VACUUM 及时清理，控制链长 < 10 |
| Undo 空间膨胀 | 磁盘占用增加 | 定期清理过期 Undo |
| 快照内存占用 | 内存压力 | 限制活跃事务数量，超出则报错 |
| 长事务阻塞 GC | 版本堆积 | 警告长事务，建议拆分 |

## Open Questions

1. **版本淘汰策略**：当版本链超过一定长度（如 20）时，是否强制触发 GC？
2. **大事务支持**：对于修改超过 N 个版本的长事务，是否需要特殊处理？
3. **与其他隔离级别兼容**：是否需要支持 READ COMMITTED？目前只实现 SI。
