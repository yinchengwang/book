# Build My DB - 技术设计

## Context

这是一个从零构建数据库内核的项目，目标是在学习数据库原理的同时产出一个实际可用的工具。

**当前状态：**
- 尚未开始实现
- 项目已有丰富的索引代码（ART、B+Tree、LSM、SkipList）可复用

**约束：**
- 单人业余项目，无截止日期
- 存储介质：磁盘文件
- 语言：C / C++
- 规模：中等（几 GB 级别）

## Goals / Non-Goals

**Goals：**
- 构建一个功能完整的数据库内核
- 分4期迭代，每期产出可用版本
- 第1期：KV数据库（有存储、有索引）
- 第2期：支持简单SQL和事务
- 第3期：MVCC + 多种索引 + 查询优化
- 第4期：分布式支持

**Non-Goals：**
- 不追求与PostgreSQL/MySQL的功能完整度
- 不做分布式/多节点（4期再做）
- 不做存储过程/触发器（后期考虑）
- 不做高并发压测优化

## Decisions

### 决策1：页面大小 8KB

**选择：** 8KB 页面大小
**理由：**
- 4KB：系统页大小，I/O效率高，但存储开销大
- 8KB：PostgreSQL默认，折中方案
- 16KB/32KB：适合大字段，但内存占用高

**替代方案：**
- 4KB：更细粒度，但页数多管理复杂
- 16KB：减少页数，但小记录浪费空间

---

### 决策2：以 ART 为默认索引

**选择：** Adaptive Radix Tree（自适应基数树）
**理由：**
- 项目已有完整实现（src/index/tree/art/）
- 支持字符串键（SQL场景更通用）
- 内存占用低，前缀压缩好
- 范围查询友好

**替代方案：**
- B+Tree：最经典，但字符串性能差
- LSM-Tree：写优化，但读性能差
- SkipList：简单，但空间效率低

---

### 决策3：WAL 先行的事务策略

**选择：** 先实现WAL，再做MVCC
**理由：**
- WAL相对简单，崩溃恢复立竿见影
- MVCC需要更多设计工作
- 符合"先简单后复杂"原则

**替代方案：**
- 直接上MVCC：复杂度高，但一步到位
- 无事务MVP：简单，但数据安全无保障

---

### 决策4：C语言为主，部分C++

**选择：** 底层用C，上层用C++
**理由：**
- 存储引擎/索引：性能关键，用C
- SQL解析/执行：复杂度高，用C++
- 项目整体风格一致

**替代方案：**
- 纯C：避免C++开销，但代码量大
- 纯C++：开发效率高，但二进制大

---

### 决策5：文件布局

```
db/
├── data.db        # 主数据文件（页面存储）
├── wal.log        # WAL日志文件
└── meta.db        # 元数据（表结构、索引信息）
```

**理由：**
- 分离WAL便于恢复
- 元数据独立便于扩展
- 简单直观

## Risks / Trade-offs

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| 范围蔓延 | 功能太多做不完 | 严格按4期划分，每期独立可用 |
| 性能问题 | 自研存储可能慢 | 参考PostgreSQL设计，后续优化 |
| 设计缺陷 | 早期设计影响后期 | 设计文档评审，必要时重构 |
| 热情消退 | 长期项目容易放弃 | 每期设小目标，保持成就感 |
