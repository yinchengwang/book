# 查询优化器设计

## Context

当前查询执行器直接执行 SQL 解析生成的 AST，没有经过任何优化。需要构建一个查询优化器来提升查询性能。

数据库已有：存储引擎、MVCC、SQL 解析器、查询执行器骨架。

## Goals / Non-Goals

**Goals:**
- 实现规则优化器（RBO）：谓词下推、常量折叠、子查询展开
- 实现代价优化器（CBO）：基于统计信息的代价估算
- 实现索引选择器：自动选择最优索引
- 实现 EXPLAIN 命令：可视化执行计划

**Non-Goals:**
- 不实现完整的成本模型（先简化）
- 不实现分布式查询优化
- 不实现自适应查询优化

## Decisions

### 决策 1: 优化器架构 - 级联式优化

**选择：** 级联式优化框架

```
SQL → Parser → AST → Optimizer → Plan → Executor
                         ↓
                    规则优化 (RBO)
                         ↓
                    代价优化 (CBO)
```

**原因：**
- PostgreSQL 和 MySQL 使用的方法
- 规则优化先简化查询，代价优化精细调优
- 易于扩展新规则

### 决策 2: 计划表示 - 树形结构

**选择：** 树形计划节点

```
PlanNode
├── type: SCAN, FILTER, JOIN, PROJECT, ...
├── children: 子计划
├── cost: 代价估算
├── rows: 估计行数
└── specific fields per type
```

**原因：**
- 易于递归遍历和优化
- 类似 Volcano/Cascades 风格
- 支持树变换

### 决策 3: 统计信息存储 - 内联表头

**选择：** 在表头存储简单统计信息

表元数据包含：
- `row_count`: 估计行数
- `page_count`: 页面数
- `column_stats`: 列统计（基数、NULL比例）

**原因：**
- 实现简单，无需额外统计收集
- 对于 MVP 足够
- 未来可扩展为 ANALYZE 命令

### 决策 4: 代价模型 - 简单线性模型

**选择：** 代价 = IO成本 + CPU成本

```
cost = page_reads * seq_page_cost + rows * cpu_tuple_cost
```

**原因：**
- 简单实现
- 可区分顺序扫描 vs 索引扫描
- 未来可增加网络、内存成本

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 优化开销 | 复杂查询优化慢 | 限制优化迭代次数 |
| 统计不准 | 选择错误计划 | 提供手动 ANALYZE |
| 规则冲突 | 不同规则产生冲突 | 规则有序执行 |

## Open Questions

1. **优化级别**：是否需要多级优化（快速/完整）？
2. **Hint 语法**：是否支持用户指定的优化提示？
3. **缓存计划**：是否缓存优化后的执行计划？
