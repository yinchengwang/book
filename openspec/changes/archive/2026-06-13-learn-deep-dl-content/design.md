## Context

reading-radar 现有 300 个知识点的 items-registry，但 learn-deep 深度文章只覆盖了 93 个（linux/db/vdb 三栈）。C、CPP、DS、PY 四个栈合计 ~181 个知识点完全没有 learn-deep 内容；即使已有的 93 篇也缺乏深度学习维度的论述。

项目的 learn.html 渲染链路要求每个知识点必须有一个 item.id → markdown 的映射（通过 `LEARN_DEEP_CONTENT[item.id]` 或 `fetch("data/learn-deep/<id>.md")`），因此不能按主题聚合——必须保持 1:1 的每知识点一篇。

核心挑战：如何在 token 消耗可控的前提下，为 300 个知识点批量生成/补充高质量的深度学习关联内容。

## Goals / Non-Goals

**Goals:**
- 为全部 300 个知识点生成 learn-deep markdown 文章，每篇内容与 DL 关联度匹配
- 为 93 篇已有文章追加 DL 章节，不破坏原文
- 总 token 消耗控制在 ~200K（远低于逐篇 agent 的 500K+）
- 所有内容可渲染（更新 learn-deep-bundle.js）
- 生成过程自动编排，不需要人工逐条确认

**Non-Goals:**
- 不修改 items-registry.js 或任何现有数据源
- 不修改 learn.html 的渲染逻辑
- 不新增第三方依赖
- 不涵盖 DL 内容的深度审核（只做抽样校验）
- 不解决文章内容的学术准确性——仅保证技术连接合理

## Decisions

### Decision 1: Workflow 编排 + 分批子 agent（方式 B）

**选择：** 用 Claude Code 的 Workflow 工具自动编排整个流程。

**架构概览：**

```
                    ┌─────────────────────────┐
                    │    Workflow 入口脚本     │
                    │   learn-deep-workflow.js │
                    └─────────┬───────────────┘
                              │
             ┌────────────────┼────────────────┐
             ▼                ▼                 ▼
     ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
     │ Phase 1      │ │ Phase 2      │ │ Phase 3      │
     │ 分类 300 个   │ │ 分批生成 300  │ │ 集成+验证     │
     │ 知识点        │ │ 篇内容       │ │              │
     │ (1个子agent)  │ │ (20个子agent)│ │ (脚本)       │
     └──────────────┘ └──────────────┘ └──────────────┘
```

**备选方案对比：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 纯 Agent 逐个 | 质量最高 | ~500K+ tokens，2h+ |
| **B: Workflow 分批** | **~200K tokens，15-20min** | **需要一次编排** |
| C: 脚本模板填充 | ~55K tokens，最快 | 模板化内容质量低 |

**理由：** Workflow 是专为多子 agent 编排设计的工具，可以在后台并行运行，一个脚本控制全过程，token 开销可控。

### Decision 2: 20 批分组策略

每批 ~15 个知识点，按技术栈分组以保持主题连贯性。

```
┌────────┬──────────────────────┬──────┬────────┬─────────┐
│ 批号   │ 栈                   │ 数量 │ 强度分布 │ 目标篇幅 │
├────────┼──────────────────────┼──────┼────────┼─────────┤
│ 01-03  │ C 语言               │ 44   │ 弱~中   │ 200-500t │
│ 04-06  │ C++                  │ 42   │ 弱~中   │ 200-500t │
│ 07-10  │ DS 数据结构          │ 52   │ 中~强   │ 500-1000t│
│ 11-13  │ DB 数据库 (48点)     │ 48   │ 中~强   │ 300-500t │
│ 14     │ VDB 向量数据库       │ 15   │ 全强    │ 1000-1500│
│ 15-17  │ Python (含已有33篇)  │ 43   │ 中~强   │ 500-1000t│
│ 18-20  │ Linux (含已有56篇)   │ 56   │ 弱~中   │ 200-400t │
├────────┼──────────────────────┼──────┼────────┼─────────┤
│ 合计   │ 7 栈                 │ 300  │         │          │
└────────┴──────────────────────┴──────┴────────┴─────────┘
```

**已有文章的补充策略：**
- DB 和 Linux 栈的已有文章在生成批次中标记为 `supplement:true`
- 子 agent 会在生成时输出补充章节而非全文
- 集成阶段脚本负责追加以保持原文不变

### Decision 3: 每批子 agent 的 Prompt 结构

每批启动一个子 agent，收到以下结构化内容：

```
[系统指令]
你是一名深度学习技术写作专家。
为以下 N 个知识点各写一篇浅显易懂的 "该知识点在深度学习中的角色" 文章。
每条输出用 <!-- item: <id> --> 包裹以便脚本解析。

[上下文]
- 技术栈: <stack name>
- 这 N 个知识点的 items-registry 定义（id/title/desc/tags）
- 每个知识点的 DL 关联度（strong/medium/weak）
- 已有文章标记（如果有）

[范例]
一篇该栈已有的 learn-deep 文章（选取一篇强关联的）作为风格参考

[模板]
- Strong: 四节完整文章，含代码
- Medium: 两节简洁文章，选含代码
- Weak: 一段话

[输出要求]
- 每条内容开头必须是 <!-- item: <itemId> -->
- 内容之间用 --- 分隔
- 学术准确，不编造不存在的研究或框架
```

### Decision 4: 子 agent 的结构化输出格式

子 agent 不返回自由文本，而是返回可解析的结构化 markdown：

```markdown
<!-- item: c-dynamic-memory -->
# 动态内存管理与深度学习

## 一、与动态内存管理的关联

...

---

<!-- item: c-pointer -->
# 指针与深度学习

...

---

<!-- item: c-ds-basic -->
# 链表与深度学习

...
```

输出后由主 Workflow 脚本按 `<!-- item: ... -->` 分割解析。

### Decision 5: 集成阶段使用 Node.js 脚本而非子 agent

文件写入、追加、bundle 重建等纯工程操作由 Workflow 中的本地脚本（`scriptPhase`）完成，不消耗 LLM token。

```
集成脚本流程:
  1. 读取 Phase 2 的输出（JSON，key=itemId, value=markdown）
  2. 遍历所有 300 个 itemId：
     a. 如果已有 .md 文件 → 在文件末尾追加（先备份）
     b. 如果无 .md 文件 → 新建 data/learn-deep/<id>.md
  3. 读取 data/learn-deep/*.md 重建 LEARN_DEEP_CONTENT bundle
  4. 验证覆盖率（300/300）
  5. 输出集成报告
```

### Decision 6: 已有 93 篇文章的备份策略

修改现有文件前，先复制到 `data/learn-deep/.backup/` 目录。

### Decision 7: Bundle 重建方式

用脚本扫描 `data/learn-deep/` 下所有 `.md` 文件，拼接成：

```javascript
window.LEARN_DEEP_CONTENT = {
  "db-aries": `<原始内容>\n\n## 深度学习中的 ARIES\n\n...`,
  "c-pointer": `# 指针与深度学习\n\n...`,
  // ... 300 entries
};
```

覆盖写入 `data/app/learn-deep-bundle.js`。

## Risks / Trade-offs

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 子 agent 输出格式不统一，脚本解析失败 | 中 | 高 | 在 prompt 中明确要求 `<!-- item: -->` 标记，并加后处理校验 |
| 强关联知识点的 DL 内容不准确（编造研究） | 中 | 中 | 在 prompt 中加入"不编造不存在的研究"约束，抽样 10% 人工 review |
| 部分知识点与 DL 的关联太牵强 | 低 | 低 | Weak tier 只需一段话保证最低覆盖即可 |
| 追加操作破坏已有文章的格式 | 低 | 高 | 备份 + 追加前校验文件格式 |
| Bundle 文件过大（384K tokens 基础上增加） | 中 | 低 | 预计增加 ~50% 体积，纯静态无运行时影响 |
| 部分已有文章在子目录而非根目录 | 低 | 中 | 集成脚本需要处理 data/learn-deep/{db,linux,vdb}/ 和根目录两种路径；新的文章统一放根目录 |

## Open Questions

1. **DB/VDB 栈重叠问题**：部分 VDB 知识点（如 `db-ivf-variants`）已有很强的 DL 内容，是否需要写"第二篇"DL 文章，还是直接标注已有足够覆盖？
2. **Linux 栈的大量已有文章**：56 篇已覆盖多种主题，但其中哪些真的值得补 DL 关联？调度器、存储、网络各有不同的连接强度——是否值得在分类阶段更细致地区分？
3. **子 agent 的模型选择**：用 Sonnet 还是 Haiku？Sonnet 质量更高但更贵，Haiku 便宜但可能更模板化。建议强相关批次用 Sonnet，弱相关用 Haiku。
