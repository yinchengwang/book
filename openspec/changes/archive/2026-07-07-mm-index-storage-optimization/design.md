# 多模态数据库索引存储优化设计

## Context

### 背景

当前多模态数据库存储引擎的索引实现较为简单，主要问题包括：

1. **向量存储**：使用连续二进制文件，无分页管理；HNSW 索引不支持增量持久化；无量化压缩
2. **图存储**：使用 KV 存储，无专门的图存储优化；边遍历效率低
3. **时序存储**：无时间范围索引；聚合查询需要全表扫描
4. **空间存储**：R-Tree 仅在内存中，无法持久化
5. **文档存储**：使用线性扫描，查询性能差

### 目标产品

参考以下业界实现进行优化：
- **向量**：Milvus 2.x、Faiss、Pinecone
- **图**：Neo4j、JanusGraph、Age
- **时序**：TimescaleDB、InfluxDB、Gorilla TSDB
- **空间**：PostGIS、GeoMesa、S2 Geometry
- **文档**：Elasticsearch、MeiliSearch

## Goals / Non-Goals

**Goals:**
- 为每种索引类型实现业界标准的高效存储格式
- 支持增量持久化和崩溃恢复
- 减少存储空间（通过压缩）
- 提升查询性能（通过索引优化）
- 保持向后兼容（不破坏现有 API）

**Non-Goals:**
- 不实现分布式存储（单体架构优先）
- 不实现完整的 SQL 查询优化器
- 不实现多租户隔离
- 不实现完整的 ACID 事务（部分支持 WAL）

## Decisions

### 决策 1: 统一分页管理层

**选择**: 采用固定页面大小（4KB/8KB/16KB 可配置）的分页管理

**理由**:
- 与操作系统页面对齐，减少 I/O 碎片
- 支持 mmap 内存映射，直接映射文件到内存
- 便于实现 Buffer Pool（类 PostgreSQL 架构）
- 支持增量持久化（只写脏页）

**替代方案**:
- 变长页面：更灵活但实现复杂，不适合向量化存储
- 无页面管理：简单但无法支持 mmap 和增量持久化

### 决策 2: 向量量化压缩采用 PQ 而非 SQ

**选择**: Product Quantization (PQ) 作为主要量化方法

**理由**:
- PQ 可实现 10-50x 压缩比，远超 Scalar Quantization (SQ)
- 支持 OPQ (Optimized PQ) 进一步提升精度
- Faiss 生态成熟，易于集成
- 查询时使用 ADIS (Asymmetric Distance Computation) 加速

**替代方案**:
- SQ：压缩比低（4-8x），精度高但存储开销大
- Binary Quantization：压缩比高但精度损失大

### 决策 3: 图存储采用 CSR 格式

**选择**: CSR (Compressed Sparse Row) 作为边存储主格式

**理由**:
- O(1) 获取顶点的所有出边
- 内存连续，缓存友好
- 易于实现增量更新（CSR + COO 混合）
- 学术界和工业界广泛采用

**替代方案**:
- 邻接表：更新友好但查询效率低
- 邻接矩阵：空间复杂度 O(V²)，不适合大图

### 决策 4: 时序索引采用分段 + 稀疏索引

**选择**: 按时间范围分段，每段使用稀疏索引

**理由**:
- 时间范围查询可快速定位段
- 每段内部使用稀疏索引（如 min/max 时间戳）
- 支持 TTL 清理（删除整个段）
- 与 Gorilla 压缩块集成

**替代方案**:
- LSM-Tree：复杂，写放大严重
- B-Tree 按时间分区：实现简单但范围查询需跨多个 B-Tree

### 决策 5: 空间索引使用 Hilbert 曲线

**选择**: Hilbert 曲线作为空间填充曲线

**理由**:
- Hilbert 曲线保持空间局部性优于 Z-Order
- 减少 I/O 随机访问
- 支持多维 KNN 查询优化
- 与 R-Tree 结合使用效果最佳

**替代方案**:
- Z-Order (Morton Code)：实现简单但局部性差
- 四叉树/八叉树：实现复杂，不适合高维

### 决策 6: 文档索引采用倒排索引

**选择**: 倒排索引作为文档存储主索引

**理由**:
- 支持全文搜索，O(1) 词项查找
- 便于实现 BM25 打分
- 支持分词器插件化
- Elasticsearch 生态成熟

**替代方案**:
- B+Tree 按文档 ID 索引：查询效率低
- 线性扫描：无法支持全文搜索

## Data Structures

### 1. 向量分页存储 (VecPage)

```
┌────────────────────────────────────────┐
│           VectorPageHeader (64B)       │
├────────────────────────────────────────┤
│  page_id: int32_t                      │
│  vector_count: int32_t                 │
│  capacity: int32_t                     │
│  next_page: int32_t                    │
│  dirty: bool                           │
│  lsn: uint64_t (日志序列号)            │
├────────────────────────────────────────┤
│           Vector Data                  │
│  [vector_id, vector_data, ...]         │
│  每个向量: id (4B) + float[dim]        │
├────────────────────────────────────────┤
│           Page Footer (16B)            │
│  checksum: uint32_t                    │
│  magic: 0x5650 ("VP")                  │
└────────────────────────────────────────┘
```

### 2. 图 CSR 存储

```
Vertices File:
┌────────────────────────────────────────┐
│  vertex_count: uint64_t                │
│  label_count: uint32_t                 │
├────────────────────────────────────────┤
│  Vertex[0]: label_id, prop_offset      │
│  Vertex[1]: label_id, prop_offset      │
│  ...                                   │
└────────────────────────────────────────┘

Edges File:
┌────────────────────────────────────────┐
│  edge_count: uint64_t                  │
│  src_count: uint32_t (CSR 偏移数)      │
├────────────────────────────────────────┤
│  offsets[0..V]: 第 i 个顶点的出边起始   │
│  edges[0..E-1]: 目标顶点 ID + 边类型    │
│  edge_props: 边属性数据                 │
└────────────────────────────────────────┘
```

### 3. 时序分段索引

```
┌────────────────────────────────────────┐
│           Segment Index File           │
├────────────────────────────────────────┤
│  seg_count: uint32_t                   │
│  seg_size: uint32_t (点数/段)          │
├────────────────────────────────────────┤
│  SegmentMeta[0]: start_ts, end_ts,     │
│                  file_offset, count    │
│  SegmentMeta[1]: ...                   │
│  ...                                   │
└────────────────────────────────────────┘

┌────────────────────────────────────────┐
│           Data Segment File            │
├────────────────────────────────────────┤
│  Block[0]: Gorilla 压缩块              │
│  Block[1]: Gorilla 压缩块              │
│  ...                                   │
└────────────────────────────────────────┘
```

### 4. R-Tree 持久化格式

```
┌────────────────────────────────────────┐
│           RTreeFileHeader (64B)        │
├────────────────────────────────────────┤
│  magic: "RTRE"                         │
│  version: uint16_t                     │
│  root_offset: uint64_t                 │
│  node_count: uint32_t                  │
│  item_count: uint32_t                  │
├────────────────────────────────────────┤
│           Node Pages                   │
│  每页 4KB，包含多个节点                │
│  叶子节点: [id, bbox, data_offset]     │
│  内部节点: [child_ptr, bbox]           │
└────────────────────────────────────────┘
```

### 5. 文档倒排索引

```
┌────────────────────────────────────────┐
│           InvertedIndex File           │
├────────────────────────────────────────┤
│  Term Dictionary (FST)                 │
│  term -> term_id, posting_list_ptr     │
├────────────────────────────────────────┤
│  Posting Lists                         │
│  term_id -> [doc_id, freq, positions]  │
├────────────────────────────────────────┤
│  Document Store                        │
│  doc_id -> doc_data (JSON)             │
└────────────────────────────────────────┘
```

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| PQ 量化精度损失 | 查询结果不精确 | 支持可配置压缩比；提供非量化搜索作为备选 |
| CSR 增量更新开销 | 频繁小更新导致重排 | 使用 CSR+COO 混合；定期合并 |
| 分页管理内存开销 | 页面元数据占用内存 | 限制 Buffer Pool 大小；实现 LRU 淘汰 |
| R-Tree 持久化复杂度 | 实现难度大 | 参考 PostgreSQL GiST 格式；分阶段实现 |
| 倒排索引构建耗时 | 大批量导入慢 | 支持后台增量构建；提供批量导入工具 |

## Migration Plan

### Phase 1: 向量存储优化 (优先级: 高)
1. 实现 VecPage 分页管理层
2. 添加 HNSW 索引增量持久化
3. 集成 PQ 量化压缩
4. 迁移现有向量数据到新格式

### Phase 2: 图存储优化 (优先级: 中)
1. 实现 GraphCSR 存储格式
2. 添加顶点标签索引
3. 实现 Hilbert 曲线索引
4. 迁移现有图数据到新格式

### Phase 3: 时序存储优化 (优先级: 中)
1. 实现 TsSegment 分段索引
2. 优化 Gorilla 压缩算法
3. 实现物化视图
4. 迁移现有时序数据

### Phase 4: 空间存储优化 (优先级: 低)
1. 实现 RTreeFile 持久化格式
2. 添加 Hilbert 曲线优化
3. 扩展 OGC 几何支持

### Phase 5: 文档存储优化 (优先级: 低)
1. 实现 DocInvertedIndex 倒排索引
2. 集成 BM25 打分
3. 支持分词器插件

### 回滚策略
- 每个阶段保留旧格式兼容层
- 数据迁移前完整备份
- 支持从备份恢复

## Open Questions

1. **页面大小选择**: 4KB/8KB/16KB 哪个最优？需要基准测试
2. **PQ 量化参数**: M=48, nbits=8 是否适合所有场景？
3. **CSR 合并策略**: 何时触发 CSR+COO 合并？
4. **Hilbert 曲线阶数**: 如何动态选择？
5. **文档分词器**: 是否需要支持多种分词器？
