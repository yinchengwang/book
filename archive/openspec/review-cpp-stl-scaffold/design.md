## Context

`mystl::vector` 通过模板参数 `Alloc` 持有分配器实例，构造元素时已经使用 `alloc_.construct`，但 `pop_back` 当前直接调用元素析构函数。这样虽然默认分配器下通常能销毁正确对象，却绕过了自定义分配器的 `destroy`，导致分配器生命周期钩子不可观测且语义不一致。工作区中的 `vector.h` 还包含其他未提交修改，因此本变更必须精确编辑和暂存。

## Goals / Non-Goals

**Goals:**
- 非空 vector 调用 `pop_back` 后，先将 `finish_` 指向原尾元素，再通过 `alloc_.destroy(finish_)` 销毁该元素。
- 用自定义计数分配器验证销毁调用、销毁地址和单元素边界。
- 保持公开 API 与现有代码风格不变。

**Non-Goals:**
- 不定义空 vector 调用 `pop_back` 的行为；其仍与 `std::vector` 一样属于未定义行为。
- 不重构其他销毁路径或修改范围插入等无关代码。
- 不引入新的测试框架或依赖。

## Decisions

1. **使用自定义计数分配器进行黑盒回归测试。** 该分配器继承现有 `mystl::allocator<T>`，重写 `destroy` 并记录调用次数与地址。相比只检查 `size()` 或元素析构计数，这能直接证明容器调用了分配器接口，并验证地址是 `data() + old_size - 1`。
2. **创建独立的 `vector_pop_back_test.cpp`。** 该测试文件仅依赖标准输出与真实 `mystl::vector`，不依赖当前工作区里尚未跟踪的新测试框架，并可单独编译运行、清晰展示 RED/GREEN 结果。
3. **生产代码只替换销毁表达式。** 保留 `--finish_` 的顺序，将 `finish_->~T()` 改为 `alloc_.destroy(finish_)`，满足最小修复原则。
4. **使用补丁级精确暂存。** `vector.h` 现有其他未提交修改不属于本任务，提交时仅暂存 `pop_back` 对应补丁块。

## Risks / Trade-offs

- [自定义分配器中的静态观测状态可能跨测试残留] → 每个测试场景开始前显式重置状态。
- [基础类型 `int` 的析构难以体现错误] → 通过分配器 `destroy` 钩子记录调用及地址，而不是依赖类型析构副作用。
- [工作区并发改动可能混入提交] → 提交前检查 staged diff，并只按路径与补丁暂存任务相关内容。
