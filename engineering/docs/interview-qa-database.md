# 数据库存储引擎面试问答库

## 4.1 核心问题库

### 4.1.1 Buffer Pool 相关问题（12 题）

**Q1: Buffer Pool 是什么？为什么需要它？**

Buffer Pool 是数据库内存中的页面缓存，用于弥合内存和磁盘之间 100,000 倍的性能差距。它将热点数据保留在内存中，避免每次访问都读磁盘。没有它，数据库性能会下降 4-5 个数量级。

**Q2: Clock-Sweep 算法如何工作？**

Clock-Sweep 维护一个环形数组，每个槽位有一个 `usage_count` 计数器。置换时从当前指针（clock hand）开始扫描：如果 `usage_count > 0`，递减并跳过（给页面"二次机会"）；如果 `usage_count == 0` 且页面未被 pin，选择此页面置换。这样近似了 LRU 的行为但实现更简单。

**Q3: 为什么用 Clock 而不是 LRU？**

LRU 需要维护全局双向链表，每次访问都要更新链表（高并发下锁竞争严重），而 Clock 只需要一个原子递增的指针。此外，LRU 容易被顺序扫描"污染"——扫描页面全部移到链表头，挤走真正的热点数据。Clock 中扫描页面只有 `usage_count=1`，很快就被置换出去。

**Q4: 脏页是什么？如何管理？**

脏页（dirty page）是内存中被修改但尚未写入磁盘的页面。管理策略包括：
- **脏页标记**：修改时设置 `dirty` 标志
- **检查点刷盘**：定期将所有脏页写入磁盘
- **bgwriter 后台刷盘**：持续、平滑地刷脏页
- **LSN 追踪**：记录每个脏页最后修改的 LSN

**Q5: Pin 和 Unpin 的作用是什么？**

Pin（固定）通过增加 `refcount` 防止页面在使用中被置换。这是 Buffer Pool 并发控制的基础：只要页面被 Pin，置换算法就不会选它。Unpin 减少 `refcount`，当 `refcount=0` 时页面可以被置换。

**Q6: 如何提高 Buffer Pool 命中率？**

- 增大 `shared_buffers`（通常设为 25% RAM）
- 使用预取（prefetch）提前加载需要的页面
- 优化查询避免全表扫描
- 合理设置 `effective_cache_size`
- 监控命中率，目标是 > 95%

**Q7: 全表扫描对 Buffer Pool 有什么影响？如何避免？**

全表扫描会读取大量页面，可能将缓存中的热点数据全部挤出。缓解方法：
- 使用"环形缓冲区"（ring buffer）隔离扫描页面
- 扫描页面使用较低的 `usage_count`
- PostgreSQL 的 `buffer ring` 策略

**Q8: 什么是 Buffer 的 IO 等待？**

当两个线程需要同一个页面，但该页面正在从磁盘加载时（`io_in_progress = true`），后来的线程需要在条件变量上等待 IO 完成，而不是发起重复的 IO 请求。这减少了不必要的磁盘 IO。

**Q9: 如何监控 Buffer Pool 的性能？**

关键指标：
- **命中率**：`hits / (hits + misses)`，健康值 > 95%
- **脏页比例**：`dirty_bufs / total_bufs`，通常 10-30%
- **置换频率**：单位时间的 eviction 次数
- **等待事件**：IO 等待比例

**Q10: 新型存储（NVMe、PMEM）对 Buffer Pool 设计有什么影响？**

NVMe 缩小了随机和顺序 IO 的差距，减少了对预取和大页面缓存的依赖。持久化内存（PMEM）延迟接近 DRAM 且掉电不丢失，可能改变 Buffer Pool 的设计假设——不再需要区分"内存"和"持久化存储"。

**Q11: Buffer Pool 如何处理 NUMA 架构？**

在多插槽服务器上，本地内存比远端内存快得多。解决方案：
- 每个 NUMA 节点维护独立的 Buffer Pool 分区
- 尽量将页面分配到访问它的 CPU 所在的内存节点
- PostgreSQL 的 `numa_support` 配置

**Q12: 检查点对 Buffer Pool 的影响是什么？**

检查点触发所有脏页刷盘，造成 IO 风暴。影响包括：前台事务延迟增加（等待 IO）、Buffer Pool 压力增大（页面被刷走后需要重新读取）。缓解方法：增量检查点、`checkpoint_completion_target` 参数。

### 4.1.2 BTree 相关问题（16 题）

**Q13: BTree 为什么适合作为数据库索引？**

BTree 的节点大小设计为等于磁盘页面大小，每个节点可存放大量键（高扇出），因此树高很小（通常 3-4 层）。这意味着一次点查询只需要 1-2 次磁盘读取（根节点通常缓存在内存）。叶子节点有序排列支持高效的范围扫描。支持 O(log n) 的插入和删除。

**Q14: BTree 和 B+Tree 的区别？**

| | BTree | B+Tree |
|---|-------|--------|
| 数据存放 | 内部节点也存放数据 | 只在叶子节点存放数据 |
| 范围查询 | 需要中序遍历 | 叶子节点有链表，顺序遍历 |
| 内部节点大小 | 更小（存数据） | 更大（只存键），扇出更高 |
| 典型使用 | 文件系统（NTFS） | 数据库（MySQL InnoDB） |

**Q15: 节点分裂的过程是什么？**

当节点插入后超过容量：
1. 计算分裂点（通常 n/2）
2. 后半部分键复制到新节点
3. 中间键上推到父节点
4. 更新兄弟指针
5. 如果父节点也满了，递归分裂（最坏到根）

**Q16: 什么时候会发生节点合并？为什么需要合并？**

当节点删除后条目数低于最小阈值（通常 n/2）时触发。合并的原因：
- 保持空间利用率（不低于 50%）
- 维持树平衡（树高不变）
- 但 PostgreSQL 实际上很少主动合并（删除空间留待后续插入复用）

**Q17: 什么是 B-Link Tree？**

B-Link Tree 是 BTree 的并发优化：每个节点增加一个指向右兄弟的指针。这样即使读操作赶上了正在分裂的节点，也能通过兄弟指针找到正确的键。这使得读操作不需要锁（乐观并发控制）。

**Q18: BTree 的范围扫描如何实现？**

1. 从根向下查找到范围的起始键
2. 在叶子节点找到 >= 起始键的第一个条目
3. 顺序遍历叶子节点，直到键 > 结束键
4. 利用叶子节点的兄弟指针跳转到下一页

时间复杂度：O(log n + k)，k 是返回的条目数。

**Q19: 为什么 BTree 不适合作为时序数据库的默认索引？**

时序数据特点是追加写（时间戳递增），BTree 的插入会导致叶子节点的频繁分裂，产生随机 IO。而 LSM Tree 的追加写模式更适合这种工作负载。

**Q20: 索引分裂对并发有什么影响？**

节点分裂需要持有该节点及其父节点的排他锁。在分裂期间，该子树上的所有操作都会被阻塞。因此频繁分裂会显著降低并发度。B-Link Tree 通过乐观并发控制缓解了这一问题。

**Q21: 复合索引在 BTree 中如何存储和查找？**

复合索引将多列键拼接为单个键：
```
CREATE INDEX idx ON t(a, b, c);
键 = (a, b, c)
```
查找时按列序比较：先比较 a，相同则比较 b，以此类推。这意味着索引 (a, b, c) 可以加速 `WHERE a = 1` 但不能加速 `WHERE b = 2`（最左前缀原则）。

**Q22: 为什么唯一索引的键不能重复？如何保证？**

BTree 唯一索引在插入时先查找键是否存在。如果存在且不是当前事务自己插入的，返回唯一约束错误。并发情况下，通过页面级锁保证检查和插入的原子性。

**Q23: PostgreSQL 的 GIN 和 GiST 索引是什么？它们与 BTree 的关系？**

- **BTree**：适用于可排序的标量键（`=`，`<`，`>`）
- **GIN**（Generalized Inverted Index）：适用于复合值的包含查询（数组、全文搜索）
- **GiST**（Generalized Search Tree）：适用于几何、空间数据类型

它们都是"搜索树"的不同实现，BTree 是最基础和广泛使用的。

**Q24: 什么情况下索引不被使用？**

常见原因：
- 查询条件使用了函数：`WHERE LOWER(name) = 'alice'`
- 隐式类型转换：`WHERE id = '123'`（id 是 int）
- 选择性太低：优化器认为全表扫描更快
- 统计信息过期
- 复合索引不满足最左前缀原则

**Q25: BTree 的持久化如何保证一致性？**

1. 修改前先写 WAL（记录分裂/合并的操作）
2. WAL 写入成功后修改内存中的页面
3. 标记页面为脏
4. 检查点或后台刷盘将页面写回磁盘
5. 崩溃恢复时通过 WAL 重放未完成的页面修改

**Q26: 如何优化 BTree 索引的性能？**

- 选择合适的填充因子（fillfactor），留出空间给后续插入
- 定期 REINDEX 减少碎片
- 使用 BRIN 索引替代 BTree（对于大型有序表）
- 分区表减少单个索引的大小
- 避免在高频更新的列上建立过多索引

**Q27: BTree 索引的"最左前缀原则"是什么？**

对于复合索引 `(a, b, c)`：
- ✅ `WHERE a = 1` — 使用索引
- ✅ `WHERE a = 1 AND b = 2` — 使用索引
- ✅ `WHERE a = 1 AND b = 2 AND c = 3` — 使用索引
- ❌ `WHERE b = 2` — 不使用索引（跳过了 a）
- ❌ `WHERE c = 3` — 不使用索引

**Q28: 为什么 MySQL InnoDB 用 B+Tree 而不用 BTree？**

B+Tree 将数据只存放在叶子节点，好处：
- 内部节点只存放键，扇出更高，树更矮
- 叶子节点有链表，范围扫描非常高效
- 聚簇索引的叶子节点直接存放行数据

### 4.1.3 MVCC 相关问题（12 题）

**Q29: MVCC 的核心思想是什么？用一句话解释。**

MVCC 通过为数据维护多个版本，使得读操作不需要等待写操作完成，从而实现读写不阻塞的并发控制。

**Q30: 快照隔离（Snapshot Isolation）是什么？**

每个事务在开始时获取一个数据库快照，事务期间只能看到快照创建时已提交的数据。快照由 `xmin`（最小活跃事务）、`xmax`（最大事务 + 1）和 `xip_list`（活跃事务列表）组成。

**Q31: 如何判断一个元组版本是否可见？**

五规则：
1. 自己创建的 → 可见
2. xmax 已提交 → 不可见（被删除）
3. xmin 未提交 → 不可见
4. xmin 在活跃列表中 → 不可见
5. xmin < xmax（快照）且不在活跃列表 → 可见

**Q32: MVCC 如何实现事务回滚？**

MySQL 通过 Undo Log 中保存的旧值恢复到之前的状态。PostgreSQL 只需要将 xmax 标记为回滚事务即可（标记为 Never Committed），无需物理恢复数据——这是 PostgreSQL 回滚 O(1) 的原因。

**Q33: 读已提交和可重复读在 MVCC 中有什么区别？**

- 读已提交：每次语句执行前重新获取快照，每次能看到新提交的数据
- 可重复读：事务开始时获取一次快照，整个事务期间不变，保证同一事务内多次读取结果一致

**Q34: 什么是写偏斜（Write Skew）？MVCC 如何处理？**

写偏斜是快照隔离级别下的序列化异常：两个事务读取不同的行但写入了基于读的结果，导致约束违反。PostgreSQL 的 SSI（Serializable Snapshot Isolation）通过 SIREAD 锁检测这种冲突并中止其中一个事务。

**Q35: MVCC 如何造成表膨胀？怎么解决？**

UPDATE/DELETE 创建新版本，旧版本成为"死元组"（dead tuple）。如果不及时清理，表会膨胀。解决方案：
- PostgreSQL：VACUUM（自动或手动）
- MySQL：后台 Purge 线程
- 避免长事务（阻止垃圾回收）

**Q36: MySQL 和 PostgreSQL 的 MVCC 实现有什么根本区别？**

MySQL 将旧版本存储在独立的 Undo 段中（通过 ROLL_PTR 链接），PostgreSQL 将新旧版本存放在同一个 Heap 页面中（通过 xmin/xmax 区分）。MySQL 的方式更工程化（垃圾回收不碰 Heap），PostgreSQL 的方式更自然但需要 VACUUM。

**Q37: 为什么说 VACUUM 是 PostgreSQL 的阿喀琉斯之踵？**

VACUUM 清理死元组时需要扫描整个表，是 IO 密集型操作。如果 autovacuum 跟不上写入速率，表会持续膨胀，查询越来越慢，最终需要手动 VACUUM FULL（锁表重建）。长事务更会阻止 VACUUM 的进行。

**Q38: MVCC 对索引有什么影响？**

索引条目也遵循 MVCC — UPDATE 在索引中创建新条目，旧条目保留。这导致：
- 索引膨胀（`btree_page_items` 可见许多死条目）
- 唯一索引检查需要在 MVCC 上下文中进行
- 索引扫描需要过滤不可见的条目

**Q39: 事务 ID 回卷（Wraparound）是什么问题？如何解决？**

事务 ID 是 32 位整数，用完后会回卷到 0。此时旧事务看起来像未来的事务，造成数据丢失。解决方案：
- 冻结（Freeze）：将老元组的 xmin 标记为 FrozenTransactionId（永远可见）
- 定期 VACUUM FREEZE
- PostgreSQL 10+ 使用 64 位事务 ID

**Q40: 如何在 MVCC 中实现 SELECT ... FOR UPDATE？**

`SELECT ... FOR UPDATE` 不仅要读取可见版本，还要在最新版本上加排他锁。这涉及：
1. 找到最新版本（不管是否可见）
2. 加行锁
3. 等待未提交的修改
4. 返回最新已提交的版本

### 4.1.4 WAL 相关问题（10 题）

**Q41: WAL 的"先写日志"原则是什么？为什么必须遵守？**

WAL 要求在修改数据页之前，先将修改操作写入日志并确保日志已持久化（fsync）。这保证了即使数据页在写入磁盘前崩溃，也能通过重放 WAL 恢复数据。

**Q42: LSN 是什么？在 WAL 中扮演什么角色？**

LSN（Log Sequence Number）是 WAL 中每个记录的全局唯一序号，通常是日志文件号 + 文件内偏移的组合。作用：
- 精确定位每个 WAL 记录
- 判断页面是否需要 Redo（页面 LSN vs 记录 LSN）
- 确定恢复起点

**Q43: ARIES 恢复算法的三个阶段是什么？**

1. **分析（Analysis）**：从最后检查点扫描 WAL，重建脏页表和活跃事务表
2. **重做（Redo）**：从脏页表中最小的 recLSN 开始，按 LSN 顺序重放所有操作
3. **回滚（Undo）**：逆序回滚未提交事务，写入 CLR（Compensation Log Record）

**Q44: 检查点的作用是什么？检查点间隔应该设多少？**

检查点将所有脏页刷盘并记录一个恢复起点。作用：
- 缩短恢复时间（不需要从头重放）
- 截断旧 WAL 文件
- 间隔取决于恢复时间目标（RTO）和 IO 预算

**Q45: 为什么 fsync 对性能影响这么大？**

fsync 强制将数据从操作系统缓存写入磁盘物理介质。一次 fsync 延迟约 1-10ms。如果每次事务提交都 fsync，TPS 上限就是 100-1000。而批量 fsync 或异步提交可以将 TPS 提高 10-100 倍。

**Q46: 组提交（Group Commit）是什么？如何实现？**

组提交将多个事务的提交合并为一次 fsync：
1. 事务提交时将 WAL 写入并进入等待队列
2. 第一个事务触发 fsync
3. fsync 完成后唤醒队列中所有事务
4. 这样 10 个事务的提交只需一次 fsync

**Q47: 逻辑复制和物理复制有什么区别？**

- 物理复制：复制 WAL 记录的字节流，备库与主库逐字节一致，必须同架构同版本
- 逻辑复制：复制 SQL 操作，可以跨版本、跨架构，只复制特定表

**Q48: WAL 文件太大怎么办？**

- 增大检查点频率（缩减 WAL 保留时间）
- 启用 WAL 压缩
- 增加 `wal_keep_segments` 避免不必要的保留
- 使用 `pg_archivecleanup` 清理归档
- 监控复制槽（replication slot）不用的 WAL

**Q49: 崩溃恢复后，数据库如何保证一致性？**

通过 ARIES 恢复：
1. Redo 所有已提交事务（确保持久性）
2. Undo 所有未提交事务（确保原子性）
3. CLR 保证即使恢复过程中再次崩溃也能恢复

**Q50: WAL 的物理日志和逻辑日志各有什么优缺点？**

- 物理日志：精确恢复页面状态，实现简单，但不跨平台、记录大
- 逻辑日志：记录 SQL 操作，跨平台可读，但恢复需要重新执行（可能有非确定性）

### 4.1.5 索引相关问题（12 题）

**Q51: 聚簇索引和非聚簇索引的区别？**

| | 聚簇索引 | 非聚簇索引 |
|---|---------|-----------|
| 数据存放 | 叶子节点存放行数据 | 叶子节点存放行指针 |
| 数量 | 每表只有 1 个 | 可以多个 |
| 回表 | 不需要 | 可能需要回表 |
| 示例 | MySQL InnoDB 主键索引 | PostgreSQL 所有索引 |

**Q52: 覆盖索引是什么？有什么好处？**

覆盖索引是指查询所需的所有列都在索引中，不需要回表访问数据页。好处：减少 IO（不需要额外的随机读）、减少 Buffer Pool 污染。

**Q53: Hash 索引和 BTree 索引的区别？**

| | Hash 索引 | BTree 索引 |
|---|----------|-----------|
| 查找 | O(1) | O(log n) |
| 范围查询 | ❌ 不支持 | ✅ 支持 |
| 有序输出 | ❌ | ✅ |
| 并发 | 较好 | 一般 |
| 使用场景 | 等值查询（KV 存储） | OLTP 通用查询 |

**Q54: 什么是索引碎片？如何整理？**

索引碎片是页面中因频繁插入/删除造成的空间利用率低下。表现：页面半满、逻辑顺序与物理顺序不一致。整理方法：
- PostgreSQL：`REINDEX INDEX idx_name`
- MySQL：`OPTIMIZE TABLE t`
- 或定期重建索引

**Q55: 倒排索引（Inverted Index）的原理？**

倒排索引为每个词（term）维护一个包含该词的文档列表：
```
"数据库" → [doc1, doc3, doc7, doc15]
"BTree"  → [doc2, doc3, doc9]
```
用于全文搜索，查询时取交集/并集。PostgreSQL 的 GIN 索引就是倒排索引的实现。

**Q56: 为什么说"不要在索引列上使用函数"？**

因为 `WHERE LOWER(name) = 'alice'` 中，`LOWER(name)` 是一个表达式树，优化器无法将其映射到 `name` 列的索引。除非创建表达式索引：`CREATE INDEX idx ON t (LOWER(name))`。

**Q57: 什么是索引选择率（Selectivity）？它如何影响查询计划？**

选择率 = 满足条件的行数 / 总行数。选择率越低（越有选择性），索引越有用。典型规则：
- 选择率 < 5%：使用索引
- 选择率 5-30%：可能使用 Bitmap 扫描
- 选择率 > 30%：全表扫描

**Q58: Bitmap 索引扫描是什么？如何使用？**

Bitmap 扫描分为两步：
1. 先通过索引找到所有满足条件的行的物理位置，构建 Bitmap
2. 再按物理顺序读取页面（减少随机 IO）

适用于选择率中等的情况，在多个索引条件下效率尤其高。

**Q59: 索引下推（Index Condition Pushdown）是什么？**

将 WHERE 条件的一部分推到索引扫描层执行，在读取数据行之前就过滤掉不符合条件的行。这减少了回表次数和不必要的行读取。MySQL 5.6+ 和 PostgreSQL 都支持。

**Q60: 前缀索引（Prefix Index）的使用场景和限制？**

适用于长字符串列（如 URL、TEXT），只索引前 N 个字符。优点：节省空间、提高缓存效率。限制：不支持 ORDER BY、GROUP BY；降低选择性。

**Q61: 空间索引（Spatial Index）的原理？**

空间索引使用 R-Tree 或 KD-Tree 存储空间对象的 MBR（最小外接矩形），支持"附近"、"包含"、"相交"等空间查询。PostgreSQL 使用 GiST 索引实现。适用于 GIS 数据。

**Q62: 什么是索引合并（Index Merge）？**

当一个查询的 WHERE 条件涉及多个索引列时，分别使用不同的索引扫描，然后合并结果（交集/并集）。MySQL 的 Index Merge 优化需要谨慎评估——可能不如全表扫描快。

## 4.2 高级问题库

### 4.2.1 分布式事务相关问题（5 题）

**Q63: 分布式事务如何保证 ACID？**

通过两阶段提交（2PC）：
1. 准备阶段：协调者询问所有参与者是否可以提交
2. 提交阶段：所有参与者回复 YES 后，协调者发送 COMMIT

2PC 的问题：协调者故障导致资源锁定、参与者故障需要恢复机制。

**Q64: 两阶段提交（2PC）有什么问题？怎么优化？**

问题：
- 阻塞：协调者故障时参与者无限等待
- 性能：需要 2 轮网络往返
- 单点故障：协调者宕机

优化：
- 三阶段提交（3PC）：增加超时机制
- Percolator：使用分布式锁表 + 异步提交
- Saga 模式：最终一致性

**Q65: 分布式事务中的"脑裂"问题如何解决？**

Fencing Token 机制：给每个领导者分配单调递增的 token，存储系统拒绝来自旧 token 的写入请求。Raft 用 term 号实现这一点。

**Q66: 解释 TCC（Try-Confirm-Cancel）模式？**

TCC 是柔性的分布式事务模式：
- **Try**：预留资源（锁定库存）
- **Confirm**：使用资源（扣减库存）
- **Cancel**：释放资源（解除锁定）

相比 2PC，TCC 无阻塞，但业务代码需要实现 Try/Confirm/Cancel 三个接口。

**Q67: 跨分片查询的事务隔离如何保证？**

- 快照隔离 + 全局事务 ID
- 读取时获取各个分片的快照并合并
- 写入时使用 2PC 或 Percolator
- CockroachDB 的并行提交（Parallel Commit）减少延迟

### 4.2.2 一致性协议相关问题（5 题）

**Q68: Raft 协议的基本角色和流程？**

三个角色：Leader、Follower、Candidate。

流程：
1. Leader 接收客户端请求
2. Leader 将日志条目复制到所有 Follower
3. 多数派确认后 Leader 提交
4. Leader 通知 Follower 提交

关键保证：选举安全、日志匹配、Leader 完整性。

**Q69: Raft 和 Paxos 的本质区别？**

Raft 是为了易理解性设计的，Paxos 更早但更难理解。Raft 通过强 Leader、日志匹配特性简化了状态空间。两者在正确性上等价。

**Q70: 什么是 Quorum？为什么需要多数派？**

Quorum（法定人数）是 N/2+1 个节点。任何两个多数派必定有交集，因此通过多数派确认可以保证一致性。在 Leader 故障后，至少有一个节点有最新的日志，保证新 Leader 不会丢失数据。

**Q71: 脑裂时如何保证一致性？**

Raft 的 term 机制：每个选举周期递增 term 号。如果旧 Leader 在网络分区中仍存活，当它收到更高 term 的请求时，会退位为 Follower。存储层通过 Fencing Token 拒绝旧 Leader 的写入。

**Q72: 最终一致性和强一致性的区别？什么场景用哪个？**

- 强一致性：所有节点看到的数据始终一致（如银行转账）
- 最终一致性：数据最终会一致，但可能短暂不一致（如社交媒体的点赞数）

选择：金融场景必须强一致，社交/推荐场景可以接受最终一致。

### 4.2.3 性能优化相关问题（5 题）

**Q73: 如何诊断数据库的 IO 瓶颈？**

1. 查看等待事件：`pg_stat_activity` 中 `wait_event='DataFileRead'`
2. 查看 Buffer 命中率：低命中率意味着大量磁盘读取
3. 查看 IOPS：`iostat` 看磁盘利用率
4. 查看慢查询日志：`log_min_duration_statement`
5. 检查 VACUUM 是否在运行

**Q74: 如何优化数据库的写入性能？**

- 批量写入取代逐条 INSERT
- 禁用不必要的索引（加载完成后再创建）
- 增大 WAL Buffer
- 调整检查点参数
- 使用 UNLOGGED 表（不需要 WAL 的临时数据）
- 合理的 fillfactor（减少页分裂）

**Q75: 为什么有时全表扫描比索引扫描快？**

当需要读取表的大部分行（如 >30%）时：
- 索引扫描 → 随机 IO（跳来跳去读取数据页）
- 全表扫描 → 顺序 IO（连续读取页面）

顺序 IO 比随机 IO 快 10-100 倍，因此对于大结果集的查询，全表扫描可能更快。

**Q76: 如何分析和优化慢查询？**

步骤：
1. `EXPLAIN ANALYZE` 查看执行计划
2. 检查是否使用索引（Seq Scan vs Index Scan）
3. 检查统计信息是否过期（`ANALYZE`）
4. 检查 Nested Loop → 可能需要 Hash Join
5. 检查是否有不必要的排序或物化
6. 考虑查询重写

**Q77: 数据库连接池的大小如何确定？**

公式：`pool_size = (core_count * 2) + effective_spindle_count`

但实际取决于工作负载：
- CPU 密集型：core_count 或略多
- IO 密集型：可以更多（等待 IO 时不占 CPU）
- 活动连接数超过 core_count * 2-3 通常没有额外收益

## 4.3 场景设计问题

### 4.3.1 如何设计一个数据库（8 题）

**Q78: 如果让你从零设计一个数据库，你会怎么开始？**

1. 明确设计目标（OLTP？OLAP？时序？）
2. 设计存储引擎（行存储 vs 列存储）
3. 选择索引结构（BTree / LSM / Hash）
4. 实现事务支持（ACID → MVCC + WAL）
5. 设计查询执行器（解析 → 优化 → 执行）
6. 考虑并发控制（锁 / MVCC / 乐观并发）
7. 扩展性设计（单机 → 分布式）

**Q79: 存储引擎的架构层次是什么？**

```
┌──────────────────────────────┐
│       Access Method 层       │  ← 关系/元组抽象
├──────────────────────────────┤
│       Buffer Pool 层         │  ← 页面缓存 + 置换
├──────────────────────────────┤
│       WAL 层                 │  ← 日志 + 恢复
├──────────────────────────────┤
│       磁盘管理 层             │  ← 文件 + 页面 IO
└──────────────────────────────┘
```

**Q80: 设计 KV 存储需要考虑什么？**

- 键的排序（是否支持范围查询）
- 数据结构（Hash 表、跳表、BTree）
- 持久化（WAL + 快照）
- 并发（读写锁粒度）
- 压缩（键值压缩）

**Q81: 如何设计一个支持事务的存储引擎？**

核心组件：
1. **MVCC**：多版本并发控制
2. **WAL**：写前日志，保证原子性和持久性
3. **锁管理器**：行级锁 + 死锁检测
4. **检查点**：定期保存状态
5. **恢复**：ARIES 三阶段恢复

**Q82: 设计时序数据库有什么特殊考虑？**

- 写入模式：追加写为主（LSM Tree 优于 BTree）
- 数据压缩：时序数据模式固定，压缩比高
- 自动降采样（Rollup）：旧数据降低精度
- 自动过期（TTL）：按时间分区，自动删除
- 查询模式：时间范围 + 聚合函数

**Q83: 如何设计一个分布式 KV 存储？**

- 数据分片：一致性哈希 / Range Partition
- 复制：Raft / Multi-Paxos
- 路由：元数据服务 / client-side routing
- 共识：Leader 选举 + 日志复制
- 事务：Percolator / 2PC

**Q84: 从单机到分布式数据库，最大的挑战是什么？**

1. **网络延迟**：远程调用从微秒变成毫秒
2. **部分故障**：网络分区、节点宕机成为常态
3. **一致性**：CAP 定理限制了设计选择
4. **事务**：跨节点 ACID 变得复杂
5. **查询优化**：分布式 JOIN 和聚合需要新算法

**Q85: 你设计的数据库如何支持 SQL？**

1. **解析器**：词法分析 + 语法分析生成 AST
2. **语义分析**：类型检查、权限检查
3. **查询优化**：代价估算 + 规则重写
4. **执行器**：火山模型 / 向量化执行
5. **存储适配层**：将 SQL 操作映射到存储引擎 API

### 4.3.2 如何优化查询性能（5 题）

**Q86: 一个慢查询 `SELECT * FROM orders WHERE user_id = 123 ORDER BY created_at DESC LIMIT 10`，如何优化？**

1. 创建复合索引：`CREATE INDEX idx ON orders (user_id, created_at DESC)`
2. 只选择需要的列而非 `SELECT *`
3. 检查索引是否被使用：`EXPLAIN ANALYZE`
4. 如果表很大，考虑按 `user_id` 分区
5. 检查统计信息是否最新

**Q87: 数据库热点的解决方案？**

- 反向索引：反转热点键，分散到不同页面
- 分区：按时间或其他维度分区
- 读写分离：读请求分到只读副本
- 缓存：Redis/应用层缓存热点数据
- 批量处理：合并热点更新

**Q88: 如何设计数据库的分库分表方案？**

1. 选择分片键（Sharding Key）：访问最频繁的条件
2. 选择分片策略：Hash（均匀）、Range（有序）、List（业务分组）
3. 处理跨分片问题：查询路由、分布式 JOIN、全局唯一 ID
4. 动态扩容：一致性哈希 + 双写迁移

**Q89: 如何平衡索引数量和写入性能？**

每增加一个索引，INSERT/UPDATE 需要更新额外的 BTree。策略：
- 只创建查询需要的索引
- 监控未使用的索引（`pg_stat_user_indexes`）
- 对写入密集型表减少索引数量
- 用延迟索引创建（加载数据后再建索引）

**Q90: 数据库扩容有哪些策略？如何处理扩容期间的数据一致性？**

策略：
1. 垂直扩容（Scale Up）：加内存、加 CPU
2. 水平扩容（Scale Out）：分片到更多节点

水平扩容方案：
- 一致性哈希 + 虚拟节点
- 双写迁移：同时写新旧分片，逐步切换
- 在线重分片（Resharding）：批量拷贝 + 追赶写入

### 4.3.3 如何处理高并发（5 题）

**Q91: 10000 QPS 的写入场景，存储引擎如何应对？**

1. 批量写入：每批 100-1000 条
2. WAL 组提交：减少 fsync 次数
3. 增大 Buffer Pool：减少磁盘 IO
4. 预分配页面：避免分配页面的开销
5. 无锁数据结构：减少锁竞争
6. 分区表：将写入分布到多个数据文件

**Q92: 高并发下如何避免死锁？**

- 按固定顺序获取锁（如按行 ID 排序）
- 设置锁超时（`lock_timeout`）
- 使用死锁检测 + 自动回滚（`deadlock_timeout`）
- 尽量使用短事务
- 避免在事务中等待用户输入

**Q93: 秒杀场景下数据库如何扛住高并发？**

1. 缓存预热：将库存数量放入 Redis
2. 异步扣减：先扣 Redis，异步同步到 DB
3. 限流：令牌桶/漏斗算法控制并发
4. 排队：使用消息队列削峰
5. 数据库乐观锁：`UPDATE ... WHERE stock > 0`

**Q94: 乐观锁和悲观锁的选择？**

- 乐观锁（CAS）：并发冲突少时高效（库存充足时），冲突多时重试开销大
- 悲观锁（SELECT FOR UPDATE）：冲突多时可靠，但降低并发度

选择：低冲突场景用乐观锁，高冲突场景用悲观锁 + 排队。

**Q95: 如何设计一个支持千万并发的数据库系统？**

这是分布式架构的完整方案：
1. **分片**：将数据分散到数千个节点
2. **缓存**：多级缓存（L1 应用缓存 → L2 Redis → L3 DB）
3. **读写分离**：一主多从，读请求分到从库
4. **异步写入**：消息队列缓冲写入
5. **连接池**：合理的连接池大小
6. **限流**：保护系统不过载
7. **监控**：实时监控延迟和错误率
