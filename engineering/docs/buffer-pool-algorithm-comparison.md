# Buffer Pool 置换算法对比：LRU vs Clock vs LFU

## 对比概述

| 维度 | LRU | Clock | LFU |
|------|-----|-------|-----|
| 核心思想 | 置换最久未使用的页面 | 近似 LRU，给页面"二次机会" | 置换使用频率最低的页面 |
| 数据结构 | 双向链表 + Hash 表 | 环形数组 + 指针 | 最小堆 |
| 时间复杂度 | O(1) | O(1) | O(log n) |
| 空间开销 | 高（链表） | 低（环形数组） | 中（堆） |
| 并发友好 | 差（需要锁链表） | 好（原子指针递增） | 差（需维护堆） |
| 实际使用 | 较少（通用缓存） | **PostgreSQL / Oracle** | 少（特殊场景） |

## LRU (Least Recently Used)

### 原理

```
LRU 本质：如果页面最近被访问过，将来也更可能被访问。

链表结构：
┌───┐   ┌───┐   ┌───┐   ┌───┐
│ A │ → │ B │ → │ C │ → │ D │
│   │ ← │   │ ← │   │ ← │   │
└───┘   └───┘   └───┘   └───┘
 头                                   尾
 最新访问                            最久访问

访问策略：
- 每次访问页面将其移到链表头
- 替换时取链表尾的页面
```

### 问题

```
LRU 的几个问题：

问题 1：链表更新开销
每次页面访问都需要更新链表，而数据库每秒可能访问数十万页面。

问题 2：顺序扫描污染
SELECT * FROM table 扫描所有页面，会将它们全移到链表头，
使真正应该留在缓存的热点页面被挤出去。

问题 3：并发竞争
全局链表需要细粒度锁，多线程场景下性能急剧下降。

问题 4：内存开销
每个页面都需要维护 prev/next 指针。
```

## Clock (二次机会算法)

### 原理

```
Clock 本质：环形数组中，每个页面有一个 usage_count 计数器。

环形结构：
        ┌─────────────────────────────┐
        │                             │
        ▼                             │
   ┌───┬───┬───┬───┬───┬───┬───┐   │
   │ 0 │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ ... │
   └───┴───┴───┴───┴───┴───┴───┘   │
        ▲              ▲              │
        │              │              │
   (scanning)    (selected!)

置换过程：
1. 如果 current->usage_count > 0：
   current->usage_count--，继续扫描下一个
2. 如果 current->usage_count == 0 且未 pinned：
   选择此页面
3. 继续循环
```

### 为什么 PostgreSQL 选择 Clock

```
PostgreSQL 选择 Clock-Sweep 的原因：

✅ 优点：
1. 无链表开销：每个页面只需要一个 int 字段
2. 顺序扫描友好：扫描页面 usage_count=1，很快被置换
3. 并发友好：只需要一个原子递增的指针
4. 自适应：热点页面 usage_count 持续增加

❌ 缺点：
1. 精度不及 LRU：不能精确反映访问顺序
2. 冷启动慢：初始阶段像随机置换
```

## LFU (Least Frequently Used)

### 原理

```
LFU 本质：记录每个页面的访问次数，置换最不常访问的页面。

堆结构：
         ┌───┐
    ┌─── │ E │ ───┐
    │    └───┘    │
    ▼             ▼
  ┌───┐        ┌───┐
  │ A │        │ B │
  └───┘        └───┘
  (3次)        (5次)

  ┌───┐
  │ C │
  └───┘
  (1次) ← 最不常访问，被置换

保持频率计数：
- 每次访问 counter++
- 替换时选 counter 最小的
```

### 问题

```
LFU 的问题：

问题 1：历史包袱
过去很热的数据一直占用缓存，即使现在不再访问。

问题 2：频率衰减
需要定期衰减频率计数，否则老数据永远不被置换。

问题 3：更新开销
每次访问需要更新堆，O(log n)。

问题 4：频率存储
需要额外存储频率计数，在内存受限时不可接受。

典型应用：
- 缓存 CDN 内容（频率变化慢）
- 静态内容服务（访问模式稳定）
```

## 对比分析

### 场景对比

```
场景 1：随机访问
LRU:  B (跟踪访问顺序，但链表开销大)
Clock:C (reasonable)
LFU:  C (需要维护频率计数)

场景 2：顺序扫描
LRU:  D (扫描污染热点数据)
Clock:A (顺序访问 usage_count=1，快速被置换)
LFU:  C (顺序扫描频率增加少)

场景 3：热点数据 + 冷数据
LRU:  A (精确跟踪热点)
Clock:B (usage_count 区分热点)
LFU:  B (频率区分热点，但有历史包袱)

场景 4：高并发
LRU:  D (链表锁竞争)
Clock:A (单一原子指针)
LFU:  D (堆锁竞争)
```

### 实现复杂度

| 算法 | 代码复杂度 | 调优参数 | 测试难度 |
|------|----------|---------|---------|
| LRU | 中 | 无 | 中 |
| Clock | 低 | `usage_count` 上限 | 低 |
| LFU | 高 | 衰减策略、频率范围 | 高 |

## 结论

| 场景 | 推荐算法 |
|------|----------|
| 通用数据库（写多读少） | **Clock** — PostgreSQL 的选择 |
| 只读数据库（读多写少） | LRU + 扫描检测 |
| 静态数据缓存 | LFU |
| 高并发 + 低延迟 | Clock + 预取优化 |

Clock-Sweep 之所以被 PostgreSQL 选择，是因为它在多种工作负载下提供了最好的性价比。
