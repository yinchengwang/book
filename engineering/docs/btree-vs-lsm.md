# BTree vs LSM Tree 对比分析

## 核心差异

| 维度 | BTree | LSM Tree |
|------|-------|----------|
| 写入方式 | 原地更新（in-place） | 追加写（append-only） |
| 读取方式 | 单次查找 O(log n) | 多层查找（读放大） |
| 空间放大 | 低（分裂碎片） | 高（合并过程） |
| 写放大 | 高（随机 IO + 页分裂） | 低（顺序 IO） |
| 适用场景 | OLTP（在线事务） | OLAP / 时序 / 日志 |

## BTree 深入分析

### 优势

```
1. 读性能稳定
   - 单次查找代价 = 树高 × 页面读取
   - 树高通常 3-4 层（缓存 2-3 层后，只需 1-2 次磁盘读取）
   - 对于点查询，延迟可预测

2. 成熟的生态
   - PostgreSQL、MySQL (InnoDB)、Oracle
   - 几十年的工程优化经验
   - 丰富的索引类型和优化器支持

3. 空间利用率
   - 原地更新不产生多版本
   - 页面填充率高（50-100%）
```

### 劣势

```
1. 写性能差
   - 随机 IO：每次写入可能访问不同的页面
   - 页分裂：需要读取兄弟页面，写回多个页面
   - 碎片化：频繁分裂和合并导致页面碎片

2. 并发控制复杂
   - 写操作需要锁保护
   - 分裂/合并影响并发度
   - 死锁风险

3. 顺序写入不优
   - 即使是顺序键，也可能分散在不同页面
   - 页面缓存管理开销
```

## LSM Tree 深入分析

### 架构

```
LSM Tree 的多层结构：

Level 0 (内存):  MemTable (跳表/红黑树)
    │ flush (顺序写)
    ▼
Level 1 (磁盘):  SSTable 1, SSTable 2, ...
    │ compaction
    ▼
Level 2 (磁盘):  SSTable 1, SSTable 2, ...
    │ compaction
    ▼
Level N (磁盘):  更大的 SSTable 集合
```

```
写入流程：
1. 写入 MemTable (内存，O(log n) 的跳表插入)
2. MemTable 满时 flush 到 Level 1 (磁盘，顺序写)
3. Level 1 满时 compaction 到 Level 2 (读并排序，再写回)

读取流程：
1. 先查 MemTable (O(log n) 查找)
2. 再查 Level 1 的 SSTable (二分查找 + 块内扫描)
3. 再查 Level 2...直到找到
4. 使用 Bloom Filter 加速：如果 Bloom Filter 说"不存在"，跳过该层
```

### 优势

```
1. 写性能极佳
   - 所有写入都是顺序 IO
   - 不需要原地更新
   - WAL 自然地内置于日志追加写

2. 压缩友好
   - SSTable 可以高效压缩
   - 列式存储可进一步压缩

3. 并发简单
   - 写操作几乎无锁
   - 读操作只需要读已有文件
```

### 劣势

```
1. 读放大
   - 需要查询多层
   - 每层都需要 IO
   - 使用 Bloom Filter 缓解但不能消除

2. 空间放大
   - Compaction 期间新旧 SSTable 并存
   - 需要额外的磁盘空间
   - 合并过程中写放大可达 10-100x

3. Compaction 开销
   - 清理合并文件消耗大量 IO
   - 可能与前台请求竞争资源
   - 不合理的 Compaction 策略导致性能悬崖
```

## 场景选择指南

### OLTP — 选 BTree

```
OLTP 特征：
- 大量点查询（SELECT ... WHERE id = ?）
- 小事务、短查询
- 读写混合
- 并发度高

BTree 在 OLTP 的优势：
- 点查询低延迟（1-2 次磁盘读取）
- 丰富的索引类型（位图扫描、连接）
- 成熟的并发控制
```

### 时序数据库 — 选 LSM Tree

```
时序数据库特征：
- 大量顺序写入（时间戳递增）
- 查询以时间范围为主
- 很少更新/删除历史数据
- 数据压缩很重要

LSM Tree 在时序的优势：
- 顺序写入完美匹配时间戳递增
- SSTable 压缩效果好
- Compaction 可以按时间桶自然分组
```

### 全文本搜索 — 选 LSM Tree

```
全文本搜索特征：
- 写入大量倒排索引条目
- 查询是全文检索操作
- 索引键分散

LSM Tree 的优势：
- 写入倒排索引是典型的海量多键场景
- 不需要原地更新索引
```

### 混合场景

```
有些数据库尝试两者兼得：

CockroachDB：
- 使用 RocksDB (LSM) 作为底层存储
- 在 SQL 层实现 BTree 语义
- 适用于分布式、高写入场景

MongoDB (WiredTiger)：
- 默认使用 BTree
- 支持 LSM 模式
- 根据工作负载选择
```

## 性能对比

| 指标 | BTree | LSM Tree |
|------|-------|----------|
| 随机写 IOPS | 100-1000 | 10K-100K |
| 随机读 IOPS | 10K-100K | 1K-10K |
| 顺序写吞吐量 | 50-200MB/s | 200-500MB/s |
| 顺序读吞吐量 | 200-500MB/s | 100-300MB/s |
| 空间放大 | 1.2-1.5x | 2-10x |
| 写放大 | 1x | 5-50x |

## 未来趋势

```
新兴存储对设计的影响：

NVMe SSD：
- 随机 IO 和顺序 IO 差距缩小
- BTree 的优势更明显
- LSM 的顺序写优势减弱

持久化内存 (PMEM)：
- 完全消除 IO 瓶颈
- BTree 因其简单性可能成为唯一选择
- LSM 的多层结构反而成为负担

分布式存储：
- 多副本降低了单个节点的持久性要求
- LSM 的压缩在复制场景下可能浪费计算
```

## 结论

| 需求 | 选择 |
|------|------|
| 事务性应用、点查询为主 | **BTree** |
| 日志、时序、批量写入 | **LSM Tree** |
| 混合负载 | BTree + 缓存层 |
| 分布式、高写入 | LSM Tree + 分区 |

关键洞察：**选择 BTree 还是 LSM Tree 不是技术问题，而是你的工作负载特征问题。**
