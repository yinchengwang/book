# WAL 设计权衡分析

## 概述

WAL（Write-Ahead Logging）的设计涉及多个维度的权衡。本文分析这些设计决策如何影响系统的正确性、性能和可靠性。

## 权衡维度总览

| 维度 | 选项 A | 选项 B | 本实现 |
|------|--------|--------|--------|
| 存储介质 | 单独文件 | 共享文件 | 单独 WAL 文件 |
| 记录粒度 | 物理日志 | 逻辑日志 | 物理+逻辑混合 |
| 刷盘策略 | 同步 fsync | 异步 | 同步（事务提交） |
| 检查点 | 全量 | 增量 | 全量 |
| 恢复方向 | 前向 Redo | 双向 | 前向 Redo |

## 1. 物理日志 vs 逻辑日志

### 物理日志

```
物理日志记录页面的精确字节变化：

Record: [page_id=42, offset=128, old_bytes="ABC", new_bytes="XYZ"]

优点：
✅ 精确恢复页面到特定状态
✅ 不依赖表结构修改
✅ 实现简单

缺点：
❌ 需要完整页面写入（full_page_writes）
❌ 记录体积大（页面越大越大）
❌ 复制到不同架构有兼容性问题
```

### 逻辑日志

```
逻辑日志记录SQL操作：

Record: [txn_id=1, op="INSERT INTO t VALUES (1, 'A')"]

优点：
✅ 记录体积小
✅ 可读性好（可直接审计）
✅ 跨平台兼容
✅ 支持逻辑复制

缺点：
❌ 恢复时需重新执行SQL
❌ 不确定执行结果（函数可能是非确定性的）
❌ 依赖表结构
```

### 混合方案

```
PostgreSQL 的混合方案：

1. 对于数据: 使用物理日志（页面变化）
2. 对于DDL:  使用逻辑日志（SQL语句）
3. 全文页面: full_page_writes 确保第一次修改时写完整页面

我们的实现：
- 使用逻辑格式（BEGIN/INSERT/UPDATE/DELETE/COMMIT）
- 因为使用简化的存储引擎，不需要物理日志的并发安全保证
```

## 2. WAL 刷盘策略

### 刷盘时机的权衡

```
时间线：

操作:    BEGIN   INSERT  INSERT  COMMIT
         │       │       │       │
LSN:     100     200     300     400
                                 │
刷盘策略:                          │
                                 ▼
A. 每次操作后 fsync (最安全，最慢)
B. 事务提交时 fsync (平衡)
C. 定时批量 fsync (快，可能丢数据)
D. 不强制 fsync (最快，不安全)
```

### 性能模型

```
刷盘频率对 TPS 的影响（简化模型）：

每次提交都 fsync:
  TPS = 1 / (事务处理时间 + fsync 延迟)
  假设 fsync ~ 10ms → 最多 100 TPS

批量 fsync (10 个事务):
  TPS = 10 / (10 * 事务处理时间 + fsync 延迟)
  同样 10ms → 接近 1000 TPS

组提交 (Group Commit):
  多个提交合并为一个 fsync
  达到接近批量 fsync 的效果，但延迟更低
```

### 组提交

```c
// 简单的组提交实现
void group_commit_wait(TransactionId xid) {
    // 1. 加入等待队列
    add_to_waiting_queue(xid);

    // 2. 如果当前没有进行中的 fsync
    if (!fsync_in_progress) {
        fsync_in_progress = true;

        // 3. 收集队列中的事务并 fsync
        int n_committed = collect_waiting_transactions();
        fsync_wal_file();
        wake_waiting_transactions(n_committed);
    }

    // 4. 等待被唤醒
    wait_for_fsync_completion(xid);
}
```

## 3. 检查点设计

### 全量 vs 增量检查点

```
全量检查点：
- 定期保存所有脏页
- 耗时：几秒到几分钟
- 恢复：从检查点开始重放

增量检查点：
- 每次刷一部分脏页
- 持续进行，平滑 IO
- 恢复：需要处理脏页表

PostgreSQL 使用 "增量检查点"：
目的：避免全量检查点导致的 IO 风暴
```

### 检查点频率

```
检查点频率的策略空间：

频繁检查点（如每分钟）：
✅ 恢复时间短
✅ WAL 文件小
❌ IO 开销大

宽松检查点（如每小时）：
✅ IO 开销小
✅ 前台事务延迟低
❌ 恢复时间长
❌ WAL 文件大

最佳策略：基于工作负载自适应
- 检查点间隔 = max(min_wal_size, wal_segment_size * checkpoint_segments) / 写入速率
```

## 4. 恢复性能

### 恢复时间预估

```
恢复时间模型：

T_recovery = T_analysis + T_redo + T_undo

T_analysis = O(WAL 记录数)
T_redo     = O(重做页面数 × 页面 IO 时间)
T_undo     = O(回滚事务数 × 操作数)

优化方向的权衡：

优化 T_redo：
- 缩短检查点间隔（→ 增加检查点 IO 开销）
- 检查点并行化

优化 T_undo：
- 事务不能太长（→ 约束应用设计）
- 回滚段优化
```

### 恢复的恢复

```
恢复过程中如果再次崩溃怎么办？

ARIES 的优雅之处：
┌─────────────────────────────────────────────┐
│ 1. Redo 是幂等的（可以重复执行）              │
│ 2. Undo 记录 (CLR) 也是 WAL 记录             │
│ 3. 恢复过程本身产生 CLR（记录 Undo 操作）      │
│                                             │
│ 所以第二次恢复可以继续第一次恢复了多少           │
│ ↓                                           │
│ CLR 的存在保证了恢复的幂等性和可恢复性          │
└─────────────────────────────────────────────┘
```

## 5. 复制与 WAL

### WAL 在复制中的作用

```
主备复制通过 WAL 流实现：

主库：                              备库：
WAL 写入 → WAL 发送进程 → ...
                                WAL 接收进程 → WAL 应用
                                (可配置同步/异步)
                                │
                                ├─ 同步：主库等待备库确认
                                └─ 异步：主库不等待

WAL 格式的选择：
- 物理日志：精确复制页面 → 必须同架构
- 逻辑日志：复制操作 → 可以跨版本/架构
```

## 6. WAL 压缩

### 压缩 vs 性能

```
WAL 压缩的影响：

不压缩：
✅ 延迟低
✅ CPU 开销小
❌ 磁盘空间大
❌ 网络带宽大（复制）

压缩：
✅ 磁盘空间小（50-70% 节省）
✅ 网络带宽小
❌ CPU 开销大
❌ 延迟可能增加

典型配置：
- 本地存储：不压缩（磁盘便宜）
- 云存储：压缩（按量计费）
- 网络复制：压缩（带宽昂贵）
```

## 结论：推荐配置

| 工作负载 | 刷盘策略 | 检查点间隔 | 压缩 | 复制 |
|--------|---------|-----------|------|------|
| 金融交易 | 同步 fsync | 1 分钟 | 否 | 同步 |
| 电商 | 组提交 | 5 分钟 | 否 | 同步 |
| 分析 | 异步 | 30 分钟 | 是 | 异步 |
| 日志 | 异步 | 不检查点 | 是 | 异步 |
| 开发/测试 | 不 fsync | 10 分钟 | 否 | 否 |

正确的选择取决于你对性能、可靠性和成本的优先级排序。
